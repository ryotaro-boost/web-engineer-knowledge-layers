指定された末端知識ドキュメント（details/）を単体でレビューしてください。

## 入力
$ARGUMENTS（形式: `詳細ファイル名`、例: `CORS`, `DFSとBFS.md`）

拡張子 `.md` は省略可能。

## 手順

1. `details/` 配下から対象ファイルを特定し読み込む
2. フロントマターの `parent` フィールドから親トピックファイルを特定し読み込む（文脈把握）
3. 親トピック内で対象 detail への `[[リンク]]` がインラインで存在するか確認する
4. 以下のレビュー観点で精査し、問題点をカテゴリごとにまとめて報告する

## レビュー観点

### A. 事実の正確性・ハルシネーション検出
- 技術的な説明に誤りがないか
- コード例が正しく動作するか（構文エラー、ロジックエラー）
- 計算量や性能特性の記述が正確か
- 言語仕様やAPI の説明が正しいか（例: メソッド名、引数、返り値）
- 歴史的事実（年号、人名、規格名）が正確か
- 「多くの」「一般的に」等の一般化が過度でないか

### B. 最新性（Currency）
- 言語やランタイムのバージョンに依存する情報が最新か
  - 新しいメジャーバージョンで変更された仕様
  - 非推奨（deprecated）になった API やパターンの使用
- ライブラリやツールのバージョン情報が古くないか
- 統計データ（普及率、シェアなど）が概ね最新か

### C. コンテンツ品質
- **一言で言うと**（blockquote）が冒頭にあり、概念を的確に要約しているか
- コード例の言語カバレッジが十分か（TypeScript, PHP, Ruby, Go, Python のうち最低2言語）
- コード例が最小限かつ実行可能か
- よくある落とし穴セクションがあり、実務で有用な内容か
- 概念の説明が表面的すぎず、本質的な理解を促す深さがあるか
- 図解（Mermaid 記法）が概念の理解に適切に活用されているか
- 比較表やフロー図など、直感的理解を助ける表現が使われているか

### D. 構造・一貫性
- フロントマターに必須フィールドが揃っているか（`layer`, `parent: "[[親トピック名]]"`, `type: detail`, `created`）
- 親トピックファイル内に、この detail への `[[リンク]]` がインラインで配置されているか（末尾一覧ではなく本文中）
- 本文中の `[[リンク]]` のリンク先が実在するか
- 日本語で記述され、技術用語の初出時に英語が併記されているか
- 見出しが `# タイトル（英語名）` の形式になっているか

## 報告フォーマット

各問題を以下の形式で報告する:

```
### [カテゴリ] ファイル名:行番号
- **問題:** 具体的な問題の説明
- **修正案:** 推奨する修正内容
```

最後に総評（全体的な品質、特に良い点、改善の優先度）を記載する。

## 注意事項
- 軽微なスタイルの好みや、教材としての意図的な簡略化は指摘しない
- 事実誤りと最新性の問題を最優先で報告する
- Web検索が利用可能な場合は、事実確認にWebSearchを活用すること
- 修正の適用は報告後にユーザーの確認を取ってから行うこと（自動では修正しない）
