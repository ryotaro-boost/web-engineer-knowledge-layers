指定されたレイヤーのトピックドキュメントと関連する末端知識ドキュメント（details/）をレビューしてください。

## 入力
$ARGUMENTS（形式: `LayerN`、例: `Layer0`）

## 手順

1. 対象レイヤーフォルダ内の全 `.md` ファイル（`_index.md` 含む）を読む
2. 各トピックファイルから参照されている `details/` 配下のファイルも全て読む
3. 以下のレビュー観点で各ファイルを精査し、問題点をカテゴリごとにまとめて報告する

## レビュー観点

### A. 事実の正確性・ハルシネーション検出
- 技術的な説明に誤りがないか
- コード例が正しく動作するか（構文エラー、ロジックエラー）
- 計算量や性能特性の記述が正確か
- 言語仕様やAPI の説明が正しいか（例: メソッド名、引数、返り値）
- 歴史的事実（年号、人名、規格名）が正確か
- 「多くの」「一般的に」等の一般化が過度でないか

### B. 最新性（Currency）
- 言語やランタイムのバージョンに依存する情報が最新か
  - 新しいメジャーバージョンで変更された仕様（例: Python の GIL、PHP のバージョン、Node.js の API）
  - 非推奨（deprecated）になった API やパターンの使用
- ライブラリやツールのバージョン情報が古くないか
- 統計データ（普及率、シェアなど）が概ね最新か
- OS やブラウザの仕様変更が反映されているか

### C. 補足情報の過不足
- 必須セクション（なぜ必要か / どの問題を解決するか / 他の仕組みとの関係 / 誤解されやすいポイント / ベストプラクティス）が揃っているか
- コード例の言語カバレッジに偏りがないか（TypeScript, PHP, Ruby, Go, Python が対象言語）
- `[[リンク]]` のリンク先が実在するか
- details/ のカバレッジ — トピック本文中で深掘りすべき概念に detail ドキュメントがあるか
- 関連トピックへの相互リンクが適切か

### D. 一貫性
- テンプレート構造（CLAUDE.md 規約）に従っているか
- フロントマターの形式が正しいか（layer, topic/parent, type, status, created）
- details/ ファイルに `parent` と `type: detail` が含まれているか
- 詳細へのリンクが本文中にインラインで配置されているか（末尾一覧ではなく）

## 報告フォーマット

各問題を以下の形式で報告する:

```
### [カテゴリ] ファイル名:行番号
- **問題:** 具体的な問題の説明
- **修正案:** 推奨する修正内容
```

最後に総評（全体的な品質、特に良い点、改善の優先度）を記載する。

## 注意事項
- 軽微なスタイルの好みや、教材としての意図的な簡略化は指摘しない
- 事実誤りと最新性の問題を最優先で報告する
- Web検索が利用可能な場合は、事実確認にWebSearchを活用すること
- 修正の適用は報告後にユーザーの確認を取ってから行うこと（自動では修正しない）
