# CLAUDE.md - Web Engineer Knowledge Layers プロジェクト規約

## プロジェクト概要
Webエンジニアリングの知識を8層（Layer 0-7）に分けて体系的に学習・文書化するプロジェクト。
Obsidian Vault内で運用する。

## 言語規約
- ドキュメントはすべて**日本語**で記述する
- 技術用語は初出時に英語を併記する（例：「計算量（Big-O）」）
- 英語の専門用語はそのままカタカナにせず原語を使う場合がある（例：SOLID, CORS, CDN）

## ファイル命名規約
- フォルダ名: `LayerN-日本語名/` （例: `Layer0-CS基礎/`）
- ファイル名: 日本語トピック名.md（スペースなし、英語略語はハイフン区切り）
- 各レイヤーフォルダに `_index.md` を配置（レイヤー概要 + トピック一覧）

## トピックドキュメントのテンプレート
各トピックは以下の5つの視点で深掘りする:

### 必須セクション
1. **なぜ必要か** - この技術/概念が存在する理由
2. **どの問題を解決するか** - 具体的な課題とその解決方法
3. **他の仕組みとどう関係するか** - 他レイヤー・他トピックとの依存関係（`[[リンク]]` を積極的に使用）
4. **誤解されやすいポイント** - よくある間違いや混同
5. **設計のベストプラクティス** - 実務での推奨パターン

### 任意セクション
- **AIによる実装のアンチパターン** - LLM生成コードで頻出する過剰設計・誤用パターンとその対策（例：不要なフォールバック、過剰な抽象化、防御的すぎるエラーハンドリング）
- **具体例・コードスニペット** - 理解を助ける実例
- **参考リソース** - 書籍、記事、動画等のリンク
- **学習メモ** - 個人的な気づきや疑問点

## Obsidian リンク規約
- トピック間の参照は `[[トピック名]]` を使用する
- レイヤーをまたぐ参照を積極的に行う（例: `[[インデックス]]` から `[[ファイルシステムとIO]]` へ）
- セキュリティ（Layer 6）とパフォーマンス（Layer 5）は横断的関心事として他レイヤーから頻繁にリンクする

## 学習進捗の記録
- 各 `_index.md` にトピックごとの学習ステータスを記録する
  - 🔴 未着手 / 🟡 学習中 / 🟢 完了
- ステータスはインラインで管理する

## 末端知識ドキュメント（details/）
- プロジェクトルート直下に `details/` ディレクトリを配置する（レイヤー横断で共有するため）
- トピックの個別概念や深掘り情報を格納する（例: `details/ハッシュテーブル.md`）
- フロントマターに `parent: "[[親トピック名]]"` と `type: detail` を含める
- テンプレート構造に厳密に従う必要はないが、概念説明・複数言語のコード例・落とし穴を含めること
- **詳細へのリンクはトピックファイルの本文中にインラインで配置する**（末尾の一覧ではなく、関連するテキスト付近に `[[詳細名]]` を埋め込み、読み進める導線で自然に辿れるようにする）

## Claude Code への指示
- 新しいトピックドキュメントを作成する際は `_templates/topic-template.md` の構造に従うこと
- `[[リンク]]` は実在するファイル名と一致させること
- コード例は実行可能な最小限のスニペットとし、言語を明示すること（```python, ```javascript 等）
- 1トピック = 1ファイルの原則を守ること
- `/study-topic` コマンドでトピックドキュメントを生成できる
