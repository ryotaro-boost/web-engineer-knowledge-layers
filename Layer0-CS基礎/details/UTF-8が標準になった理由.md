# UTF-8 が標準になった理由

> **一言で言うと:** UTF-16 は NUL バイト混入と ASCII 非互換という致命的な弱点を抱えており、既存の Unix/Web インフラと共存できなかった。UTF-8 は既存のエコシステムを壊さずに Unicode を導入できる唯一の選択肢だった。

## 概念

### UTF-16 が先に採用された歴史的背景

Unicode の初期（1990年代）、業界は UTF-16 に大きく賭けていた。Windows NT（1993年）、Java（1995年）、JavaScript（1995年）はいずれも内部文字列を UTF-16 で扱う設計を採用した。当時の Unicode は U+FFFF 以内に収まる想定（65,536文字で十分）だったため、固定長2バイトの UCS-2（UTF-16 の前身）は合理的な選択だった。

### UTF-8 が勝った決定的な理由

しかし Web の世界では UTF-8 が圧倒的に有利だった。

| 観点 | UTF-8 | UTF-16 |
|------|-------|--------|
| ASCII 互換性 | 既存の ASCII テキストがそのまま有効 | すべての ASCII 文字に NUL バイト（0x00）が混入する |
| 既存インフラとの互換 | C言語の `strlen()`・`strcmp()` がそのまま動く | NUL バイトで文字列が途中終端する |
| HTTP/HTML との親和性 | プロトコルが ASCII ベースなのでシームレス | バイナリ転送扱いが必要になる |
| バイト順問題 | なし（バイト単位の直列処理） | エンディアン依存（BOM が必要） |
| 英語圏のサイズ効率 | 英数字は1バイト | 英数字でも2バイト（2倍のサイズ） |
| セキュリティ | NUL バイトが混入しない | 0x00 バイトの混入で古いシステムが誤動作するリスク |

### NUL バイト問題が決定的だった

UTF-16 で ASCII 文字 `A`（U+0041）をエンコードすると `0x00 0x41`（BE）または `0x41 0x00`（LE）となり、0x00 が現れる。Unix 系 OS のファイルパス処理、C言語の文字列関数、多くのネットワークプロトコルは NUL バイトを文字列終端として扱うため、UTF-16 のバイト列をそのまま渡すと文字列が途中で切れる。

UTF-8 にはこの問題がない（NUL 文字自身を除き 0x00 が現れない設計）。これにより、既存の膨大な C/Unix ベースのソフトウェア資産を書き換えることなく Unicode 対応ができた。

### Web での普及の経緯

1990年代後半〜2000年代にかけて Web が爆発的に普及する中で、既存のインフラ（HTTP サーバー、プロキシ、ファイルシステム、シェルスクリプト）との互換性を壊さずに国際化できる UTF-8 が自然に選ばれた。

- **2003年頃** — UTF-8 の採用率が急上昇を開始
- **2008年** — Web 上で最も使われるエンコーディングとなる
- **現在** — 約98%のページが UTF-8 を使用。WHATWG の HTML Standard は新規コンテンツに UTF-8 を推奨

### 歴史的な教訓

UTF-16 を内部表現に選んだ Windows・Java・JavaScript は、後に Unicode が U+FFFF を超えて拡張された際に[[サロゲートペア・結合文字・書記素クラスタ|サロゲートペア]]という複雑さを抱え込むことになった。「固定長で効率的」という UTF-16 の利点は、可変長（サロゲートペア）が必要になった時点で失われた。

一方で UTF-8 は最初から可変長設計であり、Unicode の拡張（絵文字の追加など）に対しても設計変更なしで対応できている。

## 現在も UTF-16 が使われている場所

UTF-8 が Web の標準であっても、UTF-16 は以下の場所で今も現役である:

| システム | 理由 |
|---------|------|
| Windows API | NT カーネルが UTF-16 ベースで設計された（1993年の判断が今も残る） |
| Java の `String` | JVM 仕様で UTF-16 が規定されている |
| JavaScript の `String` | ECMAScript 仕様で UTF-16 コードユニットの列と定義 |
| macOS の `NSString` | Core Foundation が UTF-16 ベース |
| .NET の `string` | CLR が UTF-16 を内部表現に採用 |

これらのシステムでは入出力時に UTF-8 ↔ UTF-16 の変換が発生する。この変換コストは通常無視できるが、大量のテキスト処理ではボトルネックになりうる。

## よくある落とし穴

1. **「UTF-16 は非効率だからダメ」は単純化しすぎ** — CJK 文字（日本語・中国語・韓国語）は UTF-16 で2バイト、UTF-8 で3バイト。CJK 主体のテキストでは UTF-16 の方がコンパクト。ただし HTTP 圧縮後の差は無視できるため、Web では UTF-8 統一が正解
2. **「UTF-8 に変換すれば安全」ではない** — 不正な UTF-8 シーケンス（overlong encoding など）を受け入れるとセキュリティ脆弱性になる。バリデーション付きのデコードが必要
