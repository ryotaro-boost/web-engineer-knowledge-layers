# ハッシュテーブル（Hash Table）

> **一言で言うと:** キー（key）からハッシュ関数で配列のインデックスを計算し、値（value）を格納・検索するデータ構造。平均 O(1) で検索・挿入・削除ができる。

## 概念

### 基本的な仕組み

ハッシュテーブルは内部的に「配列」を持っている。キーを受け取ると、以下の流れで値にアクセスする。

```
キー "alice" → ハッシュ関数 → 整数 4827 → 配列サイズで剰余 → インデックス 3 → 配列[3] に格納
```

1. **ハッシュ関数（Hash Function）** — 任意のキーを固定長の整数に変換する関数。同じキーからは常に同じ整数が返る（決定的）。
2. **バケット（Bucket）** — ハッシュ値から計算されたインデックスに対応する格納場所。
3. **衝突（Collision）** — 異なるキーが同じインデックスに割り当てられること。必ず起きる（鳩の巣原理）。

### 衝突の解決方法

| 方式 | 仕組み | 特徴 |
|------|--------|------|
| チェイニング（Chaining） | 各バケットにリンクリストや配列を持つ | 実装が単純、メモリを余分に使う |
| オープンアドレス法（Open Addressing） | 衝突時に別の空きバケットを探す | メモリ効率が良い、削除が複雑 |

多くの言語の組み込み実装はチェイニングを採用している。

### 負荷率（Load Factor）

```
負荷率 = 要素数 / バケット数
```

負荷率が高くなると衝突が増えて性能が劣化する。多くの実装では負荷率が閾値（0.75 など）を超えるとバケット配列を拡張し、全要素を再配置する（リハッシュ）。このリハッシュは O(n) のコストがかかるが、償却計算（amortized analysis）で挿入は平均 O(1) となる。

### 順序について

ハッシュテーブルは本来「順序を保証しない」データ構造である。ただし言語によって挙動が異なる：

- **挿入順を保持する:** JavaScript の `Map`、Python 3.7+ の `dict`、Ruby の `Hash`、PHP 配列
- **順序を保証しない:** Go の `map`（意図的にランダム化されている）

## 各言語での使用例

### TypeScript

TypeScript では `Object`、`Map`、`Record` の3つが主な選択肢。

```typescript
// --- Object（最も一般的、キーは string | symbol のみ） ---
const userRoles: Record<string, string> = {
  "alice": "admin",
  "bob": "editor",
  "charlie": "viewer",
};

console.log(userRoles["alice"]); // "admin"
console.log("bob" in userRoles); // true

// 走査
for (const [name, role] of Object.entries(userRoles)) {
  console.log(`${name}: ${role}`);
}

// --- Map（キーに任意の型を使える、頻繁な追加・削除に強い） ---
interface User {
  id: number;
  name: string;
}

const sessionMap = new Map<User, string>();
const alice: User = { id: 1, name: "Alice" };
const bob: User = { id: 2, name: "Bob" };

sessionMap.set(alice, "session_abc123");
sessionMap.set(bob, "session_def456");

console.log(sessionMap.get(alice)); // "session_abc123"
console.log(sessionMap.size);       // 2

// --- Object vs Map の使い分け ---
// Object: JSON との相互変換が自然、構造が固定的
// Map:    キーが文字列以外、要素数が頻繁に変わる、サイズを .size で取得したい
```

### PHP

PHP の配列（array）は実際にはハッシュテーブルとして実装されている。言語レベルで配列と連想配列が統合されているのは PHP の特徴。

```php
<?php
// --- 連想配列（PHP の array はハッシュテーブル） ---
$prices = [
    "apple"  => 150,
    "banana" => 100,
    "cherry" => 300,
];

echo $prices["apple"]; // 150

// キーの存在確認
if (array_key_exists("banana", $prices)) {
    echo "banana exists";
}

// isset は null の場合 false を返すので注意
$data = ["key" => null];
var_dump(array_key_exists("key", $data)); // true
var_dump(isset($data["key"]));            // false（null なので）

// 走査（挿入順が保持される）
foreach ($prices as $fruit => $price) {
    echo "{$fruit}: {$price}円\n";
}

// --- 実務的な例: リクエストパラメータのバリデーション ---
$rules = [
    "name"  => "required|string|max:255",
    "email" => "required|email",
    "age"   => "nullable|integer|min:0",
];

$errors = [];
foreach ($rules as $field => $rule) {
    if (str_contains($rule, "required") && empty($_POST[$field])) {
        $errors[$field] = "{$field} is required";
    }
}

// --- グルーピング（配列のキーでグループ化） ---
$users = [
    ["name" => "Alice", "dept" => "engineering"],
    ["name" => "Bob", "dept" => "marketing"],
    ["name" => "Charlie", "dept" => "engineering"],
];

$byDept = [];
foreach ($users as $user) {
    $byDept[$user["dept"]][] = $user["name"];
}
// ["engineering" => ["Alice", "Charlie"], "marketing" => ["Bob"]]
```

### Ruby

Ruby の `Hash` は挿入順を保持するハッシュテーブル。シンボル（Symbol）をキーに使うのが慣習的。

```ruby
# --- 基本操作 ---
prices = {
  apple: 150,    # シンボルキー（推奨）
  banana: 100,
  cherry: 300,
}

puts prices[:apple]       # 150
puts prices.key?(:grape)  # false

# デフォルト値の設定
counter = Hash.new(0)     # 存在しないキーは 0 を返す
words = ["apple", "banana", "apple", "cherry", "apple"]
words.each { |w| counter[w] += 1 }
puts counter  # {"apple"=>3, "banana"=>1, "cherry"=>1}

# --- 実務的な例: グルーピングと集約 ---
orders = [
  { product: "Widget", region: "East", amount: 100 },
  { product: "Gadget", region: "West", amount: 200 },
  { product: "Widget", region: "East", amount: 150 },
  { product: "Gadget", region: "East", amount: 300 },
]

# group_by で Hash にグルーピング
by_region = orders.group_by { |o| o[:region] }
# {"East"=>[{...}, {...}, {...}], "West"=>[{...}]}

# each_with_object で集約
totals = orders.each_with_object(Hash.new(0)) do |order, sum|
  sum[order[:region]] += order[:amount]
end
# {"East"=>550, "West"=>200}

# --- fetch: キーが無い場合に例外 or デフォルト値 ---
config = { timeout: 30, retries: 3 }
config.fetch(:timeout)            # 30
config.fetch(:missing, "default") # "default"
config.fetch(:missing)            # KeyError（バグの早期発見に有用）

# --- dig: ネストした Hash の安全なアクセス ---
response = { data: { user: { name: "Alice" } } }
response.dig(:data, :user, :name)    # "Alice"
response.dig(:data, :address, :city) # nil（エラーにならない）
```

### Go

Go の `map` は参照型で、キーの型は `==` で比較可能な型（comparable）に限られる。意図的に走査順がランダム化されている。

```go
package main

import (
	"fmt"
	"sort"
	"sync"
)

func main() {
	// --- 基本操作 ---
	prices := map[string]int{
		"apple":  150,
		"banana": 100,
		"cherry": 300,
	}

	fmt.Println(prices["apple"]) // 150

	// キーの存在確認（comma ok イディオム）
	price, ok := prices["grape"]
	if !ok {
		fmt.Println("grape not found") // ゼロ値(0)と「存在しない」を区別できる
	}
	_ = price

	// 削除
	delete(prices, "banana")

	// --- 走査順は非決定的（毎回変わる） ---
	// ソート済みで走査したい場合はキーを取り出してソート
	keys := make([]string, 0, len(prices))
	for k := range prices {
		keys = append(keys, k)
	}
	sort.Strings(keys)
	for _, k := range keys {
		fmt.Printf("%s: %d\n", k, prices[k])
	}

	// --- 実務的な例: 集約 ---
	words := []string{"go", "rust", "go", "python", "go", "rust"}
	counter := make(map[string]int)
	for _, w := range words {
		counter[w]++ // ゼロ値が 0 なので初期化不要
	}
	fmt.Println(counter) // map[go:3 python:1 rust:2]

	// --- 並行アクセス: sync.Map ---
	// 通常の map はスレッドセーフではない。
	// 並行読み書きが必要な場合は sync.Map を使う。
	var cache sync.Map

	// 書き込み
	cache.Store("key1", "value1")

	// 読み取り
	if val, ok := cache.Load("key1"); ok {
		fmt.Println(val) // "value1"
	}

	// LoadOrStore: 存在すれば取得、なければ格納して取得
	actual, loaded := cache.LoadOrStore("key2", "value2")
	fmt.Println(actual, loaded) // "value2" false
}
```

## 言語間の比較まとめ

| 特性 | TypeScript | PHP | Ruby | Go |
|------|-----------|-----|------|----|
| 主な型 | `Object` / `Map` | `array` | `Hash` | `map` |
| 挿入順保持 | `Map`: Yes, `Object`: Yes (ES2015+) | Yes | Yes | No（ランダム） |
| キー型の制約 | `Object`: string/symbol, `Map`: 任意 | string/int | 任意（hashable） | comparable 型 |
| null安全アクセス | `?.` (optional chaining) | `??` (null合体) | `dig` | comma ok |
| デフォルト値 | `??` 演算子 | `??` 演算子 | `Hash.new(default)` | ゼロ値が自動 |
| スレッドセーフ版 | — (シングルスレッド) | — (プロセスモデル) | — (`Concurrent::Map` gem) | `sync.Map` |

## よくある落とし穴

1. **Go の map はゼロ値を返す** — 存在しないキーにアクセスするとエラーではなくゼロ値（int なら 0、string なら ""）が返る。comma ok パターンで存在確認すること。
2. **PHP の `isset` vs `array_key_exists`** — `isset` はキーが存在しても値が `null` だと `false` を返す。
3. **JavaScript の Object にはプロトタイプ汚染リスクがある** — ユーザー入力をキーにする場合は `Map` を使うか、`Object.create(null)` で作成する。
4. **Ruby のシンボル vs 文字列キー** — `{ "name" => "Alice" }` と `{ name: "Alice" }` は異なるキー。API レスポンス（文字列キー）をシンボルキーでアクセスしようとして `nil` になるミスが多い。
