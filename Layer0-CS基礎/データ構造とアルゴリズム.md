# データ構造とアルゴリズム

> **一言で言うと:** データをどう整理し（データ構造）、どう処理するか（アルゴリズム）の基礎で、「検索が O(1) なのか O(n) なのか」がシステム全体のスケーラビリティを決める。

## なぜ必要か

プログラムは突き詰めると「データを受け取り、変換し、返す」ことの繰り返しである。データの持ち方と処理手順を意識しないと、次のような問題が発生する。

- **パフォーマンス崩壊** — ユーザー数が100人のときは動くが、10万人になった途端にレスポンスが数十秒になる。原因の多くは「配列を毎回線形探索している」「ネストしたループで O(n²) になっている」といった構造的な問題。
- **メモリの無駄遣い** — 不要なコピーや不適切なデータ構造の選択で、メモリ消費が爆発する。
- **コードの複雑化** — 適切なデータ構造を使えば数行で済む処理を、配列とif文の塊で書いてしまい、バグの温床になる。

## どの問題を解決するか

### 問題1: 「探す」コストが高すぎる

最も頻出する操作は「検索」である。配列の[[線形探索]]は O(n) だが、[[ハッシュテーブル]]なら平均 O(1) で済む。

| 操作 | 配列（Array） | ハッシュテーブル（Hash Table） | 二分探索木（BST） |
|------|-------------|--------------------------|----------------|
| 検索 | O(n) | O(1) 平均 | O(log n) |
| 挿入 | O(1) 末尾 / O(n) 先頭 | O(1) 平均 | O(log n) |
| 削除 | O(n) | O(1) 平均 | O(log n) |

### 問題2: 「順序」や「関係性」を表現できない

- **木構造（Tree）** — DOM、ファイルシステム、JSON のパースツリーなど、階層的なデータを表現する。
- **グラフ（Graph）** — SNSの友人関係、ページ間のリンク、マイクロサービス間の依存関係など、多対多の関係を表現する。
- **キュー（Queue）** — ジョブキュー、イベントループのタスクキュー（[[並行性の基本概念]]）、メッセージブローカーなど、「先入れ先出し」の処理順序を保証する。
- **スタック（Stack）** — コールスタック、ブラウザの戻る/進む履歴、undo 機能など、「後入れ先出し」を実現する。

### 問題3: 最適な処理手順がわからない

アルゴリズムは「問題をどの手順で解くか」の定石集である。

- **ソート** — 表示順の並び替え、[[バイナリサーチ]]の前処理、重複排除の前処理。
- **探索** — 深さ優先探索（DFS）と幅優先探索（BFS）で木やグラフを走査する。
- **動的計画法（DP）** — 重複する部分問題を効率的に解く（例：最短経路、編集距離）。

## 他の仕組みとどう関係するか

- **下位レイヤーとの関係:** データ構造はメモリ上のデータ配置そのものである。配列が連続メモリ領域でキャッシュに乗りやすい、リンクリストがポインタで散らばるといった特性は、ハードウェアレベルの知識と直結する。
- **同レイヤーとの関係:** [[計算量-BigO]] はデータ構造・アルゴリズムの「性能を評価する言語」であり、セットで理解する必要がある。[[並行性の基本概念]] では、スレッドセーフなキューやロックフリーデータ構造が重要になる。[[文字コード-UTF8]] の文字列もバイト配列として内部的にはデータ構造の一種。
- **上位レイヤーとの関係:** Layer 1 の [[プロセスとスレッド]] ではキューやスタックが実行管理に使われる。Layer 3 の [[インデックス]] はB木（B-Tree）というデータ構造そのもの。Layer 5 の [[パフォーマンス最適化]] ではデータ構造の選択が直接的な改善手段になる。

## 誤解されやすいポイント

1. **「[[ハッシュテーブル]]は常に最速」という思い込み** — ハッシュテーブルの検索は平均 O(1) だが、ハッシュ衝突が多いと O(n) に退化する。また、順序を保持しない（JavaScript の `Map` は挿入順を保持するが、これは言語仕様であってハッシュテーブルの性質ではない）。範囲検索やソート済みデータの走査には二分探索木の方が適している。

2. **「配列は遅い」という過度な一般化** — 配列は連続メモリに格納されるため CPU キャッシュとの相性が良く、小さなデータセットでは理論的に遅いはずのO(n) 操作が、キャッシュミスの多い O(log n) の木構造より速いことがある。実測が大事。

3. **「Web開発ではアルゴリズムは不要」という誤解** — フレームワークが多くを隠蔽しているが、以下のような場面で知識が問われる：
   - APIレスポンスのページネーション設計（オフセット vs カーソル）
   - フロントエンドのリスト仮想化（どのアイテムを描画するかの二分探索）
   - バックエンドのバッチ処理での効率的なデータ結合
   - データベースクエリの最適化（N+1問題の理解）

4. **「ソートは `Array.sort()` を呼べば十分」** — 呼ぶだけなら確かに十分だが、比較関数の書き方を間違える（数値を文字列としてソートしてしまう）、安定ソートかどうかを気にしない、大量データでのソートがボトルネックになっている場合に改善策を知らない、といった問題が起きる。

## 設計のベストプラクティス

### 推奨パターン

- **「何を最も頻繁にやるか」でデータ構造を選ぶ** — 検索が多いならハッシュテーブル、順序付きアクセスが多いなら配列やソート済みリスト、階層構造なら木。
- **Set を活用する** — 「この要素は存在するか？」の判定には配列の `includes()` ではなく `Set` を使う。O(n) → O(1)。
- **Map を活用する** — キーが文字列以外（オブジェクト参照など）の場合や、頻繁な追加・削除がある場合は `Object` より `Map` が適切。
- **イミュータブルな操作を意識する** — 特にフロントエンドでは、配列やオブジェクトを直接変更せず新しいものを作る（React の状態管理）。ただしパフォーマンスが問題になる場合はミュータブルな操作も選択肢。

### アンチパターン

- **配列をマップ代わりに使う** — `users.find(u => u.id === targetId)` を繰り返し呼ぶなら、最初に `Map` に変換しておく。
- **ネストしたループで結合する** — 2つの配列を照合するときに二重ループ O(n²) を書くのではなく、一方をハッシュマップにして O(n) にする。
- **巨大な配列のスプレッド乱用** — `[...bigArray, newItem]` は毎回 O(n) のコピーが発生する。頻繁に行うならパフォーマンスを意識した設計にする。

## AIによる実装のアンチパターン

LLM にデータ構造やアルゴリズムに関するコードを生成させると、以下のような過剰設計が頻出する。

| アンチパターン | なぜ問題か | 対策 |
|---|---|---|
| **コレクションの不要なラッパークラス** — 配列や Map をわざわざ `UserCollection` のようなクラスで包み、`add`, `remove`, `findById` 等のメソッドを定義する | 標準コレクションの API で十分な場面で抽象化レイヤーが増え、可読性とメンテナンス性が下がる。使い捨てのスクリプトや小規模モジュールには不要 | 組み込みの `Map`, `Set`, `Array` メソッドをそのまま使う。ドメイン固有のロジック（バリデーション等）がない限りラッパーは作らない |
| **防御的すぎるnullチェックの連鎖** — `if (!arr) return []; if (arr.length === 0) return []; ...` のように、起こりえない状態に対するガード節を大量に追加する | コードが肥大化し、本来の処理ロジックが埋もれる。内部関数で入力が保証されている場面では無意味 | システム境界（API入力、ユーザー入力）でのみバリデーションし、内部ロジックでは型システムを信頼する |
| **不要なユーティリティ関数の量産** — `isEmpty(arr)`, `safeGet(map, key, default)` のような薄いラッパー関数を大量に作る | 言語の標準機能（`arr.length === 0`, `map.get(key) ?? default`）で1行で書ける処理を関数化する意味がない | 標準の演算子やメソッドを直接使う。チーム内で統一が必要な場合のみユーティリティを検討 |
| **単純な配列操作への過剰なアルゴリズム適用** — 10件の配列に対してバイナリサーチを導入したり、Map に変換してから検索する | 小さなデータセットでは線形探索の方がシンプルで十分速い。コードの複雑さが無駄に増える | まず n の現実的な上限を確認する。n ≤ 数十件なら `find()` や `includes()` で十分 |

## 具体例

### 例1: 検索コストの改善 — 配列の[[線形探索]] vs [[ハッシュテーブル|ハッシュマップ]]

**対応する問題:** 問題1「探すコストが高すぎる」

**使用シーン:** EC サイトで注文一覧を表示する際、注文データ（user_id のみ持つ）からユーザー名を引く処理。注文が1万件・ユーザーが5千人いると、配列探索では最大5000万回の比較が走る。

```javascript
// ❌ 配列から毎回検索 — O(n) × ループ回数 = O(n²)
const users = [
  { id: 1, name: "Alice" },
  { id: 2, name: "Bob" },
  { id: 3, name: "Charlie" },
];

const orderUserIds = [3, 1, 2, 1, 3];
const names = orderUserIds.map(id => {
  const user = users.find(u => u.id === id); // 毎回 O(n)
  return user?.name;
});

// ✅ 先にMapに変換 — O(n) + O(m) = O(n+m)
const userMap = new Map(users.map(u => [u.id, u]));
const namesFast = orderUserIds.map(id => userMap.get(id)?.name);
```

### 例2: 重複排除 — Set の活用

**対応する問題:** 問題1「探すコストが高すぎる」

**使用シーン:** フォームのタグ入力で、ユーザーが重複タグを入力していないかバリデーションする。メールの宛先リストで同じアドレスが複数入っていないか確認する。

```javascript
// ❌ 配列の includes — O(n) × n回 = O(n²)
function hasDuplicates(arr) {
  for (let i = 0; i < arr.length; i++) {
    if (arr.slice(i + 1).includes(arr[i])) return true;
  }
  return false;
}

// ✅ Set を使う — O(n)
function hasDuplicates(arr) {
  return new Set(arr).size !== arr.length;
}
```

### 例3: キュー — 非同期タスクの順序制御

**対応する問題:** 問題2「順序や関係性を表現できない」— キューの「先入れ先出し（FIFO）」

**使用シーン:** 画像アップロード機能で、ユーザーが複数ファイルを選択したとき、アップロード順序を保証しつつ同時実行数を制限する。メッセージングアプリでの送信キュー、API のレートリミット付きリクエスト処理なども同じパターン。

```typescript
// アップロードキュー: 最大2件ずつ並行処理し、FIFO順序を保証
class UploadQueue {
  private queue: File[] = [];
  private running = 0;
  private maxConcurrent = 2;

  enqueue(file: File) {
    this.queue.push(file);   // 末尾に追加（enqueue）
    this.processNext();
  }

  private async processNext() {
    if (this.running >= this.maxConcurrent || this.queue.length === 0) return;

    const file = this.queue.shift()!; // 先頭から取り出し（dequeue）
    this.running++;
    try {
      await this.upload(file);
    } finally {
      this.running--;
      this.processNext();
    }
  }

  private async upload(file: File): Promise<void> {
    // 実際のアップロード処理
  }
}
```

### 例4: スタック — undo/redo の実装

**対応する問題:** 問題2「順序や関係性を表現できない」— スタックの「後入れ先出し（LIFO）」

**使用シーン:** テキストエディタやお絵描きアプリの「元に戻す/やり直す」機能。ブラウザの戻る/進むボタンも同じ原理。

```typescript
// undo/redo をスタック2本で実現
class UndoRedoManager<T> {
  private undoStack: T[] = [];
  private redoStack: T[] = [];

  execute(state: T) {
    this.undoStack.push(state); // 現在の状態をスタックに積む
    this.redoStack = [];        // 新しい操作が入ったら redo 履歴は破棄
  }

  undo(): T | undefined {
    const state = this.undoStack.pop(); // 最後に積んだ状態を取り出す（LIFO）
    if (state) this.redoStack.push(state);
    return this.undoStack.at(-1);       // 一つ前の状態を返す
  }

  redo(): T | undefined {
    const state = this.redoStack.pop();
    if (state) this.undoStack.push(state);
    return state;
  }
}
```

### 例5: 木構造の走査 — BFS と DFS の使い分け

**対応する問題:** 問題3「最適な処理手順がわからない」— 探索アルゴリズム

**使用シーン:**
- **BFS（幅優先）** — カテゴリツリーの階層ごとの表示、SNSの「友達の友達」探索、最短経路の探索
- **DFS（深さ優先）** — ファイルシステムの全走査、JSON の深いネストの探索、依存関係の解決

```javascript
// BFS: カテゴリツリーを階層ごとに取得（管理画面のパンくずリスト生成等）
function getByLevel(root) {
  const result = [];
  const queue = [root];

  while (queue.length > 0) {
    const levelSize = queue.length;
    const level = [];
    for (let i = 0; i < levelSize; i++) {
      const node = queue.shift();
      level.push(node.name);
      for (const child of node.children ?? []) {
        queue.push(child);
      }
    }
    result.push(level);
  }
  return result;
}

// DFS: ネストしたカテゴリから特定カテゴリへのパスを探す
function findPath(node, target, path = []) {
  path.push(node.name);
  if (node.name === target) return [...path];

  for (const child of node.children ?? []) {
    const found = findPath(child, target, path);
    if (found) return found;
  }
  path.pop(); // バックトラック
  return null;
}

// 使用例
const categories = {
  name: "全商品",
  children: [
    { name: "家電", children: [
      { name: "PC", children: [{ name: "ノートPC" }] },
      { name: "スマホ" },
    ]},
    { name: "書籍" },
  ],
};

console.log(getByLevel(categories));
// [["全商品"], ["家電", "書籍"], ["PC", "スマホ"], ["ノートPC"]]

console.log(findPath(categories, "ノートPC"));
// ["全商品", "家電", "PC", "ノートPC"]
```

### 例6: ハッシュマップによる集約 — Python

**対応する問題:** 問題1「探すコストが高すぎる」+ 問題3「最適な処理手順がわからない」

**使用シーン:** アクセスログの集計、売上データのカテゴリ別集約、ワードカウントなど。

```python
from collections import defaultdict, deque

# defaultdict — キーが存在しない場合のデフォルト値を設定
word_count = defaultdict(int)
for word in ["apple", "banana", "apple", "cherry", "banana", "apple"]:
    word_count[word] += 1
# {'apple': 3, 'banana': 2, 'cherry': 1}

# deque — 両端キュー（先頭・末尾の追加/削除が O(1)）
# 使用シーン: 直近N件のログを保持するリングバッファ的な用途
recent_logs = deque(maxlen=100)  # 最大100件、超えたら古いものから消える
recent_logs.append("2026-03-27 GET /api/users")
recent_logs.append("2026-03-27 POST /api/orders")
```
