# 並行性の基本概念

> **一言で言うと:** 「2つ以上の処理が同時に動くとき何が起きるか」を理解し、競合状態（Race Condition）やデッドロック（Deadlock）といった並行処理特有の問題を予測・回避するための基礎知識。

## なぜ必要か

現代のソフトウェアは、1つの処理を順番に実行するだけでは成り立たない。Webサーバーは何千ものリクエストを同時に捌き、データベースは複数のトランザクションを並行して処理する。JavaScriptの `async/await` も、Node.js のイベントループも、すべて「複数の処理をどう協調させるか」という問題の上に成り立っている。

この基礎を知らないと：
- 非同期処理で「たまにしか起きないバグ」に悩まされる
- データベースのトランザクション分離レベルを正しく選べない
- マルチスレッド環境で共有データを壊す
- 本番環境でしか再現しない不具合をデバッグできない

## どの問題を解決するか

### 1. 競合状態（Race Condition）
**課題:** 複数の処理が同じデータに同時にアクセスし、実行順序によって結果が変わってしまう。

```
スレッドA: balance を読む → 1000
スレッドB: balance を読む → 1000
スレッドA: balance = 1000 - 200 → 800 を書く
スレッドB: balance = 1000 - 300 → 700 を書く
結果: 500 であるべきが 700 になる（200円の引き落としが消失）
```

**解決:** [[ロック]]（Lock）、ミューテックス（Mutex）、アトミック操作（Atomic Operation）で「同時アクセスを制御」する。

### 2. [[デッドロック]]（Deadlock）
**課題:** 2つ以上の処理が互いのロック解放を待ち、永遠に進まなくなる。

```
スレッドA: リソースXをロック → リソースYを待つ
スレッドB: リソースYをロック → リソースXを待つ
→ 両方とも永遠に待ち続ける
```

**解決:** ロック順序の統一、タイムアウト付き[[ロック]]、[[デッドロック]]検出アルゴリズム。

### 3. 飢餓状態（Starvation）
**課題:** 特定の処理がリソースを永遠に獲得できず、実行が進まない。優先度の高いスレッドがリソースを占有し続けると発生する。

**解決:** 公平なスケジューリング（Fair Scheduling）、優先度逆転対策（Priority Inheritance）。

### 4. データの可視性問題（Visibility Problem）
**課題:** あるスレッドが変数を更新しても、別のスレッドからはCPUキャッシュの影響で古い値が見える。

**解決:** メモリバリア（Memory Barrier）、`volatile` 宣言、同期プリミティブの使用。

## 他の仕組みとどう関係するか

- **下位レイヤーとの関係:**
  - ハードウェアレベルのCPUキャッシュコヒーレンス、メモリモデルが並行性の基盤となる
  - [[データ構造とアルゴリズム]] — スレッドセーフなデータ構造（Concurrent HashMap等）は内部で並行制御を実装している
  - [[計算量-BigO]] — ロックの粒度がスループットに影響し、並行処理の実効的な計算量を変える

- **同レイヤーとの関係:**
  - 並行性は CS の基礎理論として、非決定性やチューリングマシンの拡張（並行計算モデル）と関連する

- **上位レイヤーとの関係:**
  - → [[プロセスとスレッド]]（Layer 1）: OS がどのように並行実行を実現するかの実装
  - → [[イベントループ]]（Layer 2）: JavaScript/Node.js の並行モデルの核心
  - → [[非同期処理]]（Layer 2）: Promise, async/await は並行性を扱うための抽象
  - → [[トランザクション]]（Layer 3）: DBの並行アクセス制御に直結（ACID特性のIsolation）
  - → [[コネクションプール]]（Layer 3）: 共有リソースの並行管理

## 誤解されやすいポイント

### 1. 「並行（Concurrent）」と「並列（Parallel）」は同じ
**実際:** 並行は「複数の処理が進行中である」こと。並列は「複数の処理が物理的に同時に実行される」こと。シングルコアCPUでも、タイムスライシングによって並行処理は可能だが、並列処理はできない。コア数とスレッド数の組み合わせによる違いの詳細は [[シングルコア・マルチコアとスレッドモデル]] を参照。

この違いが明確に現れるのが Node.js と Go の対比である。Node.js はシングルスレッドのイベントループで並行処理を実現するが、CPU コアは1つしか使わないため並列ではない（Worker Threads を除く）。一方 Go は、Goroutine が Go ランタイムのスケジューラによって複数の OS スレッドに割り当てられ、マルチコア上で物理的に同時実行される。つまり Go は並行かつ並列である。Rob Pike の有名な言葉「Concurrency is not parallelism」はまさにこの区別を指しており、Go は言語設計として両方をサポートしている。

### 2. 「JavaScript はシングルスレッドだから競合状態は起きない」
**実際:** JavaScript のイベントループは確かにシングルスレッドだが、非同期処理の実行順序に依存するロジックを書けば、論理的な競合状態は発生する。例えば、2つの `fetch` の結果を共有変数に書き込む場合、どちらが先に完了するかで結果が変わりうる。

```javascript
let latestData = null;
// 2つのリクエストが「並行」して実行される
fetch('/api/v1').then(r => r.json()).then(data => { latestData = data; });
fetch('/api/v2').then(r => r.json()).then(data => { latestData = data; });
// latestData がどちらの結果になるかは非決定的
```

### 3. 「[[ロック]]を使えば安全」
**実際:** ロック自体が新たな問題（[[デッドロック]]、パフォーマンス低下、優先度逆転）を引き起こす。ロックの粒度が粗すぎるとスループットが落ち、細かすぎるとデッドロックのリスクが上がる。ロックフリー（Lock-free）アルゴリズムやメッセージパッシングなど、ロックを使わない並行制御手法も重要。

### 4. 「非同期 = 並行 = マルチスレッド」
**実際:** この3つは全く異なる概念。非同期（Asynchronous）は「完了を待たずに次に進む」という実行モデル。並行（Concurrent）は「複数の処理を同時に扱う」という構造。マルチスレッドは並行性を実現する1つの手段に過ぎない。イベントループ、コルーチン、アクターモデルなど、マルチスレッド以外の並行モデルも存在する。

## 設計のベストプラクティス

### 推奨パターン

1. **共有状態を最小化する（Share Nothing）**
   最も安全な並行プログラムは、データを共有しないプログラム。メッセージパッシングやイミュータブルデータを使い、共有ミュータブル状態を避ける。

2. **[[ロック]]の粒度を適切に設計する**
   データベース全体をロックするのではなく、行レベルロックやオプティミスティックロック（楽観的ロック）を使う。

3. **ロック順序を統一する**
   複数のリソースをロックする場合、全スレッドで同じ順序でロックを取得する。これだけで[[デッドロック]]の大半を防げる。

4. **イミュータブルなデータ構造を活用する**
   変更不可能なデータは、ロックなしで安全に共有できる。

5. **適切な並行モデルを選択する**
   - I/O バウンド → イベントループ / async-await（Node.js, Python asyncio）
   - CPU バウンド → マルチプロセス / Worker Threads
   - 複雑な状態管理 → アクターモデル（Erlang/Elixir, Akka）

### アンチパターン

- **過度な共有状態:** グローバル変数を複数スレッドから読み書きする
- **ロックの入れ子:** デッドロックの温床になる
- **busy-wait（スピンロック）の乱用:** CPU リソースを浪費する
- **sleep による同期:** `sleep(100)` で「たぶん終わってるだろう」は事故の元

## AIによる実装のアンチパターン

| アンチパターン | なぜ問題か | 対策 |
|---|---|---|
| 全てにミューテックスを付ける | 不要なロックはパフォーマンスを殺し、デッドロックリスクを増やす | 本当に共有される状態だけをロックする |
| `sleep()` で競合を「回避」 | タイミング依存で本番では破綻する | 適切な同期プリミティブ（セマフォ、条件変数）を使う |
| async/await の過剰な直列化 | `await` を逐次的に並べて並行性の利点を殺す | 独立した処理は `Promise.all()` で並行実行 |
| エラーハンドリングなしの並行処理 | 1つの失敗が他の処理を巻き込む | 個別のエラーハンドリングと適切なキャンセル処理 |
| スレッドプールサイズのハードコード | 環境によって最適値が異なる | CPU コア数ベースの動的設定、または設定可能にする |

## 具体例

### 競合状態の例と対策（Python）

```python
import threading

# 危険: 競合状態が発生する
counter = 0

def increment_unsafe():
    global counter
    for _ in range(100000):
        counter += 1  # read-modify-write は原子的でない

# 安全: Lock で排他制御
lock = threading.Lock()
safe_counter = 0

def increment_safe():
    global safe_counter
    for _ in range(100000):
        with lock:  # with 文でロックの取得・解放を保証
            safe_counter += 1

# 競合状態のデモ
threads = [threading.Thread(target=increment_unsafe) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
print(f"期待値: 400000, 実際: {counter}")  # 400000 未満になりうる

# 安全な版
threads = [threading.Thread(target=increment_safe) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
print(f"期待値: 400000, 実際: {safe_counter}")  # 常に 400000
```

### JavaScript での論理的競合状態

```javascript
// 問題: 古いリクエストの結果が新しいリクエストを上書きする
let currentRequestId = 0;

async function searchUnsafe(query) {
  const results = await fetch(`/api/search?q=${query}`).then(r => r.json());
  displayResults(results); // 古いリクエストが後から返ると上書きされる
}

// 解決: リクエストIDで最新のものだけ反映する
async function searchSafe(query) {
  const requestId = ++currentRequestId;
  const results = await fetch(`/api/search?q=${query}`).then(r => r.json());
  if (requestId === currentRequestId) {
    displayResults(results); // 最新のリクエストのみ反映
  }
}
```

### デッドロックの例（Go）

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

// デッドロックが発生する例
func deadlockExample() {
	var muA, muB sync.Mutex

	// Goroutine 1: A → B の順でロック
	go func() {
		muA.Lock()
		time.Sleep(10 * time.Millisecond)
		muB.Lock() // Goroutine 2 が muB を持っている → 待ち
		fmt.Println("Goroutine 1 完了")
		muB.Unlock()
		muA.Unlock()
	}()

	// Goroutine 2: B → A の順でロック（逆順 → デッドロック）
	go func() {
		muB.Lock()
		time.Sleep(10 * time.Millisecond)
		muA.Lock() // Goroutine 1 が muA を持っている → 待ち
		fmt.Println("Goroutine 2 完了")
		muA.Unlock()
		muB.Unlock()
	}()

	time.Sleep(1 * time.Second)
	fmt.Println("タイムアウト: デッドロックの可能性")
}

// 解決策: ロック順序を統一する（常に A → B の順）
func noDeadlockExample() {
	var muA, muB sync.Mutex

	acquire := func(name string) {
		muA.Lock()
		muB.Lock()
		fmt.Printf("%s 完了\n", name)
		muB.Unlock()
		muA.Unlock()
	}

	go acquire("Goroutine 1")
	go acquire("Goroutine 2")
	time.Sleep(1 * time.Second)
}
```

## 学習メモ

- 並行性の問題は「再現困難」なことが本質的な難しさ。テストで見つけるのではなく、設計で防ぐことが重要
- Web開発では直接スレッドを扱う機会は少ないが、DB トランザクション、キャッシュの整合性、API の冪等性など、並行性の概念が姿を変えて頻出する
- 「共有状態を減らす」が最も効果的な戦略であり、関数型プログラミングのイミュータブル志向とも通じる
