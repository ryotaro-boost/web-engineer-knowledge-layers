# 計算量（Big-O）

> **一言で言うと:** 「N が100万になったとき壊れるか？」を事前に判断する道具。入力サイズに対して処理時間やメモリがどう増えるかを表す記法。

## なぜ必要か

コードは「動く」だけでは不十分で、「規模が大きくなっても動く」ことが求められる。計算量の概念がないと、次の問題が起きる。

- **本番環境で突然遅くなる** — 開発中はデータ100件で快適に動いていたAPIが、本番で10万件になった途端にタイムアウトする。原因は O(n²) のループだが、計算量を見る習慣がないと気づけない。
- **ボトルネックの特定ができない** — 「遅い」ことはわかっても、どこが遅いのか・なぜ遅いのかを論理的に説明できない。プロファイラの結果を読んでも「なぜこの関数が遅いのか」の根本原因に辿り着けない。
- **技術選定の判断基準がない** — 配列を使うか Map を使うか、線形探索かバイナリサーチか、DB にインデックスを貼るべきか。計算量を知らないと「なんとなく」で選ぶしかなくなる。

## どの問題を解決するか

### 問題1: コードの性能を「実行前に」見積もる

計算量は、実際にベンチマークを取らなくても、コードの構造を見るだけで性能の傾向を予測できる道具である。

| 記法 | 名前 | 増え方 | n=1,000 での操作回数目安 | 実務での典型例 |
|------|------|--------|----------------------|-------------|
| O(1) | 定数時間 | 入力に依存しない | 1 | ハッシュテーブルの検索、配列のインデックスアクセス |
| O(log n) | 対数時間 | 倍々に増えても少ししか増えない | ~10 | 二分探索、B-Tree のインデックス検索 |
| O(n) | 線形時間 | 入力に比例 | 1,000 | 配列の全走査、`Array.find()` |
| O(n log n) | 準線形時間 | 線形よりやや悪い | ~10,000 | ソート（Merge Sort, Quick Sort） |
| O(n²) | 二次時間 | 入力の2乗 | 1,000,000 | 二重ループ、バブルソート |
| O(2ⁿ) | 指数時間 | 爆発的増加 | 天文学的数値 | 全組み合わせの探索（最適化なし） |

**読み方のポイント:** O(n) と O(n²) の差は、n=100 なら 100 vs 10,000（100倍）だが、n=100,000 なら 100,000 vs 10,000,000,000（10万倍）になる。**差は n が大きいほど広がる。**

### 問題2: ループのネストから「爆発」を予測する

計算量を見積もるもっとも実用的なスキルは、コードのループ構造から計算量を読み取ることである。

```
単一ループ       → O(n)
ネストしたループ  → O(n²)
ループの中で二分探索 → O(n log n)
再帰で2分岐     → O(2ⁿ) の可能性
```

### 問題3: 空間計算量（Space Complexity）も考慮する

時間だけでなく、メモリ消費も計算量で表せる。

- 入力をそのまま処理する → O(1) の追加空間
- 入力サイズ分の配列をコピーする → O(n) の追加空間
- 二次元テーブルを作る（DP など） → O(n²) の追加空間

**使用シーン:** フロントエンドで巨大なリストをコピーし続ける `[...arr, newItem]` パターンは、繰り返すと O(n²) の空間を消費する。

## 他の仕組みとどう関係するか

- **下位レイヤーとの関係:** 計算量は CPU サイクルとメモリ消費の抽象モデルである。実際の速度は CPU キャッシュのヒット率やメモリアクセスパターンにも左右されるため、計算量だけでは語れない部分もある（定数倍の差）。
- **同レイヤーとの関係:** [[データ構造とアルゴリズム]] の各操作の性能を評価する「共通言語」。データ構造を選ぶ判断基準そのものが計算量である。[[並行性の基本概念]] では、ロックの粒度や競合の頻度がスループットに影響するが、これも計算量的な思考で分析できる。
- **上位レイヤーとの関係:** Layer 3 の [[インデックス]] が B-Tree で O(log n) の検索を実現する理由は計算量で説明される。Layer 5 の [[パフォーマンス最適化]] では、計算量の改善が最も効果の大きい最適化手法となる。Layer 4 の API 設計で [[ページネーション]] がなぜ必要かも、全件取得が O(n) のレスポンスサイズを生むことで説明できる。

## 誤解されやすいポイント

1. **「O(n²) は絶対に悪い」という決めつけ** — n が小さい（数十件以下）なら O(n²) でも問題ない。管理画面のプルダウン選択肢10件をソートするのに、計算量を気にして複雑なコードにするのは本末転倒。**n がどこまで大きくなりうるか** を考えることが重要。

2. **「Big-O が同じなら同じ速度」という誤解** — O(n) 同士でも、定数倍の差で10倍速い・遅いことがある。例えば、配列の走査はメモリが連続しているためキャッシュ効率が良く、リンクリストの走査より高速。Big-O は「n を十分大きくしたときの増加傾向」であり、実際の速度ではない。

3. **「平均計算量 = 常にその速度」と思い込む** — ハッシュテーブルの検索は平均 O(1) だが、最悪 O(n) になりうる。「平均」は多数の操作を均した値であり、個々の操作では最悪ケースが起こりうる。リアルタイム性が求められるシステム（ゲーム、金融）では最悪計算量も重要。

4. **「計算量を改善すれば必ず速くなる」** — 理論上は O(n²) → O(n log n) が改善だが、n が小さいとき、定数倍が大きいアルゴリズムに変えるとかえって遅くなることがある。また、実務のボトルネックは計算量ではなく I/O（ネットワーク遅延、ディスクアクセス）であることも多い。**プロファイルしてからの最適化** が鉄則。

5. **「Big-O の O は Order の略」程度の理解で使っている** — Big-O は厳密には「上界（upper bound）」を表す。O(n) のアルゴリズムが O(n²) でもあると言えるのはそのため（上界がゆるい）。実務で厳密な数学的定義を使う場面は少ないが、「最悪でもこの程度」という意味であることは理解しておく。

## 設計のベストプラクティス

### 推奨パターン

- **まず n の範囲を把握する** — 「このデータは最大何件になるか？」を最初に確認する。n ≤ 100 なら計算量はほぼ気にしなくていい。n ≥ 10,000 なら O(n²) は要注意。n ≥ 1,000,000 なら O(n log n) 以下を目指す。
- **ネストしたループを見たら警戒する** — 二重ループ = O(n²) を自動的に疑う癖をつける。ただし内側のループが固定長（例：曜日7回）なら O(7n) = O(n) なので問題ない。
- **ハッシュマップで O(n²) → O(n) に落とす** — 2つのリストの照合は、一方を Map にすれば二重ループを解消できる。Web 開発で最も頻繁に使える改善テクニック。
- **DB のクエリを計算量で考える** — `WHERE id = ?` + インデックスは O(log n)。`WHERE name LIKE '%keyword%'` はフルスキャン O(n)。N+1 問題はループ内クエリで O(n) 回の I/O を発生させている。

### アンチパターン

- **「とりあえず全件取得」** — API で全レコードを返し、フロントエンドでフィルタリングする設計は、データ量が増えると転送量 O(n) とレンダリング O(n) の両方が爆発する。
- **ループ内で配列の `includes` / `indexOf`** — ループ O(n) × 探索 O(n) = O(n²)。Set や Map に置き換える。
- **再帰でメモ化を忘れる** — [[フィボナッチ数列]]を素朴な再帰で書くと O(2ⁿ)。メモ化するだけで O(n) になる。

## AIによる実装のアンチパターン

LLM に「パフォーマンスを意識したコード」を求めると、計算量への過剰反応が起きやすい。

| アンチパターン | なぜ問題か | 対策 |
|---|---|---|
| **早すぎるキャッシュ導入** — 数十件のデータ取得に対して Redis キャッシュやメモ化レイヤーを最初から組み込む | キャッシュは整合性管理（TTL、invalidation）というコストを伴う。n が小さいうちは単純な処理の方が保守しやすい | プロファイルで実際にボトルネックを確認してから導入する。「N が将来大きくなる **かもしれない**」は導入理由にならない |
| **小さなデータセットへの過剰な最適化** — 管理画面の選択肢20件に対して `Map` 化や `Set` 化を行い、O(n) → O(1) の改善を図る | 定数時間の差が無視できるレベル。コードの意図が読みにくくなるだけ | n の現実的な上限を見積もり、O(n²) でも問題ないサイズなら素直に書く |
| **フォールバックの多重実装** — API レスポンスが遅い場合のフォールバック、キャッシュミス時のフォールバック、DB 接続失敗時のフォールバックを全て事前に実装する | 起こるかわからない障害に備えて複雑性を大量に追加している。フォールバック自体のテストも必要になり工数が倍増する | 実際に起きている問題・起きる確率が高い問題だけに対処する。「もし〜なら」の連鎖で防御コードを積み重ねない |
| **計算量コメントの過剰付与** — 全ての関数に `// O(n)` `// O(1)` とコメントをつける | 明らかな処理（単一ループ、ハッシュ検索）にコメントは不要。ノイズになる | 非自明な計算量（O(n²) が隠れている場合など）にだけ注記する |

## 具体例

### 例1: ループ構造から計算量を読み取る

**使用シーン:** コードレビューで「この処理、データ増えたら大丈夫？」と聞かれたときの判断。

```javascript
// O(n) — 単一ループ
function sum(arr) {
  let total = 0;
  for (const num of arr) {
    total += num;  // n 回実行される
  }
  return total;
}

// O(n²) — ネストしたループ（両方が入力サイズに依存）
function findDuplicatePairs(arr) {
  const pairs = [];
  for (let i = 0; i < arr.length; i++) {         // n 回
    for (let j = i + 1; j < arr.length; j++) {    // 最大 n-1 回
      if (arr[i] === arr[j]) pairs.push([i, j]);
    }
  }
  return pairs;
}

// O(n) — ループがネストしていても内側が固定長なら定数
function formatWeeklySchedule(weeks) {
  const days = ["月", "火", "水", "木", "金", "土", "日"];
  return weeks.map(week =>               // n 回（週の数）
    days.map(day => `${day}: ${week[day] ?? "休み"}`).join(", ")
    // ↑ 常に7回 → O(7) = O(1)
  );
}
// 全体: O(n × 7) = O(n)
```

### 例2: O(n²) → O(n) への改善

**使用シーン:** バックエンドで2つのテーブル（注文リストとユーザーリスト）を結合して返すAPI。

```python
# ❌ O(n × m) — 二重ループ
def enrich_orders(orders, users):
    result = []
    for order in orders:                    # n 回
        for user in users:                  # m 回
            if user["id"] == order["user_id"]:
                result.append({
                    **order,
                    "user_name": user["name"]
                })
                break
    return result

# ✅ O(n + m) — ハッシュマップで結合
def enrich_orders(orders, users):
    user_map = {u["id"]: u for u in users}  # O(m) で構築
    return [
        {**order, "user_name": user_map[order["user_id"]]["name"]}
        for order in orders                  # O(n) で走査
        if order["user_id"] in user_map
    ]
```

### 例3: 再帰の計算量とメモ化 — [[フィボナッチ数列]]

**使用シーン:** 動的計画法が必要な場面（料金計算の組み合わせ最適化、文字列の類似度計算など）。

```javascript
// ❌ O(2ⁿ) — 素朴な再帰（同じ計算を何度も繰り返す）
function fib(n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2);
}
// fib(50) → 数分〜数時間かかる

// ✅ O(n) — メモ化で重複計算を排除
function fibMemo(n, memo = {}) {
  if (n <= 1) return n;
  if (memo[n]) return memo[n];
  memo[n] = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
  return memo[n];
}
// fibMemo(50) → 即座に完了
```

### 例4: 実務で遭遇する計算量の問題 — N+1 クエリ

**使用シーン:** ORM を使ったバックエンド開発で最も頻繁に遭遇するパフォーマンス問題。

```ruby
# ❌ N+1問題 — O(n) 回のDBクエリ
# 1回目: SELECT * FROM posts（全記事取得）
# 2〜N+1回目: 各記事ごとに SELECT * FROM users WHERE id = ?
posts = Post.all
posts.each do |post|
  puts post.user.name  # ← 毎回SQLが発行される
end

# ✅ Eager Loading — O(1) 回の追加クエリ
# 1回目: SELECT * FROM posts
# 2回目: SELECT * FROM users WHERE id IN (1, 2, 3, ...)
posts = Post.includes(:user).all
posts.each do |post|
  puts post.user.name  # ← メモリ上のデータを参照するだけ
end
```

> この例では計算量というよりI/Oの回数が問題だが、「ループの中で重い処理を呼ぶ → O(n) 回の I/O」という構造を見抜く力は計算量的思考そのもの。
