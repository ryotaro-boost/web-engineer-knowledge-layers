# 演習セットアップ — Go（net/http）

> **目的:** [[Layer7-設計アーキテクチャ/_practice/_index|実践演習]]を Go で取り組むための、最小かつ確実に動く scratch project の構築手順。

## なぜフレームワークではなく標準ライブラリか

Go コミュニティでは Web フレームワークを使わず標準ライブラリの `net/http` で書くのが一般的。Express / Slim 相当の薄いルータを書きたい場合でも、`chi` や `gorilla/mux` 程度の選択にとどまる。**Go 1.22 以降は `net/http` 自体にメソッド指定ルーティング**が入ったため、追加ライブラリすら不要。

加えて、Go の言語特性として「インターフェースを使う側で定義する」慣習が [[オニオンアーキテクチャ]] の「インターフェースは Domain 層」というルールと**思想的に衝突**する。これは TS や PHP では発生しない、**Go 固有の学習論点**。フレームワークを介さずに標準ライブラリで書くと、この衝突が見えやすい。

## 前提

- Go 1.22 以降がインストール済み（`go version`）
- Docker が動く環境（Postgres 起動用）

## 1. プロジェクト初期化

```bash
mkdir practice-01-user-registration && cd practice-01-user-registration
go mod init practice/user-registration
go mod edit -go=1.22
```

`go.mod` が生成される。これで GOPATH に依らない「モジュール」としてビルドできる。

> **`go mod edit -go=1.22`:** `go mod init` が生成する `go.mod` の `go` ディレクティブは実行環境の Go バージョンに依存する。本演習では **メソッド指定ルーティング**（`http.HandleFunc("POST /users", ...)`）を使うので、Go 1.22 以降が必須。明示的に指定しておくと、後で 1.21 環境で `go run` した時に「文法は通るがルーティングが効かない」という分かりにくい症状を防げる。

## 2. 依存パッケージの取得

```bash
go get github.com/lib/pq
go get golang.org/x/crypto/bcrypt
go get github.com/google/uuid
```

各パッケージの役割:

| パッケージ | 役割 |
|----------|------|
| `github.com/lib/pq` | Postgres ドライバ（`database/sql` インターフェース実装） |
| `golang.org/x/crypto/bcrypt` | パスワードハッシュ |
| `github.com/google/uuid` | UUID v4 生成 |

> **lib/pq vs pgx:** モダン Go では `github.com/jackc/pgx/v5` が推奨されつつあるが、`database/sql` 標準インターフェースで書きたいなら lib/pq の方が学習コストが低い。慣れてから pgx に移行するのが現実的。

> **JSON 処理は標準ライブラリ:** `encoding/json` で十分。外部依存は不要。

## 3. Postgres を Docker で起動

TypeScript / PHP 版と同じ:

```bash
docker run -d --name practice-pg \
  -p 5432:5432 \
  -e POSTGRES_PASSWORD=test \
  -e POSTGRES_DB=practice \
  postgres:16
```

## 4. テーブル作成

演習 01 用のスキーマ:

```bash
docker exec -i practice-pg psql -U postgres -d practice <<'EOF'
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  name TEXT NOT NULL
);
EOF
```

## 5. 環境変数

Go では `os.Getenv()` で環境変数を読む。`.env` を使いたい場合は `github.com/joho/godotenv` を追加するが、練習では起動時に環境変数を渡す方が手軽:

```bash
DATABASE_URL="postgres://postgres:test@localhost:5432/practice?sslmode=disable" \
  go run ./cmd/server
```

> **`sslmode=disable`:** ローカル開発では SSL 不要なので明示的に無効化する。本番では `verify-full` を指定する。

## 6. 推奨ディレクトリ構成（refactor 後）

```
practice-01-user-registration/
├── go.mod
├── go.sum
├── cmd/
│   └── server/
│       └── main.go         # Composition Root
├── domain/                 # Onion で refactor したら作る
│   ├── user.go
│   ├── email.go
│   ├── repository.go       # interface 定義（Onion 流）
│   ├── password_hasher.go
│   └── mailer.go
├── application/
│   └── user_service.go
└── infrastructure/
    ├── postgres_user_repository.go
    ├── bcrypt_password_hasher.go
    └── console_mailer.go
```

> **Go の命名規則:** ディレクトリ名・ファイル名は小文字スネークケース（`user_service.go`）。パッケージ名はディレクトリ名と一致させ、すべて小文字。型名・関数名は **公開なら CamelCase、非公開なら lowerCamelCase**。

> **`cmd/server/`:** Go では実行可能なエントリポイントを `cmd/<binary-name>/main.go` に置く慣習。これにより同じプロジェクトに複数のバイナリを生やせる（例: `cmd/server/`, `cmd/migrate/`）。

## 7. 起動


## Go 固有の学習論点

### A. インターフェースを「使う側」で定義する慣習との衝突

Go の Effective Go では:

> The bigger the interface, the weaker the abstraction.
> Accept interfaces, return structs.

これに従うと `application/user_service.go` の中で `UserRepository` インターフェースを定義し、`infrastructure/postgres_user_repository.go` がそれを **暗黙的に**実装する形になる。

```go
// application/user_service.go — Go 慣習：使う側で定義
package application

type UserRepository interface {
    FindByEmail(email string) (*domain.User, error)
    Save(user *domain.User) error
}

type UserService struct { repo UserRepository }
```

しかし [[オニオンアーキテクチャ]] のルールは **「インターフェースは Domain 層」**。つまり:

```go
// domain/repository.go — Onion 流：Domain 層に置く
package domain

type UserRepository interface {
    FindByEmail(email string) (*User, error)
    Save(user *User) error
}
```

両者は矛盾する。**演習 01 では Onion ルールに従って Domain 層に置く** が、refactor 中に「Go 慣習に逆らっている感じ」を体感するはず。それが Go ならではの学習価値。

> **どちらが正しいのか:** プロダクトコードでは「Domain 層が他のすべての層から使われるなら、Domain で定義する方が Go 慣習にも合う」という解釈が現実的。要は「最も使われるパッケージで定義する」を Go の暗黙ルールとし、それが Onion でいう Domain になる、と整理すれば矛盾しない。

### B. errors.Is / errors.As によるエラー判別

Go では try-catch がなく、エラーは戻り値。Domain で定義した sentinel error を Application で判別する:

```go
// domain/repository.go
var ErrUserNotFound = errors.New("user not found")

// application/user_service.go
if errors.Is(err, domain.ErrUserNotFound) {
    return ErrEmailAlreadyRegistered  // Application 層のエラーに変換
}
```

これにより **エラーが層の境界で適切に翻訳される**。これは Onion / Clean の境界翻訳の Go 版実装。

## トラブルシューティング

### `go: command not found`

Go がインストールされていない。[公式](https://go.dev/dl/) から最新版を入れる。Go 1.22+ が必要（メソッド指定ルーティングのため）。

### `cannot find module providing package`

`go mod tidy` で `go.mod` と実際の import を同期する:

```bash
go mod tidy
```

### `pq: SSL is not enabled on the server`

接続文字列に `?sslmode=disable` を付け忘れている。

### `http: method not allowed` / メソッド指定ルーティングが効かない

Go 1.22 未満。`go version` で確認し、`go mod edit -go=1.22` で更新する。古い Go ではハンドラ内で `if r.Method != http.MethodPost {...}` を書く必要がある。
