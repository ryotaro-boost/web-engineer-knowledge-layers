# 演習 01 — ユーザー登録（User Registration）

> **学習する論点:** 副作用（メール送信・パスワードハッシュ）をどの層に置くか、Application Service が何を「オーケストレーション」するのかを体感する。

> **環境構築:** [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-TypeScript|00_セットアップ-TypeScript]] を先に通しておくこと。Postgres 起動、`users` テーブル作成、依存パッケージのインストールがそこにまとまっている。

## ① お題

ECサイトにユーザー登録機能を実装する。次の処理を行う:

1. 入力バリデーション（メール形式、パスワード長 8 文字以上、名前必須）
2. メールアドレスの重複チェック
3. パスワードを bcrypt でハッシュ化
4. DB に保存
5. ウェルカムメール送信
6. 作成したユーザーの `id`, `email`, `name` を 201 で返す

エラーは適切な HTTP ステータスで返す（バリデーション失敗 400、重複 409）。

## ② 悪い例コード — まずこれを写経する

全ての関心が 1 ハンドラに詰まっている。これが scratch project の出発点。**自分が取り組む言語のセクション**を選んでコードをコピーし、必ず動く形で写経する。テストや CI を整える必要はない。`curl` で 201 が返ることだけ確認できれば十分。

> **なぜ写経が必要か:** 動いたコードを refactor する体験が大事。最初から綺麗なコードを書こうとすると、「悪さ」を肌で感じないまま終わる。

> **メール送信の扱い:** 本番では SendGrid / SES / Mailgun 等の外部サービスを呼ぶが、練習では `console.log` / `error_log` / `log.Printf` で代用する。「悪い例の悪さ」を維持したまま外部依存を消すための現実的な妥協。Refactor 後にこのログ出力がどの層に置かれるべきか考えるのが、後半の設計判断ポイントの 1 つ。

### TypeScript 版（Express）

参照: [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-TypeScript|00_セットアップ-TypeScript]]

```typescript
// src/server.ts
import express from "express";
import { Pool } from "pg";
import * as bcrypt from "bcrypt";
import * as crypto from "crypto";

const app = express();
app.use(express.json());

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

app.post("/users", async (req, res) => {
  const { email, password, name } = req.body;

  // バリデーション
  if (!email || typeof email !== "string" || !email.includes("@")) {
    return res.status(400).json({ error: "Invalid email" });
  }
  if (!password || typeof password !== "string" || password.length < 8) {
    return res.status(400).json({ error: "Password must be at least 8 chars" });
  }
  if (!name || typeof name !== "string" || name.length < 1) {
    return res.status(400).json({ error: "Name required" });
  }

  // 重複チェック
  const exists = await pool.query("SELECT id FROM users WHERE email = $1", [email]);
  if (exists.rows.length > 0) {
    return res.status(409).json({ error: "Email already registered" });
  }

  // パスワードハッシュ
  const passwordHash = await bcrypt.hash(password, 10);

  // DB 保存
  const userId = crypto.randomUUID();
  await pool.query(
    "INSERT INTO users (id, email, password_hash, name) VALUES ($1, $2, $3, $4)",
    [userId, email, passwordHash, name],
  );

  // ウェルカムメール（練習中は console.log で代用）
  console.log(
    `[mail] To: ${email} / Subject: ようこそ / Body: ${name} さん、登録ありがとうございます`,
  );

  res.status(201).json({ id: userId, email, name });
});

app.listen(3000, () => console.log("listening on :3000"));
```

**起動:**

```bash
DATABASE_URL=postgres://postgres:test@localhost:5432/practice npx tsx src/server.ts
```

### PHP 版（Slim 4）

参照: [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-PHP|00_セットアップ-PHP]]

```php
<?php
// public/index.php
require __DIR__ . '/../vendor/autoload.php';

use Slim\Factory\AppFactory;
use Psr\Http\Message\ResponseInterface as Response;
use Psr\Http\Message\ServerRequestInterface as Request;
use Ramsey\Uuid\Uuid;

$dotenv = Dotenv\Dotenv::createImmutable(__DIR__ . '/..');
$dotenv->load();

$pdo = new PDO(
    $_ENV['DATABASE_URL'],
    $_ENV['DATABASE_USER'],
    $_ENV['DATABASE_PASSWORD'],
    [PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION],
);

$app = AppFactory::create();
$app->addBodyParsingMiddleware();

$json = function (Response $r, int $status, array $data): Response {
    $r->getBody()->write(json_encode($data));
    return $r->withStatus($status)->withHeader('Content-Type', 'application/json');
};

$app->post('/users', function (Request $req, Response $res) use ($pdo, $json) {
    $body = $req->getParsedBody() ?? [];
    $email = $body['email'] ?? null;
    $password = $body['password'] ?? null;
    $name = $body['name'] ?? null;

    // バリデーション
    if (!is_string($email) || !str_contains($email, '@')) {
        return $json($res, 400, ['error' => 'Invalid email']);
    }
    if (!is_string($password) || strlen($password) < 8) {
        return $json($res, 400, ['error' => 'Password must be at least 8 chars']);
    }
    if (!is_string($name) || strlen($name) < 1) {
        return $json($res, 400, ['error' => 'Name required']);
    }

    // 重複チェック
    $stmt = $pdo->prepare('SELECT id FROM users WHERE email = :email');
    $stmt->execute(['email' => $email]);
    if ($stmt->fetch()) {
        return $json($res, 409, ['error' => 'Email already registered']);
    }

    // パスワードハッシュ
    $passwordHash = password_hash($password, PASSWORD_BCRYPT, ['cost' => 10]);

    // DB 保存
    $userId = Uuid::uuid4()->toString();
    $stmt = $pdo->prepare(
        'INSERT INTO users (id, email, password_hash, name) VALUES (:id, :email, :hash, :name)'
    );
    $stmt->execute([
        'id' => $userId,
        'email' => $email,
        'hash' => $passwordHash,
        'name' => $name,
    ]);

    // ウェルカムメール（練習中は error_log で代用）
    error_log("[mail] To: $email / Subject: ようこそ / Body: $name さん、登録ありがとうございます");

    return $json($res, 201, ['id' => $userId, 'email' => $email, 'name' => $name]);
});

$app->run();
```

**起動:**

```bash
php -S localhost:3000 -t public
```

> **PSR-7 の immutability:** `Response` は immutable なので `withStatus()` `withHeader()` の戻り値を必ず return する。書きづらいので上のコードでは `$json` ヘルパでまとめている。これも refactor 候補の「臭い」の 1 つ。

### Go 版（net/http）

参照: [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-Go|00_セットアップ-Go]]

```go
// cmd/server/main.go
package main

import (
	"database/sql"
	"encoding/json"
	"errors"
	"log"
	"net/http"
	"os"
	"strings"

	"github.com/google/uuid"
	_ "github.com/lib/pq"
	"golang.org/x/crypto/bcrypt"
)

type registerRequest struct {
	Email    string `json:"email"`
	Password string `json:"password"`
	Name     string `json:"name"`
}

type registerResponse struct {
	ID    string `json:"id"`
	Email string `json:"email"`
	Name  string `json:"name"`
}

func writeJSON(w http.ResponseWriter, status int, body any) {
	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(status)
	_ = json.NewEncoder(w).Encode(body)
}

func main() {
	db, err := sql.Open("postgres", os.Getenv("DATABASE_URL"))
	if err != nil {
		log.Fatal(err)
	}
	defer db.Close()

	http.HandleFunc("POST /users", func(w http.ResponseWriter, r *http.Request) {
		var req registerRequest
		if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
			writeJSON(w, http.StatusBadRequest, map[string]string{"error": "Invalid JSON"})
			return
		}

		// バリデーション
		if !strings.Contains(req.Email, "@") {
			writeJSON(w, http.StatusBadRequest, map[string]string{"error": "Invalid email"})
			return
		}
		if len(req.Password) < 8 {
			writeJSON(w, http.StatusBadRequest, map[string]string{"error": "Password must be at least 8 chars"})
			return
		}
		if len(req.Name) < 1 {
			writeJSON(w, http.StatusBadRequest, map[string]string{"error": "Name required"})
			return
		}

		// 重複チェック
		var existing string
		err := db.QueryRow("SELECT id FROM users WHERE email = $1", req.Email).Scan(&existing)
		if err != nil && !errors.Is(err, sql.ErrNoRows) {
			writeJSON(w, http.StatusInternalServerError, map[string]string{"error": err.Error()})
			return
		}
		if err == nil {
			writeJSON(w, http.StatusConflict, map[string]string{"error": "Email already registered"})
			return
		}

		// パスワードハッシュ
		hash, err := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)
		if err != nil {
			writeJSON(w, http.StatusInternalServerError, map[string]string{"error": err.Error()})
			return
		}

		// DB 保存
		userID := uuid.New().String()
		_, err = db.Exec(
			"INSERT INTO users (id, email, password_hash, name) VALUES ($1, $2, $3, $4)",
			userID, req.Email, hash, req.Name,
		)
		if err != nil {
			writeJSON(w, http.StatusInternalServerError, map[string]string{"error": err.Error()})
			return
		}

		// ウェルカムメール（練習中は log.Printf で代用）
		log.Printf("[mail] To: %s / Subject: ようこそ / Body: %s さん、登録ありがとうございます", req.Email, req.Name)

		writeJSON(w, http.StatusCreated, registerResponse{ID: userID, Email: req.Email, Name: req.Name})
	})

	log.Println("listening on :3000")
	log.Fatal(http.ListenAndServe(":3000", nil))
}
```

**起動:**

```bash
DATABASE_URL="postgres://postgres:test@localhost:5432/practice?sslmode=disable" \
  go run ./cmd/server
```

> **Go の冗長さ:** TS や PHP に比べて `if err != nil` の繰り返しで明らかに長い。これも「全ての関心が 1 ハンドラに詰まっている」状況が悪化する一因で、Go では特に refactor の動機になる。

## ③ 課題1 — Onion で書き直す

[[オニオンアーキテクチャ]] の構造に沿って refactor する。ディレクトリ構成のヒントは下記。**コードは自分で書く**（detail ドキュメントを横に置いて参照するのは OK）。

### ディレクトリ構成のヒント（TypeScript 例）

```
src/
├── Domain/
│   ├── User.ts                     # Entity
│   ├── Email.ts                    # Value Object
│   ├── UserRepository.ts           # interface（Domain層に置く）
│   ├── PasswordHasher.ts           # interface（Domain層に置く）
│   └── Mailer.ts                   # interface（Domain層に置く）
├── Application/
│   └── UserApplicationService.ts   # register メソッドを持つ
└── Infrastructure/
    ├── PostgresUserRepository.ts
    ├── BcryptPasswordHasher.ts
    ├── ConsoleMailer.ts            # 練習用の実装（console.log のみ）
    └── server.ts                   # Express + Composition Root
```

> **配置のルール:** インターフェースは全て Domain 層、具体実装は全て Infrastructure 層に置く。これは Onion/Clean 共通の **「依存性逆転の原則を物理的なファイル配置で表現したもの」**。なぜこれが正しいかは下の設計判断で考える。

> **PHP の場合:** PSR-4 で `App\Domain\User` のような名前空間に対応する `src/Domain/User.php` のディレクトリ構造になる。階層は同じ。`composer.json` の `autoload.psr-4` の設定との整合性に注意。

> **Go の場合:** ディレクトリ・ファイル名は小文字スネークケース（`domain/user.go`）、パッケージ名はディレクトリ名と一致。entry point は `cmd/server/main.go`。**Onion の「インターフェースは Domain 層」と Go の「使う側で定義」慣習が衝突する**ので、Onion ルールに従いつつ違和感を観察する（[[Layer7-設計アーキテクチャ/_practice/00_セットアップ-Go|Go セットアップ]] の「Go 固有の学習論点」参照）。

### 設計判断のポイント

実装中に必ず立ち止まって考えるべき問い。**答えを書いてから**[[オニオンアーキテクチャ]] と照合する:

1. **なぜ `PasswordHasher` のインターフェースを Domain 層に置くのか、自分の言葉で説明できるか？**
   - 「bcrypt は技術詳細だ。Infrastructure に置くべきだ」と言いたくなる罠の正体は何か
   - ヒント: 「パスワードはハッシュ化されるべき」というルールは誰の関心事か
2. **`Mailer` も Domain に置くのか？ ウェルカムメール送信そのものはドメインの関心事と言えるか？**
   - 「メールを送る手段」と「メールを送るタイミング・条件」のどちらがドメインで、どちらがインフラか
3. **`Email` は string でいいか、Value Object にすべきか？**
   - Value Object にすると `new Email("foo")` の時点で形式を保証できる（不正値が Domain に入らない）
   - やりすぎると小さなプロジェクトでは過剰になる。判断軸は何か
4. **重複チェックはどの層の責務か？**
   - Repository に `existsByEmail(email)` を生やすか、Application Service で `findByEmail` してから判定するか
   - ドメインルールとして「メールアドレスは一意」を表現する方法はあるか
5. **メール送信失敗時の扱い:** ユーザー作成自体は成功しているのに、メールが失敗したら 500 を返すべきか？ それとも作成は確定して別経路で再送するか？
   - 実務では「ドメインイベントを発火 → ワーカーが非同期送信」が定石。なぜか

### 出来上がりの目安

`UserApplicationService.register()` は 10〜15 行程度のオーケストレーションに収まる。30 行を超えたら、ドメインルールが Application 側に漏れ出している可能性が高い。

## ④ 課題2 — Clean で書き直す

[[クリーンアーキテクチャ]] の構造に沿って同じ機能を実装し直す。**Use Case を独立クラス化** するのが核。

### ディレクトリ構成のヒント（TypeScript 例）

```
src/
├── domain/
│   ├── User.ts
│   └── Email.ts
├── usecases/
│   ├── RegisterUserUseCase.ts        # ← 1ユースケース1クラス
│   └── ports/
│       ├── UserRepository.ts
│       ├── PasswordHasher.ts
│       └── Mailer.ts
├── adapters/
│   ├── controllers/
│   │   └── UserController.ts          # HTTP ↔ UseCase の変換
│   └── repositories/
│       └── PostgresUserRepository.ts
└── infrastructure/
    ├── BcryptPasswordHasher.ts
    ├── ConsoleMailer.ts                 # 練習用の実装（console.log のみ）
    └── server.ts                        # Composition Root
```

> **PHP の場合:** `src/UseCases/RegisterUserUseCase.php`, `src/UseCases/Ports/UserRepository.php`, `src/Adapters/Controllers/UserController.php` のように PSR-4 名前空間でマップ。ディレクトリの大文字小文字は PSR-4 で重要。
>
> **Go の場合:** `usecases/register_user.go`, `usecases/ports/user_repository.go`, `adapters/controllers/user_controller.go` のような小文字スネークケース。Go では `ports` のようなネストしたパッケージ階層は冗長になりがちで、`usecases` 内に interface を直接置く実装も多い。

### Onion との違いの体感ポイント

- インターフェース定義の場所: `domain/` ではなく `usecases/ports/` に置く（Use Case が必要とする境界として表現）
- Use Case クラスの単位: `UserApplicationService` ではなく `RegisterUserUseCase` という単機能クラス。`execute()` メソッド 1 つだけ
- Controller の存在: Onion では Express のハンドラを直接 `Infrastructure/server.ts` に書きがちだが、Clean では `UserController.register(req, res)` を `adapters/` に切り出すのが定石

### Stretch Goal（任意）

`LoginUserUseCase` も追加してみる。Onion で書いたコードと Clean で書いたコードの両方に同じ追加を行うと、**1 ユースケース追加時のファイル数の差** がはっきり見える:

- Onion: `UserApplicationService.login()` を追加（1 ファイル変更）
- Clean: `LoginUserUseCase.ts` を新規作成（1 ファイル追加）

どちらが「変更時の影響範囲」を予測しやすいか、自分の感覚で評価してみる。

## ⑤ 観察ポイント

両方を書き終えた後で振り返る:

### 内→外の依存方向は守られたか

`grep -r "import" src/Domain/` （Onion 版）を実行して、`pg`, `bcrypt`, `@sendgrid` 等が一切含まれていないことを確認する。1 つでも混入していたら、それが今回学ぶべき違反。

### バリデーションはどこに置いたか

3 つの選択肢それぞれに正当性がある:

1. **Controller / HTTP 層**: 入力形式の異常を早期に弾く（型レベルの異常）
2. **Use Case / Application Service**: ビジネスルールとしての異常（パスワード長など）
3. **Domain (Value Object)**: `new Email("foo")` で形式を保証する

実プロジェクトでは「型 → ビジネスルール → ドメイン制約」の 3 層で段階的に検証することが多い。自分の実装はこのうちどれを採用したか？ 採用しなかったレベルは何が問題か？

### Application Service / Use Case の中身

`register()` / `execute()` の本体に **`if` 文がいくつあるか**？ if 文だらけならビジネスロジックが Application 側に漏れている。Domain メソッド（例: `User.create(email, name, passwordHash)` がコンストラクタでバリデーション）に押し戻せないか検討する。

### Composition Root の場所

依存の組み立て（`new PostgresUserRepository(pool)` 等）が **どこに集約されているか**。Composition Root が複数ファイルに散らばっていたら、それは [[DIコンテナ]] の出番のサイン。

## ⑥ セルフチェックリスト

実装が完了したら 1 つずつ確認する。

### 共通（Onion / Clean 両方）

- [ ] Domain 層のファイルから `import { Pool }` `import bcrypt` `import sgMail` の **どれも import されていない**
- [ ] `npm run` で起動でき、`curl -X POST http://localhost:3000/users -H "Content-Type: application/json" -d '{"email":"a@b.c","password":"12345678","name":"A"}'` が 201 を返す
- [ ] バリデーション失敗時に **DB アクセスや bcrypt 計算が実行されない**（早期リターンが効いている）
- [ ] `Mailer` の実装を `SendGridMailer` から `ConsoleMailer` に差し替えても、Application 層 / Domain 層は **一切変更不要**
- [ ] 「メール送信が失敗した時にどう振る舞うか」の方針を自分で決めた（rollback / 別経路 / fire-and-forget のいずれか）

### Onion 固有

- [ ] `UserApplicationService` が `register()` メソッドを持ち、複数機能を追加していくと同じクラスにメソッドが増える構造になっている
- [ ] `UserRepository` インターフェースが `Domain/` 配下にある

### Clean 固有

- [ ] `RegisterUserUseCase` クラスが存在し、`execute()` メソッド 1 つだけ持つ
- [ ] `usecases/ports/` ディレクトリに `UserRepository`, `PasswordHasher`, `Mailer` のインターフェースがある
- [ ] `Controller` が `adapters/` に存在し、`server.ts` から呼ばれている

### 違反を作って観察（重要）

- [ ] `Domain/User.ts` で `import * as bcrypt from "bcrypt"` を書いて `User.setPassword(plain)` で内部ハッシュ化するように変えてみる。何が壊れるか:
  - User の単体テストに bcrypt が必要になる
  - ハッシュアルゴリズムを Argon2 に変える時、Domain ファイルを修正する必要が出る
  - User をフロントエンドから import したくなった時、bcrypt が browser bundle に紛れ込む
- [ ] 元に戻して、なぜ違反が問題なのかを自分の言葉で説明できる
