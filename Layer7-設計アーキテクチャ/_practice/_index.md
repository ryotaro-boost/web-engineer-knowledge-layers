# Layer 7 実践演習 — アーキテクチャを「書いて」体得する

> **目的:** [[クリーンアーキテクチャ]] / [[オニオンアーキテクチャ]] を「読んでわかった気」から「書いてわかる」状態に引き上げる。

## なぜ書く必要があるのか

アーキテクチャは本質的に **「ファイルをどこに置くか」「インターフェースをどちらの層で定義するか」** という配置の判断の集合体である。これは読書では絶対に体得できない種類の知識で、実装中に `import` 文を書こうとした瞬間に **「あ、ここで内→外の依存になっている」** と気づくのが本質的な学習。

逆に言えば、悪い例コードを refactor する痛みなしに概念だけ追っていると、現場で似たコードに遭遇した時に「これは違反か OK か」を判断できない。

## 演習の構造

各演習は次の 6 セクションを持つ:

| セクション | 役割 |
|----------|------|
| ① お題 | ビジネス要件（実装すべき仕様） |
| ② 悪い例コード | 全ての関心が1ハンドラに詰まった 50〜80 行の TypeScript（あなたが refactor する素材） |
| ③ 課題1: Onion で書き直す | ディレクトリ構成のヒントのみ。コードは自分で書く |
| ④ 課題2: Clean で書き直す | 同上。Use Case を独立クラス化する |
| ⑤ 観察ポイント | 両者を書いた後に振り返るための問い |
| ⑥ セルフチェックリスト | 配置の正しさを確認する項目 |

参照用の正解コードは別途用意せず、[[クリーンアーキテクチャ]] と [[オニオンアーキテクチャ]] の detail ドキュメントを **「答え合わせの教科書」** として使う。

## 推奨学習サイクル

```mermaid
graph LR
    A["① お題を読む"] --> B["② 悪い例を写経<br/>scratch project"]
    B --> C["③ Onion で refactor<br/>30〜60分"]
    C --> D["④ Clean で refactor<br/>30〜60分"]
    D --> E["⑤ detail と照合"]
    E --> F["⑥ 違反箇所を意図的に作る"]
    F --> G["⑦ 何が壊れるか観察"]

    style F fill:#fff3e0,stroke:#e65100
    style G fill:#fff3e0,stroke:#e65100
```

最後の **「意図的に違反 → 何が壊れるか観察」** が一番効く。例えば Domain 層に `import { Pool } from "pg"` を書いてみて、その層をテストする時の生 DB 必要性、別 DB に差し替える時の連鎖変更、import 行の汚染を肌で感じる。

## 言語ごとのセットアップガイド

各演習ファイル本体（お題・課題・観察ポイント・セルフチェック）は**言語非依存**で記述する。一方、scratch project の構築手順は言語ごとに別ファイルを用意している:

| 言語 | セットアップガイド | フレームワーク | 状態 |
|------|------------------|---------------|------|
| TypeScript | [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-TypeScript|00_セットアップ-TypeScript]] | Express | 作成済み |
| PHP | [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-PHP|00_セットアップ-PHP]] | Slim 4（Laravel ではない理由は当該ファイル参照） | 作成済み |
| Go | [[Layer7-設計アーキテクチャ/_practice/00_セットアップ-Go|00_セットアップ-Go]] | net/http（標準ライブラリ） | 作成済み |

最初に取り組む言語のセットアップガイドを一度だけ通す → 各演習ファイルに進む、という流れ。各演習ファイル内の「悪い例コード」は 3 言語分が並列に置かれているので、自分の言語のセクションを選んで写経する。

### 言語ごとの観察ポイント（共通）

他言語に取り組む際の差分として、以下が学習価値の高い論点:

- **Go** — インターフェースを「使う側」で定義する慣習。Onion 流の「Domain 層で定義」と思想が衝突するか？
- **PHP / Laravel** — ServiceProvider で DI を組む。Composition Root はどこに置くべきか
- **Python** — `Protocol` ベースの structural typing は Onion / Clean の interface 配置にどう影響するか
