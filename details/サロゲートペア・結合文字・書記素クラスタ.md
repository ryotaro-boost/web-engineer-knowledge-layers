# サロゲートペア・結合文字・書記素クラスタ

> **一言で言うと:** 「1文字」の定義はエンコーディング層によって異なる。コードユニット・コードポイント・書記素クラスタの3層を理解しないと、文字列の長さ計算・切り出し・正規表現が壊れる。

## 概念

### 3つの「文字の単位」

文字列処理には3つの異なる粒度がある。これを混同することがバグの根本原因になる。

```
レベル             | 例: "👨‍👩‍👧‍👦"（家族絵文字）
───────────────────┼─────────────────────────
コードユニット     | 11個（UTF-16 の 16bit 単位）
コードポイント     | 7個（U+1F468, U+200D, U+1F469, U+200D, U+1F467, U+200D, U+1F466）
書記素クラスタ     | 1個（人間が「1文字」と認識する単位）
```

---

### 1. サロゲートペア（Surrogate Pair）

#### UTF-16 固有の問題

Unicode のコードポイントは U+0000 〜 U+10FFFF の範囲だが、UTF-16 の1つのコードユニットは 16bit（0x0000 〜 0xFFFF）しか表現できない。U+10000 以上のコードポイント（絵文字、一部の漢字、数学記号など）は**2つのコードユニットの組**で表現する。これがサロゲートペアである。

```
U+1F363（🍣）の場合:
  コードポイント: 0x1F363
  上位サロゲート: 0xD83C（0xD800〜0xDBFF の範囲）
  下位サロゲート: 0xDF63（0xDC00〜0xDFFF の範囲）
  UTF-16 表現:    [0xD83C, 0xDF63]
```

#### 変換の仕組み

```
1. コードポイントから 0x10000 を引く → 20bit の値を得る
2. 上位 10bit + 0xD800 → 上位サロゲート
3. 下位 10bit + 0xDC00 → 下位サロゲート
```

#### なぜ問題になるか

JavaScript と Java は内部文字列表現に UTF-16 を使っている。そのため `string.length` はコードユニット数を返し、コードポイント数とは一致しない。

```javascript
const sushi = "🍣";
console.log(sushi.length);        // 2（サロゲートペアなので UTF-16 コードユニットが2つ）
console.log(sushi.charCodeAt(0)); // 55356 (0xD83C) — 上位サロゲート
console.log(sushi.charCodeAt(1)); // 57187 (0xDF63) — 下位サロゲート

// 正しくコードポイントを取得する方法
console.log(sushi.codePointAt(0)); // 127843 (0x1F363)
console.log([...sushi].length);    // 1（スプレッド構文はコードポイント単位で分割）
```

#### UTF-8 ではサロゲートペアは存在しない

UTF-8 は可変長（1〜4バイト）で直接コードポイントをエンコードするため、サロゲートペアという概念自体がない。サロゲートペアは **UTF-16 固有の仕組み**である。Python 3 や Go など UTF-8 / UTF-32 ベースの言語では `len("🍣")` が1を返す（Go の `len()` はバイト数を返すが、`utf8.RuneCountInString()` はコードポイント数を返す）。

### 2. 結合文字（Combining Characters）

#### 概念

一部の文字は、基底文字（base character）と結合文字（combining character）の組み合わせで表現できる。

```
"é" の2つの表現方法:

合成済み形式（NFC）:  U+00E9（é — 1コードポイント）
分解形式（NFD）:      U+0065 U+0301（e + ◌́  — 2コードポイント）

見た目はどちらも同じ "é" だが、バイト列は異なる。
```

#### 結合文字の種類

| 種類 | 例 | 用途 |
|------|-----|------|
| 結合アクセント | U+0301（́）, U+0308（̈） | アクセント記号（é, ü） |
| 結合濁点・半濁点 | U+3099（゙）, U+309A（゚） | 日本語の濁点・半濁点（が = か + ゙） |
| 結合囲み | U+20E3（⃣） | キーキャップ絵文字（1️⃣ = 1 + U+FE0F + U+20E3） |

#### なぜ問題になるか

```javascript
const nfc = "\u00E9";       // é（合成済み）
const nfd = "\u0065\u0301"; // é（分解形式: e + combining acute）

console.log(nfc === nfd);         // false（バイト列が異なる）
console.log(nfc.length);          // 1
console.log(nfd.length);          // 2
console.log(nfc.normalize("NFC") === nfd.normalize("NFC")); // true
```

ユーザー入力の比較・検索・ソートの前に**正規化**しないと、見た目が同じ文字列を「別の文字列」として扱ってしまう。

### 3. 書記素クラスタ（Grapheme Cluster）

#### 概念

書記素クラスタは「人間が1文字として認識する最小単位」である。Unicode Standard Annex #29 で定義されたルールに基づく。

1つの書記素クラスタは1つ以上のコードポイントから構成される:

| 書記素クラスタ | コードポイント | 説明 |
|---------------|---------------|------|
| `é` | U+0065 U+0301 | 基底文字 + 結合アクセント |
| `が` | U+304B U+3099 | か + 結合濁点（NFD 形式） |
| `🇯🇵` | U+1F1EF U+1F1F5 | Regional Indicator J + P |
| `👨‍👩‍👧‍👦` | U+1F468 U+200D U+1F469 U+200D U+1F467 U+200D U+1F466 | ZWJ（Zero Width Joiner）で結合された家族絵文字 |
| `👩🏽‍🔬` | U+1F469 U+1F3FD U+200D U+1F52C | 人 + 肌色修飾子 + ZWJ + 顕微鏡 |

#### ZWJ（Zero Width Joiner）シーケンス

ZWJ（U+200D）は複数の絵文字を1つの書記素クラスタに結合する:

```
👨 + ZWJ + 💻 = 👨‍💻（男性テクノロジスト）
🏳 + ZWJ + 🌈 = 🏳️‍🌈（レインボーフラグ）
```

ZWJ シーケンスをサポートしていない環境では、個々の絵文字がバラバラに表示される（フォールバック）。

## 各言語での正しい書記素クラスタ処理

### JavaScript — `Intl.Segmenter`

```javascript
function graphemeLength(str) {
  const segmenter = new Intl.Segmenter("ja", { granularity: "grapheme" });
  return [...segmenter.segment(str)].length;
}

console.log("café".length);              // 4 or 5（NFD なら5）
console.log(graphemeLength("café"));     // 4（常に正しい）

console.log("👨‍👩‍👧‍👦".length);              // 11（UTF-16 コードユニット数）
console.log([..."👨‍👩‍👧‍👦"].length);          // 7（コードポイント数）
console.log(graphemeLength("👨‍👩‍👧‍👦"));      // 1（書記素クラスタ数 = 見た目）

// 書記素クラスタ単位で安全に切り出す
function graphemeSlice(str, start, end) {
  const segmenter = new Intl.Segmenter("ja", { granularity: "grapheme" });
  const segments = [...segmenter.segment(str)];
  return segments.slice(start, end).map(s => s.segment).join("");
}

console.log(graphemeSlice("こんにちは🌸世界", 0, 6)); // "こんにちは🌸"
```

### Python — `grapheme` ライブラリ

```python
# pip install grapheme
import grapheme

text = "👨‍👩‍👧‍👦"
print(len(text))                    # 7 (コードポイント数)
print(grapheme.length(text))        # 1 (書記素クラスタ数)

# 書記素クラスタ単位で安全にスライス
text2 = "café☕"
print(grapheme.slice(text2, 0, 4))  # "café"
```

### Go — `rivo/uniseg` ライブラリ

```go
package main

import (
	"fmt"
	"github.com/rivo/uniseg"
)

func main() {
	text := "👨‍👩‍👧‍👦"
	fmt.Println(len(text))                          // 25 (バイト数)
	fmt.Println(uniseg.GraphemeClusterCount(text))   // 1 (書記素クラスタ数)
}
```

## Unicode 正規化（Normalization）

結合文字の問題に対処するため、Unicode は4つの正規化形式を定義している:

| 形式 | 名称 | 動作 | 用途 |
|------|------|------|------|
| NFC | Canonical Composition | 分解 → 再合成 | **Web 標準**。HTML フォームのデフォルト出力 |
| NFD | Canonical Decomposition | 完全に分解 | macOS のファイルシステム（HFS+ では NFD を強制。後継の APFS は入力形式を保持） |
| NFKC | Compatibility Composition | 互換分解 → 再合成 | 検索・比較用（全角半角の統一） |
| NFKD | Compatibility Decomposition | 互換分解 | テキスト解析 |

```javascript
// NFC vs NFD
const nfc = "が";                    // U+304C（合成済み）
const nfd = "が".normalize("NFD");   // U+304B U+3099（か + 濁点）

console.log(nfc.length);             // 1
console.log(nfd.length);             // 2
console.log(nfc === nfd);            // false
console.log(nfc.normalize("NFC") === nfd.normalize("NFC")); // true

// NFKC: 全角半角の正規化
const fullwidth = "Ｈｅｌｌｏ";
console.log(fullwidth.normalize("NFKC")); // "Hello"

const circled = "①②③";
console.log(circled.normalize("NFKC"));   // "123"（意味が変わる場合もあるので注意）
```

### 正規化の実務ルール

- **ユーザー入力の比較前に NFC で正規化する** — 同じ文字の異なる表現を統一
- **検索・フィルタリングには NFKC を使う** — 全角/半角の違いを吸収
- **表示用テキストには NFKC を適用しない** — `①` が `1` になるなど、意図しない変換が起きる
- **macOS との相互運用に注意** — macOS（HFS+）はファイル名を NFD で保存するため、Linux（NFC）との間でファイル名の不一致が起きる

## よくある落とし穴

1. **`string.length` を信じない** — JavaScript と Java では UTF-16 コードユニット数が返る。ユーザーに表示する「文字数」には書記素クラスタ数を使うこと。

2. **`charAt()` / `substring()` でサロゲートペアが壊れる** — JavaScript の `charAt(0)` は上位サロゲートだけを返す場合がある。`Array.from(str)` や `Intl.Segmenter` でコードポイント/書記素クラスタ単位に分割してから操作する。

3. **正規表現の `.` は書記素クラスタにマッチしない** — JavaScript の `/./u` はコードポイント単位。ZWJ シーケンスや結合文字は複数回マッチする。書記素クラスタ単位のマッチには `Intl.Segmenter` で事前分割するか、`/\p{Extended_Pictographic}/u` 等の Unicode プロパティエスケープを使う。

4. **DB のカラム長指定** — `VARCHAR(100)` の「100」がバイト数なのか文字数なのかは DBMS 次第。MySQL の `utf8mb4` では文字数（コードポイント数）だが、絵文字1文字でも1とカウントされる。

5. **macOS の NFD ファイル名** — macOS 上で作成したファイルを Linux に持ち込むと、濁点を含む日本語ファイル名が「見た目は同じだが別のファイル名」として扱われる場合がある。
