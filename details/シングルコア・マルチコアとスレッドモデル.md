# シングルコア・マルチコアとスレッドモデル

> **一言で言うと:** CPUコア数とスレッド数の組み合わせが「並行処理（Concurrency）」と「並列処理（Parallelism）」のどちらが可能かを決定する。この区別を理解することが、適切な並行モデル選択の出発点となる。

## 概念

### コアとスレッドの関係

| 用語 | 何か | 物理/論理 |
|------|------|-----------|
| コア（Core） | CPU 内の独立した演算装置。命令を実際に実行するハードウェア | 物理 |
| スレッド（Thread） | OS が管理する実行の流れ。プログラムカウンタとスタックを持つ | 論理 |
| プロセス（Process） | 独立したメモリ空間を持つ実行単位。1つ以上のスレッドを含む | 論理 |

コアは「作業台」、スレッドは「作業者」、プロセスは「作業部屋」と考えるとわかりやすい。作業台が1つしかなくても（シングルコア）、複数の作業者（スレッド）が交代で使うことはできる。

### 4つの組み合わせ

```
                  シングルスレッド          マルチスレッド
                ┌──────────────────┬──────────────────────┐
 シングルコア    │ 順次実行のみ        │ 並行（時分割で交互実行） │
                │ 並行 ✗  並列 ✗     │ 並行 ✓  並列 ✗        │
                ├──────────────────┼──────────────────────┤
 マルチコア     │ 1コアだけ使用       │ 並行かつ並列           │
                │ 並行 ✗  並列 ✗     │ 並行 ✓  並列 ✓        │
                └──────────────────┴──────────────────────┘
```

この表が示す最も重要な点は、**並行は構造の問題（ソフトウェア設計）であり、並列は実行の問題（ハードウェア能力）である**ということ。

## シングルコア × シングルスレッド — 順次実行

最も単純な実行モデル。1つの命令が終わるまで次の命令は実行されない。

```
コア: [タスクA ██████████][タスクB ██████████][タスクC ██████████]
      ──────────────────────────────────── 時間 →
```

初期のコンピュータはすべてこのモデルだった。現代でも、バッチ処理スクリプトや単純な CLI ツールはこのモデルで十分な場合が多い。

## シングルコア × マルチスレッド — 並行だが並列ではない

OS のスケジューラがスレッドを高速に切り替え（コンテキストスイッチ）、複数の処理が「同時に進んでいるように見える」状態を作る。実際には1つのコアが交互に処理しているだけで、物理的な同時実行は起きていない。

```
コア: [A ███][B ██][A ████][C ██][B ███][C ████]
      ──────────────────────────────────── 時間 →
      ↑     ↑     ↑      ↑     ↑
      コンテキストスイッチ（切り替えコストが発生）
```

### なぜそれでも有用か

CPU バウンドな処理では速くならないが、**I/O バウンドな処理**では劇的に効果がある。あるスレッドがネットワーク応答を待っている間に、別のスレッドが CPU を使える。

```
シングルスレッド:
  [リクエスト処理][────── 応答待ち ──────][結果処理][次のリクエスト...]

マルチスレッド（シングルコア）:
  スレッド1: [リクエスト処理][── 応答待ち ──][結果処理]
  スレッド2:                 [リクエスト処理][── 応答待ち ──][結果処理]
  CPU利用:   [████████████████████████████████████████████]
  → 応答待ちの間に別のリクエストを処理できる
```

### Node.js のイベントループはここに位置する

Node.js はシングルスレッドのイベントループで非同期 I/O を処理する。OS レベルでは複数のスレッドが動いている（libuv のスレッドプール）が、JavaScript コードの実行自体はシングルスレッド。このモデルは「シングルコア × マルチスレッド」の並行処理と本質的に同じ発想 — 1つの実行コンテキストで I/O 待ちの隙間を有効活用する。

```javascript
// JavaScript では1つのスレッドが並行に複数の処理を扱う
const results = await Promise.all([
  fetch('/api/users'),     // I/O 待ちの間に...
  fetch('/api/products'),  // 別のリクエストを発行
  fetch('/api/orders'),    // さらに別のリクエストも
]);
// 3つのリクエストが「並行」に進むが、CPU は1コアしか使わない
```

## マルチコア × シングルスレッド — コアを持て余す

マルチコア CPU を持っていても、シングルスレッドのプログラムは1つのコアしか使えない。残りのコアは遊んでいる。

```
コア1: [タスクA ██████████████████████████████████████]
コア2: [                    空き                      ]
コア3: [                    空き                      ]
コア4: [                    空き                      ]
```

これは現代のパフォーマンス問題でよく見られる。「CPU 使用率が 25% で頭打ち」という場合、4コアマシンでシングルスレッドのプログラムが1コアだけを使い切っている可能性が高い。

```bash
# Linux で確認: 特定のコアだけ100%になっていればこのパターン
$ top -1
# CPU0: 100% us  ← このコアだけフル稼働
# CPU1:   2% us
# CPU2:   1% us
# CPU3:   3% us
```

## マルチコア × マルチスレッド — 並行かつ並列

複数のスレッドが複数のコアで物理的に同時実行される。真の並列処理。

```
コア1: [スレッドA ██████████████████████████████]
コア2: [スレッドB ██████████████████████████████]
コア3: [スレッドC ██████████████████████████████]
コア4: [スレッドD ██████████████████████████████]
      ──────────────────────────────────── 時間 →
      4つの処理が物理的に同時実行される
```

### Go の Goroutine はこのモデルを簡単に実現する

Go のランタイムスケジューラは Goroutine を OS スレッドにマッピングし、マルチコアで並列実行する。`GOMAXPROCS` で使用するコア数を制御できる（デフォルトは全コア）。

```go
package main

import (
    "fmt"
    "runtime"
    "sync"
)

func main() {
    fmt.Println("利用可能コア数:", runtime.NumCPU())
    // GOMAXPROCS で並列度を制御（デフォルトはCPU数）
    runtime.GOMAXPROCS(runtime.NumCPU())

    var wg sync.WaitGroup
    // CPU バウンドな処理を並列実行
    for i := 0; i < runtime.NumCPU(); i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            sum := 0
            for j := 0; j < 100_000_000; j++ {
                sum += j
            }
            fmt.Printf("Goroutine %d 完了 (sum=%d)\n", id, sum)
        }(i)
    }
    wg.Wait()
    // 各 Goroutine が別々のコアで同時に実行される → 線形に近い高速化
}
```

### Python の GIL（Global Interpreter Lock）問題

Python（CPython）は GIL により、マルチスレッドでも同時に1つのスレッドしか Python バイトコードを実行できない。つまり CPU バウンドな処理ではマルチコアの恩恵を受けられない。なお、Python 3.13（2024年10月）で実験的に free-threaded mode（PEP 703）が導入され、GIL を無効化した真の並列実行が可能になりつつある。

```python
import threading
import multiprocessing
import time

def cpu_heavy():
    """CPU バウンドな処理"""
    total = 0
    for i in range(20_000_000):
        total += i

# マルチスレッド — GIL のため並列にならない
start = time.time()
threads = [threading.Thread(target=cpu_heavy) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
print(f"マルチスレッド: {time.time() - start:.2f}秒")
# → シングルスレッドとほぼ同じ時間（GIL で直列化される）

# マルチプロセス — プロセスごとに独立した GIL → 真の並列
start = time.time()
processes = [multiprocessing.Process(target=cpu_heavy) for _ in range(4)]
for p in processes:
    p.start()
for p in processes:
    p.join()
print(f"マルチプロセス: {time.time() - start:.2f}秒")
# → 約4倍速い（4コアで並列実行される）
```

### PHP — 「リクエストごとにプロセスが独立」モデル

PHP は他の言語と根本的に異なるアプローチを取る。PHP 自体にはマルチスレッドの仕組みがほとんどなく、代わりに **Web サーバー（Apache/Nginx + PHP-FPM）がプロセスレベルで並列化を担う**。

```
リクエスト1 → [PHP-FPM ワーカープロセス1] → レスポンス
リクエスト2 → [PHP-FPM ワーカープロセス2] → レスポンス
リクエスト3 → [PHP-FPM ワーカープロセス3] → レスポンス
              各プロセスは独立 → メモリ共有なし → 競合状態が起きない
```

**特徴:**
- 各リクエストは独立したプロセスで処理され、メモリを共有しない（Shared Nothing Architecture）
- 競合状態やデッドロックを言語レベルで考える必要がほぼない
- リクエストが終わるとプロセスの状態は破棄される（ステートレス）
- 並列性は PHP-FPM のワーカープロセス数で制御する

```ini
; php-fpm.conf — ワーカープロセス数の設定
pm = dynamic
pm.max_children = 50        ; 最大プロセス数（≒ 最大同時リクエスト数）
pm.start_servers = 5
pm.min_spare_servers = 5
pm.max_spare_servers = 35
```

**トレードオフ:** プロセスごとにメモリを消費するため、1プロセスあたり 30-50MB とすると、50 プロセスで 1.5-2.5GB のメモリが必要。Node.js のイベントループモデルに比べてメモリ効率は劣る。一方で、共有状態がないためバグが生まれにくく、開発者が並行性を意識しなくてよいという大きなメリットがある。

**並行処理が必要な場合:** PHP 8.1 で導入された Fiber（ファイバー）が軽量な協調的マルチタスクを提供する。Laravel の非同期処理や ReactPHP/Swoole などのライブラリがこの上に構築されている。

```php
// Fiber による協調的マルチタスク（PHP 8.1+）
$fiber = new Fiber(function (): void {
    $value = Fiber::suspend('一時停止');  // 制御を呼び出し元に返す
    echo "再開: " . $value;
});

$result = $fiber->start();     // "一時停止" が返る
$fiber->resume('データ');       // "再開: データ" が出力される
```

### Ruby — GIL（GVL）を持つが進化中

Ruby（CRuby/MRI）は Python と同様に GVL（Global VM Lock、旧称 GIL）を持つ。マルチスレッドで書いても、CPU バウンドな処理は1スレッドずつしか実行されない。

```ruby
# マルチスレッド — GVL のため CPU バウンドは並列にならない
threads = 4.times.map do
  Thread.new do
    sum = 0
    20_000_000.times { |i| sum += i }
  end
end
threads.each(&:join)
# → シングルスレッドとほぼ同じ時間
```

**ただし I/O バウンドではスレッドが有効。** GVL は I/O 待ちの間に解放されるため、複数の HTTP リクエストやDB問い合わせを並行で処理できる。Puma（Ruby の Web サーバー）はこれを活用し、マルチスレッド + マルチプロセスのハイブリッドで並行性を実現する。

```ruby
# Puma の設定例 — マルチプロセス × マルチスレッド
# config/puma.rb
workers 4          # 4 プロセス（マルチコア活用 → 並列）
threads 5, 16      # 各プロセス内 5-16 スレッド（I/O 並行化）
# → 最大 4 × 16 = 64 の同時リクエスト処理
```

```
プロセス1（GVL独立）: [スレッド1][スレッド2][スレッド3]...  → コア1
プロセス2（GVL独立）: [スレッド1][スレッド2][スレッド3]...  → コア2
プロセス3（GVL独立）: [スレッド1][スレッド2][スレッド3]...  → コア3
プロセス4（GVL独立）: [スレッド1][スレッド2][スレッド3]...  → コア4
各プロセスは独立した GVL を持つため、プロセス間は真に並列
プロセス内のスレッドは I/O バウンドな処理を並行で捌く
```

**Ruby の進化:**
- **Ractor**（Ruby 3.0+）— プロセスより軽量で、GVL に縛られない並列実行単位。アクターモデルに基づき、Ractor 間でオブジェクトを共有しない設計で安全な並列処理を実現する。

```ruby
# Ractor による真の並列処理（Ruby 3.0+）
ractors = 4.times.map do
  Ractor.new do
    sum = 0
    20_000_000.times { |i| sum += i }
    sum
  end
end
results = ractors.map(&:take)
# → 各 Ractor が別コアで並列実行される
```

### 各言語のまとめ — アーキテクチャ比較

| 言語/ランタイム | CPU バウンドの並列化 | I/O バウンドの並行化 | 並行モデルの特徴 |
|---------------|:---:|:---:|:---|
| Go (Goroutine) | マルチコアで並列 | Goroutine で並行 | ランタイムが自動でマルチコア活用 |
| Node.js | Worker Threads が必要 | イベントループで並行 | シングルスレッド + 非同期 I/O |
| Python (CPython) | マルチプロセスが必要 (GIL) | スレッドまたは asyncio で並行 | GIL で CPU 並列不可 |
| PHP | PHP-FPM のマルチプロセス | Fiber / ReactPHP / Swoole | リクエスト単位の Shared Nothing |
| Ruby (CRuby) | マルチプロセスまたは Ractor (GVL) | スレッドで並行（I/O時GVL解放） | Puma のマルチプロセス×マルチスレッド |
| Java | スレッドプールで並列 | スレッドまたは Virtual Threads で並行 | OS スレッド直接利用 |
| Rust | スレッドで並列（所有権で安全保証） | async/await (tokio等) で並行 | コンパイル時に安全性保証 |

## 並行と並列の判断フローチャート

実務で「どのモデルを使うか」を判断するための指針:

```
処理のボトルネックは何か？
├─ I/O バウンド（API呼び出し、DB問い合わせ、ファイル読み書き）
│   → 並行処理で十分（シングルスレッド + 非同期 or マルチスレッド）
│   → Node.js: async/await + Promise.all
│   → Python: asyncio or threading
│   → Go: Goroutine
│   → PHP: PHP-FPM ワーカー / Fiber / Swoole
│   → Ruby: Puma のマルチスレッド
│
└─ CPU バウンド（画像処理、暗号計算、データ変換）
    → 並列処理が必要（マルチコア活用）
    → Node.js: Worker Threads
    → Python: multiprocessing（GIL 回避）
    → Go: Goroutine（自動でマルチコア活用）
    → PHP: PHP-FPM のマルチプロセス / 外部ワーカー
    → Ruby: Ractor または multiprocess（GVL 回避）
```

## 落とし穴

### 1. 「コア数 = 最大並列数」ではない

ハイパースレッディング（Hyper-Threading / SMT）により、1つの物理コアが2つの論理コアとして見えることがある。4コア8スレッドの CPU は、OS からは8コアに見えるが、物理的な演算能力は4コア分。CPU バウンドな処理では物理コア数がボトルネックになる。

### 2. コンテキストスイッチのコストを無視する

スレッド切り替えにはレジスタの保存・復元、キャッシュの無効化などのコストがかかる。スレッド数がコア数を大幅に超えると、コンテキストスイッチのオーバーヘッドが処理時間を上回る。一般的に、CPU バウンドな処理ではスレッド数をコア数程度に抑えるのが最適。

### 3. 並列にしても線形に速くならない（アムダールの法則）

プログラムに直列部分（初期化、結果の集約など）がある限り、コアを増やしても高速化には限界がある。例えばプログラムの 10% が直列なら、残り 90% を完全に並列化しても最大10倍までしか速くならない。

```
高速化の上限 = 1 / (直列部分の割合 + 並列部分の割合 / コア数)

直列10%、コア数無限 → 最大 1/0.1 = 10倍
直列20%、コア4     → 最大 1/(0.2 + 0.8/4) = 2.5倍
```

### 4. 「マルチスレッド = 速い」という短絡的な判断

スレッドの生成コスト、ロックの競合、メモリ消費の増加を考えると、シンプルなシングルスレッドの方が速い場合も多い。特に I/O バウンドな Web アプリケーションでは、Node.js のイベントループモデルがマルチスレッドの Java サーバーよりスループットで勝つケースがある。
