# ダングリングポインタ（Dangling Pointer）

> **一言で言うと:** 解放済みのメモリ領域を指し続けるポインタ。アクセスすると未定義動作（Undefined Behavior）を引き起こし、クラッシュ・データ破壊・セキュリティ脆弱性の原因になる。

## 概念

### ダングリングポインタとは

ポインタ（メモリアドレスを保持する変数）が指す先のメモリが既に解放されている状態。解放後のメモリ領域は他のデータに再利用される可能性があるため、そこにアクセスすると「全く関係ないデータを読み書きする」ことになる。

```
正常な状態:
  ptr ──→ [有効なデータ: "Hello"]

解放後（ダングリング状態）:
  ptr ──→ [??????]  ← OSやランタイムが別の目的に再利用する可能性
                       読み取り: ゴミデータ or クラッシュ
                       書き込み: 別のデータを破壊 or クラッシュ
```

### メモリリークとの関係

[[メモリリーク]]とダングリングポインタは**メモリ管理の失敗の両極端**である。

```
メモリリーク:          参照あり、でも不要 → メモリが減り続ける
ダングリングポインタ:  参照あり、でも無効 → アクセスすると壊れる

正しい状態:            必要なら参照を保持、不要なら参照を消して解放
```

| 問題 | 原因 | 結果 |
|------|------|------|
| メモリリーク | 解放すべきメモリを解放しない | メモリ枯渇（OOM） |
| ダングリングポインタ | 解放済みメモリへの参照が残る | 未定義動作（クラッシュ、データ破壊） |
| 二重解放（Double Free） | 同じメモリを2回解放する | ヒープ構造の破壊 |

## 発生パターン

### パターン1: 解放後のアクセス（Use After Free）

最も基本的なパターン。メモリを解放した後にポインタを通じてアクセスする。

```c
#include <stdlib.h>
#include <stdio.h>

int main() {
    int *ptr = malloc(sizeof(int));
    *ptr = 42;
    printf("Before free: %d\n", *ptr); // 42

    free(ptr);
    // ptr はまだ古いアドレスを保持している（ダングリング状態）

    // ❌ 未定義動作: 解放済みメモリへのアクセス
    printf("After free: %d\n", *ptr); // ゴミ値 or クラッシュ

    // ✅ 対策: 解放後に NULL を代入
    ptr = NULL;
    // NULL チェックで安全にアクセス可能
    if (ptr != NULL) {
        printf("%d\n", *ptr);
    }
    return 0;
}
```

### パターン2: ローカル変数へのポインタを返す

関数のスタックフレームが破棄された後も、その中の変数を指すポインタを使い続ける。

```c
// ❌ ダングリングポインタ: スタック上の変数へのポインタを返す
int* createValue() {
    int value = 42;    // スタック上に確保
    return &value;     // 関数終了後、value のメモリは無効
}

int main() {
    int *ptr = createValue();
    // ptr はもう存在しない value のアドレスを指している
    printf("%d\n", *ptr); // 未定義動作
    return 0;
}

// ✅ 対策: ヒープに確保して返す（呼び出し側が解放する責任を負う）
int* createValue() {
    int *value = malloc(sizeof(int));
    *value = 42;
    return value; // ヒープ上なので関数終了後も有効
}
```

### パターン3: 二重解放（Double Free）

同じメモリを2回解放すると、ヒープの管理構造が破壊される。2回目の `free` の後、ポインタはダングリング状態になるだけでなく、ヒープアロケータが不整合な状態に陥る。

```c
// ❌ 二重解放
char *buf = malloc(256);
free(buf);
free(buf); // 未定義動作: ヒープ破壊

// ✅ 対策: 解放後に NULL 化（free(NULL) は安全に無視される）
char *buf = malloc(256);
free(buf);
buf = NULL;
free(buf); // 何も起きない（安全）
```

## Web開発との関係

### 「GC言語だから関係ない」は本当か？

JavaScript、Python、Go などのGC言語ではポインタの直接操作がないため、**従来のダングリングポインタは発生しない**。GCが「参照があるなら解放しない」ことを保証するためだ。

ただし、ダングリングポインタの**類似概念**はGC言語にも存在する。

#### JavaScript: 無効な参照パターン

```javascript
// DOM要素が削除された後もJSから参照し続ける
const button = document.getElementById('submit-btn');
document.getElementById('form-container').remove();
// button は削除された DOM 要素を参照 → メモリリークの原因
// クリックイベントは発火しないが、オブジェクト自体はGCされない

// ✅ 対策: 参照をクリアする or WeakRef を使う
let buttonRef = new WeakRef(document.getElementById('submit-btn'));
// 必要な時に deref() で取得（GC済みなら undefined）
const btn = buttonRef.deref();
if (btn) btn.click();
```

#### Go: closed channel への送信

```go
// ❌ パニック: closed channel への送信
ch := make(chan int)
close(ch)
ch <- 42 // panic: send on closed channel

// ✅ 対策: select + context でライフサイクルを管理
func worker(ctx context.Context, ch chan<- int) {
    for {
        select {
        case <-ctx.Done():
            return
        case ch <- computeValue():
        }
    }
}
```

#### Rust: コンパイル時にダングリングポインタを防ぐ

Rustは所有権（Ownership）と借用（Borrowing）のルールにより、ダングリングポインタをコンパイル時に完全に防ぐ。これがRustが「GCなしで安全」と言われる理由。

```rust
// ❌ コンパイルエラー: ダングリングリファレンス
fn create_string() -> &String {
    let s = String::from("hello");
    &s // s はこの関数の終了時にドロップされる → 参照が無効になる
}  // error[E0106]: missing lifetime specifier

// ✅ 所有権を移動（ムーブ）して返す
fn create_string() -> String {
    let s = String::from("hello");
    s // 所有権が呼び出し元に移動 → ダングリングしない
}
```

## セキュリティへの影響

ダングリングポインタは単なるバグではなく、深刻なセキュリティ脆弱性の原因になる。

### Use After Free 攻撃

1. 攻撃者がオブジェクトの解放を誘発する
2. 解放された領域に攻撃者が制御するデータを配置する（ヒープスプレー）
3. ダングリングポインタ経由でアクセスさせ、攻撃者のデータを「正当なオブジェクト」として扱わせる
4. 仮想関数テーブル（vtable）の書き換えなどで任意コード実行

ブラウザ（Chrome、Firefox）の脆弱性の多くがUse After Freeに起因する。これが、ChromeがメモリセーフなRustの採用やMiraclePtr（ダングリングポインタ検出機構）を進めている理由。

## 落とし穴

### 1. 「NULL チェックすれば安全」は不十分

`free` 後に `NULL` を代入する習慣は有効だが、同じメモリを指すポインタのコピーが複数存在する場合、1つを `NULL` にしても他のコピーはダングリングのまま。

```c
int *a = malloc(sizeof(int));
int *b = a;   // コピー
free(a);
a = NULL;     // a は安全
*b = 10;      // ❌ b はまだダングリング
```

### 2. 「デバッグビルドでは再現しない」

多くのデバッグ用メモリアロケータは解放済みメモリを特殊な値（`0xDEADBEEF` など）で埋める。本番ビルドでは解放済みメモリがすぐ再利用され、「動いているように見える」が別のデータを読み書きしている — 最も危険な状態。

### 3. GC言語でもネイティブ拡張は危険

Node.jsのネイティブアドオン（C++ Addon, napi）やPythonのC拡張では、ポインタを直接操作する。GC言語の安全性はネイティブコードの境界で消失する。
