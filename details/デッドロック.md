# デッドロック（Deadlock）

> **一言で言うと:** 2つ以上の処理が互いに相手のリソース解放を待ち合い、どちらも永遠に先に進めなくなる状態。

## 概念

### デッドロックの4条件（Coffman条件）

デッドロックは、以下の4つの条件が**すべて同時に成立した**ときにのみ発生する。逆に言えば、1つでも崩せばデッドロックは起きない。

| 条件 | 意味 | 例 |
|------|------|-----|
| 相互排他（Mutual Exclusion） | リソースは一度に1つの処理しか使えない | ミューテックスは1スレッドだけが保持 |
| 保持と待機（Hold and Wait） | リソースを保持したまま、別のリソースを待つ | ロックAを持ちながらロックBを要求 |
| 非横取り（No Preemption） | 保持中のリソースを強制的に奪えない | ロックは保持者が自発的に解放するしかない |
| 循環待ち（Circular Wait） | 処理の待ちグラフが循環する | A→B→C→A の形で互いに待つ |

### 最も単純なデッドロック

```
時刻1: スレッドA が リソースX をロック
時刻2: スレッドB が リソースY をロック
時刻3: スレッドA が リソースY を要求 → Bが保持中 → 待ち
時刻4: スレッドB が リソースX を要求 → Aが保持中 → 待ち
→ 循環待ち成立 → デッドロック
```

## 検出と対策

デッドロックへのアプローチは大きく3つある。**予防（Prevention）**、**回避（Avoidance）**、**検出と回復（Detection & Recovery）**。

### 予防（Prevention）— Coffman条件を崩す

4条件のうち1つを構造的に不可能にする。

#### 方法1: 循環待ちを崩す — ロック順序の統一

最も実用的な予防策。すべてのリソースに全順序を定め、番号の小さいものから取得する。

```python
import threading

# リソースに一意のIDを割り当て、常に小さいIDから取得
lock_a = threading.Lock()  # ID: 1
lock_b = threading.Lock()  # ID: 2

def transfer(from_lock, from_id, to_lock, to_id, amount):
    # 常にIDの小さい方を先にロック
    first, second = (from_lock, to_lock) if from_id < to_id else (to_lock, from_lock)
    with first:
        with second:
            # 送金処理
            pass
```

```go
// Go での実装例: アカウントIDの小さい方を先にロック
func transfer(a, b *Account, amount int) {
    first, second := a, b
    if a.ID > b.ID {
        first, second = b, a
    }
    first.mu.Lock()
    defer first.mu.Unlock()
    second.mu.Lock()
    defer second.mu.Unlock()

    first.Balance -= amount
    second.Balance += amount
}
```

#### 方法2: 保持と待機を崩す — 一括取得

必要なリソースをすべて一度に取得するか、1つでも取得できなければ全て解放してリトライする。

```python
import threading
import time

def acquire_all_or_none(*locks, timeout=1.0):
    """全てのロックを取得できるか、1つも取得しないかの二択"""
    acquired = []
    deadline = time.monotonic() + timeout
    for lock in locks:
        remaining = deadline - time.monotonic()
        if remaining <= 0 or not lock.acquire(timeout=remaining):
            # 取得済みのロックを全て解放
            for acquired_lock in reversed(acquired):
                acquired_lock.release()
            return False
        acquired.append(lock)
    return True
```

#### 方法3: 非横取りを崩す — タイムアウト付きロック

ロック取得に制限時間を設け、取得できなければ諦めて保持中のロックも解放する。

```java
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

ReentrantLock lockA = new ReentrantLock();
ReentrantLock lockB = new ReentrantLock();

public void safeOperation() throws InterruptedException {
    while (true) {
        if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) {
            try {
                if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) {
                    try {
                        // 両方取得成功 → 処理実行
                        return;
                    } finally {
                        lockB.unlock();
                    }
                }
            } finally {
                lockA.unlock();  // lockB が取れなかった → lockA も解放
            }
        }
        // バックオフしてリトライ
        Thread.sleep((long)(Math.random() * 50));
    }
}
```

### 回避（Avoidance）— 銀行家のアルゴリズム

実行時にリソース割り当てが安全かを判定し、デッドロックに至る可能性がある割り当てを拒否する。ダイクストラの銀行家のアルゴリズム（Banker's Algorithm）が代表的。

**現実には：** 事前にリソースの最大要求量を知る必要があり、一般的なアプリケーションでは実用的でない。OS のリソース管理や組み込みシステムで使われることがある。

### 検出と回復（Detection & Recovery）

デッドロックを許容しつつ、発生を検出して回復する。

**検出方法:**
- 待ちグラフ（Wait-for Graph）を構築し、循環を検出する
- タイムアウトで「おそらくデッドロック」と判断する
- タイムアウト後に待ちグラフを構築して循環を検出する（PostgreSQL はこの方式。`deadlock_timeout` 経過後に待ちグラフを構築し、実際に循環があるかを判定する）

**回復方法:**
- 循環の1つのトランザクションを強制終了（victim 選択）してロールバック
- PostgreSQL は `deadlock_timeout`（デフォルト1秒）後にデッドロック検出を実行し、1つのトランザクションを中断する

```sql
-- PostgreSQL でのデッドロック発生例
-- セッション1:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- ここで待機...
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- セッション2（同時に実行）:
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;
-- ここで待機...
UPDATE accounts SET balance = balance + 50 WHERE id = 1;
-- → ERROR: deadlock detected
```

## 実際の現場でのデッドロック

### Web アプリケーションでの典型パターン

#### 1. DB トランザクションの行ロック順序不一致

```javascript
// ユーザーA→Bへの送金とB→Aへの送金が同時に走ると危険
async function transferUnsafe(fromId, toId, amount) {
  await db.transaction(async (trx) => {
    await trx('accounts').where({ id: fromId }).decrement('balance', amount);
    await trx('accounts').where({ id: toId }).increment('balance', amount);
  });
}

// 安全: 常にIDの小さい方を先に更新
async function transferSafe(fromId, toId, amount) {
  const [firstId, secondId] = fromId < toId ? [fromId, toId] : [toId, fromId];
  await db.transaction(async (trx) => {
    await trx('accounts').where({ id: firstId }).forUpdate().first();
    await trx('accounts').where({ id: secondId }).forUpdate().first();
    await trx('accounts').where({ id: fromId }).decrement('balance', amount);
    await trx('accounts').where({ id: toId }).increment('balance', amount);
  });
}
```

#### 2. 外部API呼び出し中のロック保持

```python
# 悪い例: ロック中に外部APIを呼ぶ → レスポンスが遅いと全体が止まる
with lock:
    result = requests.get('https://external-api.com/data')  # 数秒かかる可能性
    shared_data.update(result.json())

# 良い例: API呼び出しをロックの外に出す
result = requests.get('https://external-api.com/data')  # ロック外で実行
with lock:
    shared_data.update(result.json())  # 最小限のクリティカルセクション
```

## 落とし穴

### 1. 「デッドロックは滅多に起きないから対策不要」

デッドロックは負荷が低いときには発生せず、本番環境でトラフィックが増えたときに突然発生する。テスト環境で再現しないからといって安全ではない。

### 2. 隠れたロック順序の不一致

直接ロックを使っていなくても、ORM の `save()` やフレームワークの内部で行ロックが発生する。SQL クエリの実行順序がデッドロックの原因になることがある。

### 3. 分散システムでのデッドロック

マイクロサービス間で「サービスAがサービスBを呼び、サービスBがサービスAを呼ぶ」という循環依存が発生すると、リソースロックがなくてもデッドロック的な状況（分散デッドロック）になる。タイムアウトとサーキットブレーカーが対策となる。

### 4. デッドロック対策のつもりが飢餓状態に

タイムアウトでリトライする方式では、同じトランザクションが何度も victim に選ばれ続けて永遠に完了しない可能性がある。リトライ時のランダムバックオフや、victim 選択の公平性が必要。
