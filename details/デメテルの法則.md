# デメテルの法則（Law of Demeter / LoD）

> **一言で言うと:** 「直接の友人とだけ話せ」。オブジェクトは自分が直接知っている相手のメソッドだけを呼び、その先の構造に手を伸ばしてはならない。

## 概念

デメテルの法則は1987年に Northeastern 大学の Demeter プロジェクトで提唱された設計ガイドラインで、**最小知識の原則（Principle of Least Knowledge）** とも呼ばれる。

あるメソッド `M` が呼び出してよいのは、以下のオブジェクトのメソッドに限定される:

1. `M` が属するオブジェクト自身（`this`）
2. `M` の引数として渡されたオブジェクト
3. `M` の中で生成されたオブジェクト
4. `M` が属するオブジェクトのフィールド

つまり**「友達の友達」に直接話しかけてはいけない**。

## なぜ重要か

デメテルの法則に違反するコードは、**オブジェクトの内部構造に依存**している。内部構造が変わると、関係ないはずの呼び出し元が壊れる。

```typescript
// ❌ 違反: Order の内部構造（customer → address → city）を知りすぎている
function getShippingCity(order: Order): string {
  return order.getCustomer().getAddress().getCity();
}
// customer の構造変更、address の構造変更、いずれもこのコードが壊れる
```

この `a.b().c().d()` のようなチェーンは**列車事故（Train Wreck）** と呼ばれる。

## 具体例

### 違反と修正

```typescript
// ❌ 違反: Wallet の存在と内部構造を知っている
class PaymentService {
  charge(customer: Customer, amount: number): void {
    const wallet = customer.getWallet();
    if (wallet.getBalance() >= amount) {
      wallet.debit(amount);
    } else {
      throw new Error('残高不足');
    }
  }
}

// ✅ 修正: Customer に「支払い」を依頼する（Tell, Don't Ask）
class Customer {
  pay(amount: number): void {
    // 支払い方法の詳細は Customer の内部に閉じる
    if (this.wallet.balance < amount) {
      throw new Error('残高不足');
    }
    this.wallet.debit(amount);
  }
}

class PaymentService {
  charge(customer: Customer, amount: number): void {
    customer.pay(amount); // Customer の内部構造を知らない
  }
}
```

## Tell, Don't Ask

デメテルの法則と密接に関連する原則が **Tell, Don't Ask**（聞くな、命じろ）。

- **Ask（聞く）:** オブジェクトの状態を取得して、外側で判断する
- **Tell（命じる）:** オブジェクトに「やりたいこと」を伝え、判断はオブジェクト内部で行う

```typescript
// ❌ Ask: 状態を聞いて外側で判断
if (user.getRole() === 'admin' && user.getStatus() === 'active') {
  // 管理者処理
}

// ✅ Tell: 判断をオブジェクトに委ねる
if (user.canPerformAdminAction()) {
  // 管理者処理
}
```

## 適用の例外

デメテルの法則は**全てのドットチェーンを禁止**しているわけではない。以下は違反にならない:

### フルーエントインターフェース / ビルダーパターン

```typescript
// ✅ 違反ではない: 同じオブジェクト（または同じ型）を返すチェーン
const query = db.select('users')
  .where('status', 'active')
  .orderBy('created_at', 'desc')
  .limit(10);
```

各メソッドが自身（またはクエリビルダー）を返しているため、内部構造を露出しているわけではない。

### データ構造（DTO / Value Object）のアクセス

```typescript
// ✅ 違反ではない: 振る舞いを持たないデータ構造のプロパティアクセス
interface Address {
  prefecture: string;
  city: string;
  street: string;
}

const city = address.city; // データの構造は公開契約の一部
```

デメテルの法則は「振る舞いを持つオブジェクト」間の結合を制限するもの。純粋なデータ構造のフィールドアクセスは対象外。

## 落とし穴

### 1. 委譲メソッドの爆発

法則を厳密に守ると、中継するだけのメソッド（ラッパー）が大量に発生する。`order.getCustomerName()` → `customer.getName()` のような薄い委譲が増えすぎたら、そもそもの設計（クラスの責務分割）を見直すべきサイン。

### 2. DTO / レスポンスオブジェクトへの過剰適用

APIレスポンスや画面表示用のDTOは、構造を公開することが目的のオブジェクト。これにデメテルの法則を適用するのは不適切。
