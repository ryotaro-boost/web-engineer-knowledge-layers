# バイナリサーチ（Binary Search / 二分探索）

> **一言で言うと:** ソート済みデータの「真ん中を見て、大きいか小さいかで半分を捨てる」を繰り返す探索。毎回データが半分になるので O(log n)。

## 概念

辞書で「ねこ」を引くとき、最初のページから順に見る人はいない。真ん中あたりを開き、「さ行」より後ろだとわかったら前半を無視する。これがバイナリサーチの発想。

```
ソート済み: [2, 5, 8, 12, 16, 23, 38, 56, 72, 91]
探す値: 23

ステップ1: 中央 = 16  → 23 > 16 → 右半分 [23, 38, 56, 72, 91]
ステップ2: 中央 = 56  → 23 < 56 → 左半分 [23, 38]
ステップ3: 中央 = 23  → 見つかった！
```

10件のデータで3ステップ。100万件でも最大20ステップ（log₂(1,000,000) ≈ 20）。

### 前提条件

**データがソート済みであること。** これが絶対条件。ソートされていないデータにバイナリサーチは使えない。

### 計算量

| ケース | 計算量 | 説明 |
|--------|--------|------|
| 最良 | O(1) | 中央が目的の値 |
| 平均・最悪 | O(log n) | 毎回半分に絞る |
| ソートが必要な場合 | O(n log n) + O(log n) | ソート + 探索 |

**n=1,000,000 のとき:**
- 線形探索: 最大 1,000,000 回の比較
- バイナリサーチ: 最大 20 回の比較

### いつ使うか / いつ使わないか

**バイナリサーチが有効な場面:**
- データがソート済み（または一度ソートすれば何度も検索する）
- データが大きい（数千件以上）
- 「ちょうどの値」だけでなく「〜以上の最小値」「〜以下の最大値」も探せる

**バイナリサーチが不向きな場面:**
- データがソートされておらず、検索が1回だけ → 線形探索の方が速い
- 完全一致のキー検索 → [[ハッシュテーブル]] の O(1) の方が速い
- リンクリスト → ランダムアクセスできないので半分に飛べない

## 基本実装

### JavaScript / TypeScript

```typescript
// 基本形: ソート済み配列から値を探す
function binarySearch(arr: number[], target: number): number {
  let left = 0;
  let right = arr.length - 1;

  while (left <= right) {
    const mid = Math.floor((left + right) / 2);

    if (arr[mid] === target) return mid;      // 見つかった
    if (arr[mid] < target)  left = mid + 1;   // 右半分へ
    else                    right = mid - 1;   // 左半分へ
  }

  return -1; // 見つからなかった
}

const sorted = [2, 5, 8, 12, 16, 23, 38, 56, 72, 91];
console.log(binarySearch(sorted, 23)); // 5
console.log(binarySearch(sorted, 10)); // -1
```

### PHP

```php
<?php
// 手動実装
function binarySearch(array $arr, int $target): int {
    $left = 0;
    $right = count($arr) - 1;

    while ($left <= $right) {
        $mid = intdiv($left + $right, 2);

        if ($arr[$mid] === $target) return $mid;
        if ($arr[$mid] < $target)   $left = $mid + 1;
        else                        $right = $mid - 1;
    }

    return -1;
}

// 組み込み関数もある（ただし使い勝手がやや独特）
$sorted = [2, 5, 8, 12, 16, 23, 38, 56, 72, 91];
echo binarySearch($sorted, 23); // 5
```

### Ruby

```ruby
sorted = [2, 5, 8, 12, 16, 23, 38, 56, 72, 91]

# bsearch — 組み込みメソッド（条件に合う要素を返す）
result = sorted.bsearch { |x| x >= 23 } # 23

# bsearch_index — インデックスを返す
idx = sorted.bsearch_index { |x| x >= 23 } # 5

# ⚠ bsearch のブロックは「求める値の方向」を返す必要がある
# find-minimum モード: true/false を返す → 条件を満たす最小要素
# find-any モード: 負/0/正 を返す → 完全一致
exact = sorted.bsearch { |x| x <=> 23 } # 23（見つからなければ nil）
```

### Go

```go
package main

import (
	"fmt"
	"sort"
	"slices" // Go 1.21+
)

func main() {
	sorted := []int{2, 5, 8, 12, 16, 23, 38, 56, 72, 91}

	// sort.SearchInts — 挿入位置を返す（値が存在しなくても位置を返す）
	idx := sort.SearchInts(sorted, 23)
	if idx < len(sorted) && sorted[idx] == 23 {
		fmt.Println("found at index", idx) // found at index 5
	}

	// slices.BinarySearch（Go 1.21+）— 見つかったかどうかも返す
	idx2, found := slices.BinarySearch(sorted, 23)
	fmt.Println(idx2, found) // 5 true
}
```

## 実務での使用シーン

### 1. フロントエンドのリスト仮想化（Virtual Scroll）

大量のアイテムリスト（数万件）で、画面に表示する範囲だけを描画する。スクロール位置から「何番目のアイテムから描画すべきか」をバイナリサーチで特定する。

```typescript
// 各アイテムの高さが異なるリストで、スクロール位置から表示開始インデックスを求める
// prefixHeights = [0, 40, 95, 150, 210, ...] （各アイテムの上端Y座標、累積値）
function findStartIndex(prefixHeights: number[], scrollTop: number): number {
  let left = 0;
  let right = prefixHeights.length - 1;

  while (left < right) {
    const mid = Math.floor((left + right) / 2);
    if (prefixHeights[mid] < scrollTop) left = mid + 1;
    else right = mid;
  }
  return left;
}
```

### 2. 「〜以上の最小値」を探す（Lower Bound）

バイナリサーチの応用で最も実用的なパターン。「ちょうどの値」がなくても「最も近い値」を見つけられる。

```typescript
// 価格帯フィルタ: 予算以下で最も高い商品を探す
function findMaxUnderBudget(sortedPrices: number[], budget: number): number {
  let left = 0;
  let right = sortedPrices.length - 1;
  let result = -1;

  while (left <= right) {
    const mid = Math.floor((left + right) / 2);
    if (sortedPrices[mid] <= budget) {
      result = mid;         // 候補を記録
      left = mid + 1;       // もっと高い価格も探す
    } else {
      right = mid - 1;
    }
  }
  return result; // budget 以下の最大値のインデックス
}

const prices = [100, 250, 500, 750, 1000, 1500, 2000];
console.log(findMaxUnderBudget(prices, 800)); // 3（750円）
```

### 3. DB のインデックス検索（B-Tree）

データベースの [[Resources/Study/Layer3-データ永続化/インデックス|インデックス]] は内部的に B-Tree（B木）を使っている。B-Tree の検索はバイナリサーチの一般化（2分岐ではなく多分岐だが、原理は同じ「範囲を絞り込む」）。

```sql
-- インデックスがある場合 → B-Tree でバイナリサーチ的に探索 O(log n)
SELECT * FROM products WHERE price <= 800 ORDER BY price DESC LIMIT 1;

-- インデックスがない場合 → 全件走査（線形探索） O(n)
```

## AIによる実装のアンチパターン

| アンチパターン | なぜ問題か | 対策 |
|---|---|---|
| **組み込みメソッドがあるのに手動実装する** — Ruby の `bsearch` や Go の `slices.BinarySearch` が使える場面で、ゼロからバイナリサーチを書く | off-by-one バグのリスクが増える。標準ライブラリはテスト済みでエッジケースも処理されている | 言語の標準メソッドを優先する。カスタム比較ロジックが必要な場合のみ手動実装 |
| **バイナリサーチの前に不要なソート済みチェック** — `if (!isSorted(arr)) arr.sort(); binarySearch(arr, target);` のようにソート済みかの検証を毎回入れる | ソート済みチェック自体が O(n) で、線形探索と変わらないコストがかかる。データの生成元でソート順を保証するのが正しい設計 | ソート済みであることはデータの出処（DBの `ORDER BY`、挿入時のソート維持）で保証する |

## よくあるバグ

1. **off-by-one エラー** — `left <= right` と `left < right` の違いで無限ループになるか、要素を見落とす。
2. **中央値のオーバーフロー** — `(left + right) / 2` は left と right が大きいとオーバーフローする。`left + Math.floor((right - left) / 2)` が安全。（JavaScript では Number の精度問題があるが、通常の配列サイズでは問題にならない）
3. **ソートされていないデータに適用** — 結果が不正になるが、エラーにはならないので気づきにくい。
