# ファイルディスクリプタ（File Descriptor）

> **一言で言うと:** プロセスが開いている I/O リソース（ファイル、ソケット、パイプなど）を識別するための非負整数。UNIX 系 OS における I/O の統一インターフェースであり、「すべてはファイルである（Everything is a file）」という UNIX 哲学の実装基盤。

## 概念

### ファイルディスクリプタとは何か

プロセスがファイルを開くと、カーネルは内部的に「オープンファイル記述（Open File Description）」という構造体を作成し、プロセスにはその参照番号として**非負整数**を返す。この整数がファイルディスクリプタ（fd）である。

```
プロセス A の fd テーブル          カーネル
┌────┬──────────────┐      ┌─────────────────────────┐
│ fd │ 参照先        │      │ オープンファイル記述       │
├────┼──────────────┤      ├─────────────────────────┤
│  0 │ ──────────────┼──→  │ stdin  (端末入力)         │
│  1 │ ──────────────┼──→  │ stdout (端末出力)         │
│  2 │ ──────────────┼──→  │ stderr (端末エラー出力)    │
│  3 │ ──────────────┼──→  │ /var/log/app.log (書込み) │
│  4 │ ──────────────┼──→  │ TCP ソケット :8080        │
└────┴──────────────┘      └─────────────────────────┘
```

**重要な点:**
- fd はプロセスごとに独立している（プロセス A の fd 3 とプロセス B の fd 3 は別のリソース）
- fd 0, 1, 2 は慣例的に標準入力（stdin）、標準出力（stdout）、標準エラー（stderr）に予約されている
- 新しい fd は常に**使用可能な最小の番号**が割り当てられる

### 「すべてはファイルである」

UNIX 系 OS では、ファイルだけでなくネットワークソケット、パイプ、デバイスなども fd を通じて同じ `read()` / `write()` インターフェースで操作できる。

| リソース | fd を得る方法 | 例 |
|---------|-------------|-----|
| 通常ファイル | `open()` | ログファイル、設定ファイル |
| ディレクトリ | `opendir()` → 内部的に `open()` | `/var/log/` |
| ソケット | `socket()` + `connect()` / `accept()` | TCP/UDP 通信 |
| パイプ | `pipe()` | プロセス間通信 |
| デバイス | `open()` | `/dev/null`, `/dev/random` |
| epoll インスタンス | `epoll_create()` | I/O 多重化 |
| タイマー | `timerfd_create()` | 時間ベースのイベント |

この統一性のおかげで、「ファイルに書く」処理と「ネットワークに送る」処理を同じコードで扱える。

## 3 層の構造

fd の仕組みを正確に理解するには、カーネル内部の 3 層構造を知る必要がある。

```
┌─ プロセス A ─┐     ┌─ プロセス B ─┐
│ fd テーブル   │     │ fd テーブル   │
│ 0 → [*]     │     │ 0 → [*]     │
│ 1 → [*]     │     │ 1 → [*]     │
│ 3 → [*]─────┼──┐  │ 3 → [*]─────┼──┐
└──────────────┘  │  └──────────────┘  │
                  ▼                    ▼
         ┌────────────────┐  ┌────────────────┐
         │オープンファイル  │  │オープンファイル  │
         │記述 (OFD)       │  │記述 (OFD)       │
         │ オフセット: 1024│  │ オフセット: 0   │
         │ フラグ: O_RDWR │  │ フラグ: O_RDONLY│
         │ inode → [*]────┼─→┤ inode → [*]────┼──┐
         └────────────────┘  └────────────────┘  │
                                                  ▼
                                          ┌──────────────┐
                                          │ inode (v-node)│
                                          │ サイズ: 4096  │
                                          │ 権限: 0644   │
                                          │ ブロック位置  │
                                          └──────────────┘
```

1. **fd テーブル（プロセスごと）** — 整数 → オープンファイル記述へのポインタ
2. **オープンファイル記述（システム全体で共有）** — 現在の読み書きオフセット、アクセスモード（読み取り/書き込み）、ステータスフラグ
3. **inode（ファイルシステム上）** — ファイルの実体情報（サイズ、権限、データブロックの位置）

この 3 層構造が分かると、以下の挙動が理解できる：
- **同じファイルを 2 回 `open()` する** → OFD が 2 つ作られる → オフセットは独立
- **`fork()` で子プロセスを作る** → fd テーブルはコピーされるが、OFD は共有される → オフセットも共有
- **`dup()` / `dup2()` で fd を複製する** → 同じ OFD を指す新しい fd ができる → オフセットは共有

## fork() と fd の共有

[[プロセスとスレッド|プロセス]]を `fork()` で複製すると、fd テーブルがコピーされるが、オープンファイル記述は**共有**される。これはWebサーバーの設計に直結する重要な挙動。

```python
import os

f = open('/tmp/shared.txt', 'w')

pid = os.fork()
if pid == 0:
    # 子プロセス: 親と同じ OFD を共有している
    f.write('child\n')   # オフセットは親と共有 → 上書きではなく続きに書く
    f.flush()
    os._exit(0)
else:
    # 親プロセス
    os.wait()
    f.write('parent\n')  # 子が書いた後のオフセットから書く
    f.flush()
    f.close()
```

```bash
cat /tmp/shared.txt
# child
# parent
```

Nginx や Apache の prefork モデルでは、親プロセスがリッスンソケット（fd）を開いてから `fork()` する。子プロセスは同じソケット fd を継承するため、全ワーカーが同じポートで `accept()` できる。

## リダイレクトの仕組み

シェルのリダイレクト（`>`, `<`, `2>&1`）は fd の操作そのもの。

```bash
# stdout（fd 1）をファイルに向ける
echo "hello" > output.txt
# 内部的には:
#   1. output.txt を open() → fd 3 を得る
#   2. dup2(3, 1) → fd 1 が output.txt を指すようになる
#   3. close(3)
#   4. echo が fd 1 に write() → ファイルに書かれる

# stderr（fd 2）を stdout（fd 1）と同じ先に向ける
command 2>&1
# 内部的には: dup2(1, 2) → fd 2 が fd 1 と同じ OFD を指す

# /dev/null に捨てる
command > /dev/null 2>&1
# fd 1 → /dev/null, fd 2 → fd 1 と同じ先（/dev/null）
```

## fd の上限とリーク

### プロセスごとの fd 上限

```bash
# ソフトリミット（現在の上限）を確認
ulimit -n
# 典型的な出力: 1024

# ハードリミット（引き上げ可能な最大値）を確認
ulimit -Hn
# 典型的な出力: 65536

# 一時的にソフトリミットを引き上げる
ulimit -n 65536
```

Webサーバーではクライアントごとにソケット fd を消費するため、同時接続数が fd の上限に制約される。高トラフィックなサーバーではリミットの引き上げが必要。

### fd リーク

開いた fd を閉じ忘れると、プロセスが使える fd が徐々に枯渇する。枯渇すると新しいファイルもソケットも開けなくなり、サービスが停止する。

```javascript
// Node.js — fd リークの例
const fs = require('fs');
const http = require('http');

http.createServer((req, res) => {
  // Bad: コールバック内でエラーが発生すると fd が閉じられない
  const fd = fs.openSync('/tmp/data.txt', 'r');
  const data = processData(fd);  // ここで例外が発生したら？
  fs.closeSync(fd);               // ← 到達しない
  res.end(data);
}).listen(3000);

// Good: try-finally または高レベル API を使う
http.createServer((req, res) => {
  const data = fs.readFileSync('/tmp/data.txt', 'utf8');  // 内部で open/close を管理
  res.end(data);
}).listen(3000);
```

```python
# Python — fd リークの検出
import os

# 現在のプロセスが開いている fd を一覧
for fd in range(1024):
    try:
        os.fstat(fd)
        print(f"fd {fd}: open")
    except OSError:
        pass
```

### リークの調査

```bash
# プロセスが開いている fd を確認（Linux）
ls -la /proc/<PID>/fd/

# lsof で特定プロセスの開いているファイルを確認
lsof -p <PID>

# fd 数をカウント
lsof -p <PID> | wc -l

# 全プロセスの fd 使用数ランキング
lsof | awk '{print $2}' | sort | uniq -c | sort -rn | head -20
```

## I/O 多重化と fd

大量のクライアント接続を効率よく扱うために、複数の fd を同時に監視する仕組みが I/O 多重化（I/O Multiplexing）である。

| 方式 | 監視可能な fd 数 | 計算量 | 備考 |
|------|:---:|:---:|------|
| `select()` | 通常 1024 まで | O(n) | 古い。fd_set をコピーするコスト |
| `poll()` | 制限なし | O(n) | select の改良版だが依然 O(n) |
| `epoll` (Linux) | 制限なし | O(1) | イベント駆動。Nginx, Node.js が使用 |
| `kqueue` (BSD/macOS) | 制限なし | O(1) | epoll と同等の仕組み |

Node.js のイベントループは内部的に `epoll`（Linux）/ `kqueue`（macOS）を使い、数万の fd（= 数万のクライアント接続）を1つのスレッドで効率的に監視している。

```python
# Python — select による I/O 多重化の基本
import select
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.bind(('0.0.0.0', 8080))
server.listen()
server.setblocking(False)

sockets = [server]

while True:
    # 複数のソケット fd を同時に監視し、読み取り可能なものだけを処理
    readable, _, _ = select.select(sockets, [], [])
    for s in readable:
        if s is server:
            conn, addr = s.accept()
            conn.setblocking(False)
            sockets.append(conn)  # 新しい fd を監視対象に追加
        else:
            data = s.recv(1024)
            if data:
                s.sendall(data)  # エコーバック
            else:
                sockets.remove(s)
                s.close()        # fd を閉じて解放
```

## 落とし穴

### 1. close() 忘れによるサイレントな障害

fd リークはすぐにはエラーにならず、数時間〜数日後に突然 `EMFILE (Too many open files)` で顕在化する。長期運行するサーバーほど深刻。

### 2. fork() 後の不要な fd の閉じ忘れ

`fork()` で子プロセスを作ると親の全 fd が継承される。子プロセスで不要な fd（例: 親のリッスンソケット）を閉じないと、親がソケットを閉じてもポートが解放されない。`close-on-exec`（`O_CLOEXEC` / `FD_CLOEXEC`）フラグで `exec()` 時に自動的に閉じる設定が推奨される。

```python
import os, fcntl

fd = os.open('/tmp/test.txt', os.O_RDONLY | os.O_CLOEXEC)
# exec() で別プログラムを起動したとき、この fd は自動的に閉じられる
```

### 3. fd 番号の再利用による誤操作

`close()` した fd の番号は、次の `open()` で再利用される。古い fd 番号を使い続けると、意図しない別のファイルやソケットに読み書きしてしまう。マルチスレッド環境では特に危険。

### 4. 標準 fd（0, 1, 2）のクローズ

デーモンプロセスで stdin/stdout/stderr を閉じると、次に `open()` したファイルが fd 0, 1, 2 に割り当てられ、ライブラリが stdout に書いたつもりのデータがそのファイルに書かれるという事故が起きる。閉じるのではなく `/dev/null` にリダイレクトするのが正解。
