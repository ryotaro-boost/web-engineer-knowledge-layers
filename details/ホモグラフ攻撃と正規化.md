# ホモグラフ攻撃と正規化（Homograph Attack & Normalization）

> **一言で言うと:** Unicode には見た目が同じ（または酷似した）異なるコードポイントが多数存在する。この性質を悪用して、正規のドメインや識別子に見せかけた偽物を作る攻撃がホモグラフ攻撃であり、Unicode 正規化はその対策の一つである。

## 概念

### ホモグラフ（Homoglyph）とは

異なるコードポイントでありながら、フォント上の見た目が同一または酷似する文字のペアをホモグラフと呼ぶ。

| 見た目 | 正規の文字 | 偽装文字 | スクリプト |
|--------|-----------|---------|-----------|
| `a` | U+0061 (Latin a) | U+0430 (Cyrillic а) | キリル文字 |
| `e` | U+0065 (Latin e) | U+0435 (Cyrillic е) | キリル文字 |
| `o` | U+006F (Latin o) | U+03BF (Greek ο) | ギリシャ文字 |
| `1` | U+0031 (Digit 1) | U+0399 (Greek Ι) | ギリシャ文字 |
| `-` | U+002D (Hyphen-Minus) | U+2010 (Hyphen) | 句読点 |
| `⁄` | U+002F (Solidus /) | U+2044 (Fraction Slash) | 記号 |

これらは**肉眼では区別できない**が、バイト列は完全に異なる。

---

### 攻撃の種類

#### 1. IDN ホモグラフ攻撃（ドメイン偽装）

国際化ドメイン名（IDN: Internationalized Domain Name）は Punycode を通じて非 ASCII 文字をドメイン名に使える。これを悪用する:

```
正規:  apple.com          → Punycode: apple.com
偽装:  аpple.com          → Punycode: xn--pple-43d.com
       ↑ キリル文字の а (U+0430)

ブラウザのアドレスバーには "аpple.com" と表示され、見た目は本物と区別できない。
```

フィッシングサイトに利用され、ユーザーが正規サイトだと誤認してログイン情報を入力してしまう。

#### 2. ソースコード攻撃（Trojan Source）

プログラムのソースコード内にホモグラフや双方向制御文字（Bidi override）を埋め込み、レビュアーには安全に見えるが実際には異なる動作をするコードを作る:

```javascript
// 見た目は同じだが異なる変数名
const userRole = "admin";     // Latin "userRole"
const uѕerRole = "guest";    // キリル文字の ѕ (U+0455) を含む

// レビュアーには同じ変数に見えるが、実際は別の変数
if (uѕerRole === "admin") {  // 常に false
  grantAccess();
}
```

#### 3. URL パスの偽装

```
正規:  https://example.com/login
偽装:  https://example.com/ⅼogin
                            ↑ U+217C (Small Roman Numeral Fifty = ⅼ)
```

URL のパス部分にホモグラフを使い、WAF やルーティングのフィルタをバイパスする。

#### 4. ユーザー名・表示名の偽装

SNS やチャットでの管理者なりすまし:

```
正規管理者:  admin
偽装:        аdmin（キリル文字の а）
偽装:        admin（全角 a + 半角 dmin）
```

---

## ブラウザの防御策

主要ブラウザは IDN ホモグラフ攻撃に対して以下の対策を実装している:

| 防御策 | 内容 |
|--------|------|
| 混合スクリプト検出 | ドメイン名に複数のスクリプト（Latin + Cyrillic など）が混在する場合、Punycode 表示に切り替える |
| 全文字同一スクリプト検出 | 全文字がキリル文字（`аррlе.com` のような完全偽装）の場合も Punycode 表示にする |
| 既知 TLD の保護 | `.com`, `.org` などの主要 TLD で特に厳格なチェックを適用 |
| コンフューザブルテーブル | Unicode が公開する「混同されやすい文字ペア」のリストに基づくフィルタリング |

```
ブラウザ表示の例:
  正規:   apple.com        → そのまま "apple.com" と表示
  偽装:   аpple.com        → "xn--pple-43d.com" と Punycode で表示（警告）
```

ただし、ブラウザの防御は完璧ではなく、新しい偽装パターンが常に発見されている。

## アプリケーション側での対策

### 1. Unicode 正規化（NFKC）の適用

NFKC（Compatibility Composition）は互換性のある文字を統一する。これによりホモグラフの一部を検出・統一できる:

```javascript
// NFKC で統一できるケース
const fullwidth = "ａｄｍｉｎ";              // 全角英字
console.log(fullwidth.normalize("NFKC"));     // "admin"

const roman = "ⅰⅱⅲ";                        // ローマ数字
console.log(roman.normalize("NFKC"));         // "iii"

// NFKC では統一できないケース（異なるスクリプトの類似文字）
const cyrillic_a = "\u0430";  // キリル а
const latin_a = "a";          // ラテン a
console.log(cyrillic_a.normalize("NFKC") === latin_a); // false（スクリプトが違うため）
```

**重要: NFKC は全角/半角やローマ数字の統一には有効だが、異なるスクリプト間のホモグラフは解決できない。**

### 2. スクリプト混在チェック（Confusable Detection）

異なる Unicode スクリプトが混在していないかをチェックする:

```python
import unicodedata

def detect_mixed_scripts(text):
    """テキストに複数のスクリプトが混在しているか検出する"""
    scripts = set()
    for char in text:
        if char.isalpha():
            # Unicode スクリプトプロパティを取得
            script = unicodedata.name(char, "").split()[0]
            scripts.add(script)
    return len(scripts) > 1, scripts

# 正規
print(detect_mixed_scripts("apple"))
# (False, {'LATIN'})

# 偽装（キリル文字混在）
print(detect_mixed_scripts("аpple"))  # а = U+0430
# (True, {'CYRILLIC', 'LATIN'})
```

### 4. 許可リスト方式

ユーザー名やドメイン名に使える文字を明示的に制限する:

```javascript
// ユーザー名に使える文字を ASCII + 一部記号に制限
function validateUsername(name) {
  const allowed = /^[a-zA-Z0-9._-]+$/;
  if (!allowed.test(name)) {
    throw new Error("使用できない文字が含まれています");
  }
  return true;
}

// 日本語を許可する場合はスクリプトを限定
function validateDisplayName(name) {
  // Latin, Hiragana, Katakana, CJK Unified のみ許可
  const allowed = /^[\p{Script=Latin}\p{Script=Hiragana}\p{Script=Katakana}\p{Script=Han}\d\s._-]+$/u;
  if (!allowed.test(name)) {
    throw new Error("許可されていない文字種が含まれています");
  }

  // 同じ文字種の中での混在はOKだが、Latin + Cyrillic のような危険な組み合わせを検出
  const hasLatin = /\p{Script=Latin}/u.test(name);
  const hasCyrillic = /\p{Script=Cyrillic}/u.test(name);
  if (hasLatin && hasCyrillic) {
    throw new Error("ラテン文字とキリル文字の混在は許可されていません");
  }
  return true;
}
```

### 5. Bidi 制御文字の除去（Trojan Source 対策）

```javascript
// ソースコードやユーザー入力から危険な制御文字を除去
function sanitizeBidiChars(text) {
  // Unicode Bidi 制御文字を除去
  return text.replace(/[\u200E\u200F\u202A-\u202E\u2066-\u2069]/g, "");
}
```

GitHub や主要 IDE は Bidi 制御文字を含むソースコードに警告を表示するようになっている。

## 防御策の使い分け

| 対策 | 効果範囲 | 適用場面 |
|------|---------|---------|
| NFKC 正規化 | 全角/半角、ローマ数字、合字 | ユーザー入力の前処理（検索、比較） |
| スクリプト混在チェック | 異なる言語体系の文字混在 | ユーザー名、ドメイン名のバリデーション |
| Confusable Detection (UTS #39) | 見た目が同じ文字ペア全般 | セキュリティクリティカルな識別子 |
| 許可リスト方式 | 許可した文字種のみ通過 | 最も確実だが多言語対応が制限される |
| Bidi 制御文字除去 | Trojan Source 攻撃 | ソースコード、コードレビュー |

## 実務でのチェックリスト

1. **ユーザー名・表示名** — ASCII のみ許可するか、許可するスクリプトを明示的に列挙する。NFKC 正規化を適用してから保存する
2. **検索・マッチング** — 比較前に NFKC 正規化する。大文字小文字の無視（case folding）と組み合わせる
3. **URL の表示** — IDN ドメインは Punycode と併記するか、混合スクリプトを検出して警告する
4. **ソースコードレビュー** — CI に非 ASCII 文字や Bidi 制御文字の検出ルールを組み込む
5. **ログ・監査** — ユーザー入力をログに記録する際、Unicode エスケープ形式（`\u0430`）も併記して目視でのすり替えを防ぐ

---

## よくある落とし穴

1. **「NFKC で正規化すれば安全」は誤り** — NFKC はスクリプト間のホモグラフ（Latin `a` vs Cyrillic `а`）を解決しない。正規化とスクリプトチェックは別のレイヤーの対策

2. **ブラウザに任せれば大丈夫、ではない** — IDN のブラウザ防御はドメイン名のみ。URL パス、メール本文、アプリ内の表示名はアプリケーションが防御する必要がある

3. **NFKC の副作用** — `①` → `1`、`㈱` → `(株)` のように意味が変わる変換もあるため、表示用テキストには適用しない。検索・比較用のインデックスにのみ使う

4. **Punycode の可読性** — `xn--n3h.com` のような Punycode 表記はユーザーにとって意味不明。「怪しい」と判断する材料にはなるが、ユーザー教育も必要
