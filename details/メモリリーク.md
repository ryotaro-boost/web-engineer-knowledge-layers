# メモリリーク（Memory Leak）

> **一言で言うと:** 不要になったメモリが解放されず、使用量が時間とともに増え続ける現象。GC言語でも「参照が残っている限り回収されない」ため日常的に発生する。

## 概念

### メモリリークとは何か

メモリリークは「確保したメモリを解放し忘れる」現象だが、GCのある言語（JavaScript、Java、Go、Python）と手動メモリ管理の言語（C、C++）では意味が異なる。

| 種類 | 原因 | 発生する言語 |
|------|------|-------------|
| **物理的リーク** | `malloc` / `new` したメモリを `free` / `delete` し忘れる | C, C++ |
| **論理的リーク** | コード上不要なのに参照が残っており、GCが回収できない | JavaScript, Java, Python, Go 等すべて |

Web開発で圧倒的に多いのは**論理的リーク**である。GCは「到達可能なオブジェクトは生きている」と判断するため、不要な参照を保持し続けるコードがリークの正体になる。

```
GCルート（グローバル変数、スタック上の変数）
  │
  ├── 必要なオブジェクト ✅ 回収しない
  ├── 不要だが参照が残っている ❌ GCは回収できない ← これがリーク
  └── 到達不能なオブジェクト ✅ GCが回収する
```

### なぜWebアプリケーションで深刻か

デスクトップアプリやCLIツールは終了時に全メモリが解放される。しかしWebサーバーやSPA（Single Page Application）は長時間動き続けるため、リークが蓄積する。

- **サーバー側:** Node.jsプロセスがリクエストを処理するたびに数KBずつ漏れると、数日〜数週間でOOM（Out of Memory）に到達する
- **ブラウザ側:** SPAでページ遷移のたびにイベントリスナーが蓄積すると、タブのメモリ使用量がGBに達しブラウザがクラッシュする

## パターン別の原因と対策

### パターン1: グローバルキャッシュの無制限成長

最も頻度の高いリーク。モジュールスコープの `Map` やオブジェクトにデータを追加し続け、削除しない。

```javascript
// ❌ リーク: エントリが増え続ける
const userSessions = new Map();

app.post('/login', (req, res) => {
  userSessions.set(req.body.userId, { token: generateToken(), loginAt: Date.now() });
  // ログアウトしない or ログアウト時に delete し忘れ → 永遠に残る
});

// ✅ 対策1: TTL付きキャッシュを使う
import { LRUCache } from 'lru-cache';
const userSessions = new LRUCache({ max: 10000, ttl: 1000 * 60 * 30 }); // 30分で期限切れ

// ✅ 対策2: WeakMap を使う（キーがGC対象になると自動削除）
// ※ WeakMap のキーはオブジェクトのみ。文字列キーには使えない
const metadata = new WeakMap();
function attachMetadata(obj, data) {
  metadata.set(obj, data); // obj への参照がなくなればエントリも消える
}
```

### パターン2: イベントリスナーの解除忘れ

ブラウザとNode.jsの両方で発生する。リスナーのクロージャが外部オブジェクトを参照するため、リスナーが残る限りそのオブジェクトもGCされない。

```javascript
// ❌ リーク: コンポーネント破棄時にリスナーが残る
class ChatRoom {
  constructor(socket) {
    this.messages = [];
    // socket はグローバルに生存 → handler → this.messages への参照が残り続ける
    socket.on('message', (msg) => {
      this.messages.push(msg);
    });
  }
}

// ✅ 対策: 破棄時にリスナーを解除する
class ChatRoom {
  constructor(socket) {
    this.messages = [];
    this.socket = socket;
    this.handler = (msg) => this.messages.push(msg);
    socket.on('message', this.handler);
  }
  destroy() {
    this.socket.off('message', this.handler);
    this.messages = null;
  }
}
```

React での典型例：

```javascript
// ❌ リーク: useEffect のクリーンアップ忘れ
useEffect(() => {
  const id = setInterval(() => {
    setCount(c => c + 1);
  }, 1000);
  // クリーンアップなし → アンマウント後もインターバルが走り続ける
}, []);

// ✅ 対策: クリーンアップ関数を返す
useEffect(() => {
  const id = setInterval(() => {
    setCount(c => c + 1);
  }, 1000);
  return () => clearInterval(id);
}, []);
```

### パターン3: クロージャによる意図しない保持

クロージャはスコープチェーン上の変数を保持する。大きなデータを持つ変数が意図せずクロージャに閉じ込められるとリークになる。

```javascript
// ❌ リーク: largeData がクロージャに閉じ込められる
function processData() {
  const largeData = fetchHugeDataset(); // 100MB のデータ
  const summary = computeSummary(largeData);

  // この関数は summary だけ使うが、同じスコープの largeData も保持してしまう
  return function getSummary() {
    return summary;
  };
}
const fn = processData(); // largeData は不要なのにGCされない

// ✅ 対策: スコープを分離する
function processData() {
  let summary;
  {
    const largeData = fetchHugeDataset();
    summary = computeSummary(largeData);
    // ブロックスコープを抜けるので largeData はGC対象になる
  }
  return function getSummary() {
    return summary;
  };
}
```

### パターン4: タイマーのクリア忘れ

`setInterval` や `setTimeout` のコールバックが外部オブジェクトを参照していると、タイマーが残る限りオブジェクトもGCされない。

```javascript
// ❌ リーク: clearInterval されないインターバル
function startPolling(endpoint, callback) {
  const cache = { data: null, retries: 0 };
  setInterval(async () => {
    cache.data = await fetch(endpoint).then(r => r.json());
    callback(cache.data);
  }, 5000);
  // インターバルIDを返していない → 停止できない
}

// ✅ 対策: AbortController パターンで停止可能にする
function startPolling(endpoint, callback, signal) {
  const id = setInterval(async () => {
    if (signal.aborted) { clearInterval(id); return; }
    const data = await fetch(endpoint, { signal }).then(r => r.json());
    callback(data);
  }, 5000);
  signal.addEventListener('abort', () => clearInterval(id));
}
```

## 検出方法

### Node.js

```bash
# ヒープスナップショットを取得して比較
node --inspect app.js
# Chrome DevTools で Memory タブ → Take heap snapshot
# 時間を空けて2回取得し、差分を確認

# process.memoryUsage() による簡易監視
node -e "setInterval(() => console.log(process.memoryUsage().heapUsed), 5000)"
```

```javascript
// プログラム内でのリーク検出用ヒープダンプ
import v8 from 'v8';
import fs from 'fs';

function dumpHeap(filename) {
  const snapshotStream = v8.writeHeapSnapshot(filename);
  console.log(`Heap snapshot written to ${snapshotStream}`);
}
// Chrome DevTools の Memory タブで .heapsnapshot ファイルを読み込み可能
```

### ブラウザ

1. Chrome DevTools → **Memory** タブ
2. **Allocation instrumentation on timeline** を選択して記録開始
3. リークが疑われる操作（ページ遷移、モーダル開閉）を繰り返す
4. 記録停止 → 解放されずに残っているオブジェクトを確認
5. **Retainers** パネルで「誰がこのオブジェクトを参照しているか」を辿る

### 判断基準

メモリ使用量が「増える」だけではリークとは限らない。V8はGCを遅延実行する。重要なのは：

```
リーク:     ┌─┐ ┌──┐ ┌───┐ ┌────┐   ← GC後の下限が上がり続ける
            │ │ │  │ │   │ │    │
         ───┘ └─┘  └─┘   └─┘    └──→ OOM

正常:       ┌─┐ ┌─┐ ┌─┐ ┌─┐
            │ │ │ │ │ │ │ │         ← GC後の下限が安定
         ───┘ └─┘ └─┘ └─┘ └────→ 安定
```

**GC後のベースライン（下限値）が時間とともに上昇するかどうか**が真のリーク判定基準。

## 落とし穴

### 1. 「WeakMap / WeakRef を使えばリークしない」

WeakMap のキーはオブジェクトのみ。文字列やNumberをキーにしたキャッシュには使えない。また WeakRef は参照先がいつGCされるか不定であり、キャッシュの「ヒット率」を保証できない。万能ではない。

### 2. 「メモリ使用量が安定したから大丈夫」

V8の `--max-old-space-size` の上限に張り付いている場合、GCが必死に回収して見かけ上安定しているだけの可能性がある。この状態ではGCの負荷が高くCPU使用率が上昇し、レスポンスタイムも悪化する。

### 3. 「外部ライブラリだからリークしないはず」

サードパーティライブラリも普通にリークする。特にイベント駆動のライブラリ（WebSocket、DB接続プール）で `close()` / `destroy()` を呼び忘れるのは定番のリーク原因。

### 4. サーバーサイドレンダリング（SSR）の落とし穴

SSRではリクエストごとにコンポーネントツリーを生成する。グローバルなシングルトンにリクエスト固有のデータを格納すると、リクエスト間でデータが蓄積してリークする。
