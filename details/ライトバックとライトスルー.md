# ライトバック（Write-Back）とライトスルー（Write-Through）

> **一言で言うと:** データの書き込みを「あとでまとめて」反映するか「即座に」反映するかという戦略の違い。速度と安全性のトレードオフであり、ファイルシステム、CPU キャッシュ、データベース、CDN など、バッファ/キャッシュが存在するあらゆる場面で登場する同じ構造の問題。

## 概念

### 書き込み方式の全体像

キャッシュやバッファを介してデータを書き込む際、「実際の格納先（ディスク、メインメモリなど）にいつ反映するか」で方式が分かれる。

```
アプリケーション
    │ write()
    ▼
┌──────────────┐
│  バッファ/    │──→ ここから先の格納先への反映タイミングが方式の違い
│  キャッシュ   │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  格納先       │  （ディスク、メインメモリ、リモートストレージなど）
└──────────────┘
```

## 主要な書き込み方式

### 1. ライトバック（Write-Back）

バッファ/キャッシュへの書き込みだけで完了とし、格納先への反映は後回しにする方式。

```
write() → バッファに書く → 即座に「成功」を返す
                ↓
         （後でまとめてディスクに書き出す）
```

**動作の流れ（OS のページキャッシュの場合）：**
1. アプリが `write()` を呼ぶ
2. データはカーネルのページキャッシュ（メモリ）に書かれる
3. `write()` は即座にリターンする（アプリはディスク書き込みを待たない）
4. ページキャッシュ上の変更されたページは**ダーティページ（Dirty Page）**としてマークされる
5. カーネルの `pdflush` / `writeback` スレッドが定期的にダーティページをディスクに書き出す

**いつディスクに書き出されるか：**
- 一定時間が経過したとき（Linux のデフォルト: 30秒）
- ダーティページの割合がしきい値を超えたとき
- `sync` / `fsync()` が明示的に呼ばれたとき
- メモリが不足してページキャッシュを回収する必要があるとき

```bash
# Linux: ダーティページの設定を確認
cat /proc/sys/vm/dirty_expire_centisecs    # ダーティページの有効期限（デフォルト 3000 = 30秒）
cat /proc/sys/vm/dirty_ratio               # ダーティページの割合がこの値を超えると書き込みプロセスがブロックされる
cat /proc/sys/vm/dirty_background_ratio    # この割合を超えるとバックグラウンド書き出しが始まる
```

```python
import os

# ライトバックの挙動を確認する例
f = open('/tmp/writeback_test.txt', 'w')
f.write('this data is in the page cache, not yet on disk')
# ↑ この時点ではディスクには書かれていない（ダーティページとして存在）

f.flush()       # Pythonの内部バッファ → OSのページキャッシュ（まだディスクではない）
os.fsync(f.fileno())  # OSのページキャッシュ → ディスク（ここで永続化される）
f.close()
```

**メリット:**
- 書き込みが非常に速い（メモリ速度で完了する）
- 複数回の書き込みをまとめて1回のディスクI/Oにできる（バッチ効果）
- アプリケーションがI/O待ちでブロックされない

**デメリット:**
- クラッシュ時にバッファ上のデータが失われる（durability の犠牲）
- データがどの時点でディスクに書かれたか分かりにくい

**使われている場所:**
- Linux のページキャッシュ（デフォルト）
- CPU の L1/L2/L3 キャッシュ（メインメモリへの書き込み）
- データベースのバッファプール（PostgreSQL の `shared_buffers`）
- RAID コントローラのバッテリバックアップ付きキャッシュ

### 2. ライトスルー（Write-Through）

バッファ/キャッシュと格納先の両方に同時に書き込む方式。格納先への書き込みが完了するまで `write()` は返らない。

```
write() → バッファに書く＋ディスクに書く → 両方完了して「成功」を返す
```

```python
# ライトスルー相当の動作を自前で実装する例
def write_through(filepath, data):
    f = open(filepath, 'w')
    f.write(data)
    f.flush()
    os.fsync(f.fileno())  # 毎回ディスクまで同期する
    f.close()
    # ↑ この関数が返った時点でデータはディスク上にある
```

**メリット:**
- データの永続性が保証される（`write()` が成功 = ディスクに書かれた）
- クラッシュしても失われるデータがない
- 実装がシンプルで挙動が予測しやすい

**デメリット:**
- 毎回ディスクI/Oを待つため、書き込みが遅い
- 小さな書き込みが頻繁にあると、ディスクI/Oがボトルネックになる

**使われている場所:**
- データベースの WAL（Write-Ahead Log）— トランザクションログは必ずディスクに書く
- NFS の `sync` マウントオプション
- 一部の組み込みシステムや金融系システム

### 3. ライトアラウンド（Write-Around）

キャッシュを経由せず、格納先に直接書き込む方式。書き込み後にそのデータを読み返す予定がない場合に有効。

```
write() → ディスクに直接書く（キャッシュには載せない）
```

**メリット:**
- 一度しか書かないデータでキャッシュが汚染されない
- キャッシュをよく読まれるデータのために温存できる

**デメリット:**
- 書き込んだ直後に読み返すと、キャッシュミスが発生して遅い

**使われている場所:**
- ログファイルの書き込み（書いた直後に読み返すことはまれ）
- バックアップデータの書き込み
- Linux の `O_DIRECT` フラグ（ページキャッシュをバイパス）

```c
// C — O_DIRECT でページキャッシュをバイパスする例
#include <fcntl.h>
int fd = open("/tmp/direct.dat", O_WRONLY | O_CREAT | O_DIRECT, 0644);
// このfdへの write() はページキャッシュを経由せず直接ディスクに書かれる
// ※ バッファのアライメント制約があるため実装は注意が必要
```

## 方式の比較

| 観点 | ライトバック | ライトスルー | ライトアラウンド |
|------|:---:|:---:|:---:|
| 書き込み速度 | 速い | 遅い | 遅い |
| データ安全性 | 低い（クラッシュで失われる） | 高い | 高い |
| 読み返し速度 | 速い（キャッシュに残る） | 速い（キャッシュに残る） | 遅い（キャッシュにない） |
| キャッシュ効率 | 書き込みデータでキャッシュ占有 | 書き込みデータでキャッシュ占有 | キャッシュを温存できる |
| 適した場面 | 読み書き頻繁、速度重視 | データ損失が許されない | 書いたら読まないデータ |

## 実世界での組み合わせ

実際のシステムでは単一の方式ではなく、データの重要度に応じて方式を使い分ける。

### データベース（PostgreSQL）の例

```
通常のデータ書き込み → ライトバック（shared_buffers 経由）
WAL（トランザクションログ） → ライトスルー（fsync で即座にディスクへ）
```

WAL はクラッシュリカバリの生命線なのでライトスルー、通常データはパフォーマンスのためにライトバック。この組み合わせにより「速度」と「安全性」の両立を実現している。

### Web アプリケーションの例

```
セッションデータ → ライトバック（Redis に書いて後で永続化）
決済トランザクション → ライトスルー（DB に同期書き込み）
アクセスログ → ライトアラウンド（バッファリングして追記、読み返さない）
```

### CDN / リバースプロキシの例

```
キャッシュ更新 → ライトスルー（オリジンとキャッシュを同時更新）
  or ライトバック（キャッシュだけ更新、TTL 切れでオリジンから再取得）
```

## 落とし穴

### 1. ライトバックの「成功」を信じすぎる

`write()` が成功を返しても、データがディスクにあるとは限らない。本当に永続化が必要なデータは `fsync()` を呼ぶか、ライトスルーの仕組み（WAL など）を使う。

### 2. ライトスルーの過剰適用

すべての書き込みをライトスルーにすると、パフォーマンスが壊滅的に落ちる。1件のHTTPリクエストでファイルに10回書き込みがあるとして、毎回 `fsync()` を呼ぶと 10 × 数ミリ秒のディスクI/Oが加算される。

### 3. バッテリバックアップ付きキャッシュの過信

RAID コントローラのバッテリバックアップ付きキャッシュは「ライトバックでも安全」とされるが、バッテリが劣化すると保護が効かなくなる。また、サーバー全損（火災など）には無力。

### 4. O_DIRECT の誤解

`O_DIRECT` はページキャッシュをバイパスするが、「確実にディスクに書かれる」保証ではない。ディスクコントローラ自体のキャッシュは依然として存在する。完全な永続化には `O_DIRECT` + `fsync()` が必要。
