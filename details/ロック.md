# ロック（Lock）

> **一言で言うと:** 共有リソースへの同時アクセスを制御し、一度に1つの処理だけがクリティカルセクションを実行できるようにする同期プリミティブ。

## 概念

### なぜロックが必要か

複数の処理が同じデータを同時に読み書きすると、実行順序によって結果が変わる（[[並行性の基本概念|競合状態]]）。ロックは「この区間は1人だけ通れる」という門番の役割を果たし、データの一貫性を保証する。

### クリティカルセクション（Critical Section）

ロックで保護される「共有リソースにアクセスするコード領域」のこと。ロックの本質は、このクリティカルセクションを**相互排他（Mutual Exclusion）**にすること。

```
ロック取得 → [クリティカルセクション: 共有データの読み書き] → ロック解放
```

クリティカルセクションは可能な限り短くすべきである。長いクリティカルセクションは他のスレッドの待機時間を増やし、並行性の利点を損なう。

## ロックの種類

### ミューテックス（Mutex — Mutual Exclusion）

最も基本的なロック。1つのスレッドだけがロックを保持でき、同じスレッドが解放する必要がある。

```python
import threading

mutex = threading.Lock()
shared_list = []

def add_item(item):
    mutex.acquire()      # ロック取得（他スレッドはここで待機）
    try:
        shared_list.append(item)
    finally:
        mutex.release()  # 必ず解放する（finally で保証）

# with 文を使うとより安全
def add_item_safe(item):
    with mutex:          # acquire + release を自動管理
        shared_list.append(item)
```

### リエントラントロック（Reentrant Lock / Recursive Mutex）

同じスレッドが同じロックを複数回取得できる。ロック取得回数をカウントし、同じ回数だけ解放されたときに他のスレッドがアクセス可能になる。

```python
import threading

rlock = threading.RLock()

def outer():
    with rlock:
        # 何か処理
        inner()  # 同じロックを再取得 → 通常の Lock ならデッドロック

def inner():
    with rlock:  # 同じスレッドなので再取得可能
        pass
```

**使いどころ:** 再帰的な処理や、ロック内で別のロック付きメソッドを呼び出す場合。ただし設計の複雑さが増すため、必要でなければ通常の Mutex を使う。

### 読み書きロック（Read-Write Lock / RWLock）

「読み取りは複数同時に許可、書き込みは排他」というロック。読み取りが圧倒的に多い場面でスループットを大幅に改善する。

```go
package main

import (
    "fmt"
    "sync"
)

var (
    data   = make(map[string]string)
    rwLock sync.RWMutex
)

// 読み取り: 複数の Goroutine が同時実行可能
func read(key string) string {
    rwLock.RLock()         // 読み取りロック
    defer rwLock.RUnlock()
    return data[key]
}

// 書き込み: 排他的（他の読み取り・書き込みを全てブロック）
func write(key, value string) {
    rwLock.Lock()          // 書き込みロック
    defer rwLock.Unlock()
    data[key] = value
}
```

| 操作 | 読み取りロック中 | 書き込みロック中 |
|------|:---:|:---:|
| 別の読み取り | 可能 | ブロック |
| 書き込み | ブロック | ブロック |

### スピンロック（Spinlock）

ロック取得を待つ間、スレッドをスリープさせずにループで繰り返しチェックする（busy-wait）。コンテキストスイッチのコストを避けられるが、CPU を消費し続ける。

**使いどころ:** ロック保持時間が非常に短い場合（マイクロ秒以下）、かつマルチコア環境。カーネル内部やリアルタイムシステムで使われることが多い。一般的なアプリケーションでは使うべきではない。

### 楽観的ロック（Optimistic Locking）

ロックを取得せずに処理を進め、書き込み時に「他の誰かが変更していないか」を検証する。競合が少ない場面で高いスループットを発揮する。

```sql
-- バージョン番号による楽観的ロック
SELECT balance, version FROM accounts WHERE id = 1;
-- balance = 1000, version = 3

UPDATE accounts
SET balance = 800, version = 4
WHERE id = 1 AND version = 3;
-- version が変わっていたら 0 行更新 → 競合検出 → リトライ
```

```javascript
// HTTP API での楽観的ロック（ETag）
// 1. リソース取得時に ETag を受け取る
const res = await fetch('/api/resource/1');
const etag = res.headers.get('ETag');

// 2. 更新時に If-Match ヘッダで送信
await fetch('/api/resource/1', {
  method: 'PUT',
  headers: { 'If-Match': etag },

});
// ETag が一致しなければ 412 Precondition Failed → リトライ
```

### 悲観的ロック（Pessimistic Locking）

アクセス前にロックを取得し、処理が終わるまで他をブロックする。競合が頻繁に起きる場面で、リトライコストを避けられる。

```sql
-- SELECT FOR UPDATE で行ロックを取得
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- 他のトランザクションはこの行を更新できない
UPDATE accounts SET balance = balance - 200 WHERE id = 1;
COMMIT;
```

## 楽観的ロック vs 悲観的ロック

| 観点 | 楽観的ロック | 悲観的ロック |
|------|------------|------------|
| 前提 | 競合はめったに起きない | 競合が頻繁に起きる |
| コスト | 競合時のリトライが高い | ロック待ちが常に発生 |
| スループット | 競合少 → 高い / 競合多 → 低い | 安定するが上限が低い |
| Web での典型例 | ETag、バージョン番号 | `SELECT FOR UPDATE` |
| 適用場面 | 読み取り多、書き込み少 | 書き込み多、データ整合性が最優先 |

## 落とし穴

### 1. ロックの取得忘れ・解放忘れ

ロックを取得したのに例外で解放されないと、他のスレッドが永遠にブロックされる。言語が提供する自動解放の仕組みを必ず使う。

```python
# 悪い例
lock.acquire()
do_something()   # ここで例外が発生するとロック解放されない
lock.release()

# 良い例
with lock:
    do_something()  # 例外が発生しても自動解放
```

```go
// Go では defer で即座に解放を予約する
mu.Lock()
defer mu.Unlock()
// 処理
```

### 2. ロックの粒度の誤り

粒度が粗すぎる（テーブル全体をロック）とスループットが激減し、細かすぎる（フィールドごとにロック）とデッドロックリスクと管理コストが増える。

### 3. ロックの中で I/O を行う

ロックを保持したままネットワーク通信やディスク I/O を行うと、I/O の遅延がそのまま他のスレッドのブロック時間になる。ロック内の処理は CPU バウンドな操作に限定すべき。

### 4. ロックとパフォーマンスのトレードオフを見誤る

「安全のために全部ロック」はシングルスレッドと同じ速度になる。並行性の利点を活かすには、本当に共有されるデータだけをロックし、可能な限りロックフリーな設計を検討する。
