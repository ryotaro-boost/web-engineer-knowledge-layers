# 暗号アルゴリズム（Cryptographic Algorithms）

> **一言で言うと:** TLS が通信を保護するために組み合わせる暗号技術の構成要素。鍵交換・認証・暗号化・メッセージ認証の4つの役割があり、それぞれに適したアルゴリズムが選択される。

## 暗号スイートの構造

TLS では複数のアルゴリズムを組み合わせた**暗号スイート（Cipher Suite）**を使う。TLS 1.3 では暗号スイートが大幅に簡素化され、安全な組み合わせのみが許可されている。

```
TLS 1.2 の暗号スイート名の読み方:
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
     ^^^^^^ ^^^      ^^^ ^^^ ^^^^^^
     鍵交換  認証     暗号化  モード ハッシュ

TLS 1.3 の暗号スイート（鍵交換と認証は別途ネゴシエーション）:
TLS_AES_256_GCM_SHA384
    ^^^ ^^^ ^^^^^^
    暗号化  モード ハッシュ
```

| 役割 | 目的 | TLS 1.3 で使われるアルゴリズム |
|------|------|------|
| **鍵交換** | 共通鍵を安全に共有する | ECDHE（X25519, secp256r1, secp384r1） |
| **認証** | サーバー（/クライアント）の身元を証明する | RSA, ECDSA, Ed25519 |
| **暗号化** | 通信データを暗号化する | AES-128-GCM, AES-256-GCM, ChaCha20-Poly1305 |
| **ハッシュ/MAC** | データの完全性を検証する | SHA-256, SHA-384（AEAD に統合） |

## 共通鍵暗号（対称暗号 / Symmetric Encryption）

送信側と受信側が**同じ鍵**でデータを暗号化・復号する方式。高速だが、鍵の共有方法が課題になる（→ 鍵交換で解決）。

### AES-GCM（Advanced Encryption Standard - Galois/Counter Mode）

現在のTLSで最も広く使われる暗号化方式。

- **AES**: ブロック暗号。鍵長128/256ビット。2001年にNISTが標準化
- **GCM**: AES をストリーム暗号的に使い、同時に認証タグ（MAC）を生成する**AEAD（Authenticated Encryption with Associated Data）**モード
- **利点**: ハードウェアアクセラレーション（AES-NI）が広く普及しており、対応CPUでは非常に高速
- **注意**: 同じ鍵とnonce（初期化ベクトル）の組み合わせを再利用すると暗号が完全に破られる

### ChaCha20-Poly1305

AES-NI を持たないデバイス（旧世代のモバイル端末等）向けの代替AEAD。

- **ChaCha20**: ストリーム暗号。ソフトウェア実装でもAESに匹敵する速度
- **Poly1305**: メッセージ認証コード（MAC）
- **利点**: AES-NI なしの環境でAES-GCMより高速。タイミング攻撃への耐性が高い
- **採用例**: Google が Android 端末向けに推進し、TLS 1.3 に標準採用

## 公開鍵暗号（非対称暗号 / Asymmetric Encryption）

**公開鍵**と**秘密鍵**のペアを使う方式。公開鍵で暗号化したデータは秘密鍵でのみ復号できる。共通鍵暗号と比べて計算コストが高い（100〜1000倍程度）ため、TLSでは鍵交換と認証にのみ使用する。

### RSA

- 最も歴史の長い公開鍵暗号方式（1977年）
- 鍵長2048ビット以上が必須（4096ビット推奨の流れ）
- TLS 1.3 では**RSA鍵交換は廃止**（前方秘匿性がないため）。認証（証明書の署名）には引き続き使用可能
- 鍵サイズが大きくなると計算コストが急増する

### ECDSA（Elliptic Curve Digital Signature Algorithm）

- 楕円曲線暗号に基づくデジタル署名アルゴリズム
- RSA と同等の安全性を、はるかに短い鍵長で実現（ECDSA 256ビット ≈ RSA 3072ビット）
- 証明書のサイズが小さくなり、TLSハンドシェイクが高速化する
- Let's Encrypt が ECDSA 証明書の発行に対応

### Ed25519

- Curve25519 上の EdDSA（Edwards-curve Digital Signature Algorithm）
- ECDSA よりさらに高速で、実装上の落とし穴が少ない（定数時間演算が容易）
- TLS 1.3 で使用可能。SSH鍵としても広く普及
- 鍵長は固定256ビット

## 鍵交換（Key Exchange）

通信の暗号化に使う共通鍵を、盗聴されうるネットワーク上で安全に共有するための仕組み。

### ECDHE（Elliptic Curve Diffie-Hellman Ephemeral）

TLS 1.3 で唯一許可されている鍵交換方式。

- **Diffie-Hellman**: 双方が公開値を交換し、共有秘密を導出する。盗聴者は公開値だけでは共有秘密を計算できない
- **Elliptic Curve**: 楕円曲線を使うことで、DHの安全性を短い鍵長で実現
- **Ephemeral（一時鍵）**: 接続ごとに新しい鍵ペアを生成する。これが**前方秘匿性（Forward Secrecy）**を保証する — サーバーの秘密鍵が将来漏洩しても、過去の通信は復号できない

```
クライアント                    サーバー
  鍵ペア生成                     鍵ペア生成
  (秘密鍵a, 公開鍵A)            (秘密鍵b, 公開鍵B)
       |                            |
       |-------- 公開鍵A ---------->|
       |<------- 公開鍵B -----------|
       |                            |
  共有秘密 = a × B             共有秘密 = b × A
  （両者が同じ値を得る。盗聴者はA, Bだけでは計算不可能）
```

### X25519

- Curve25519 上の ECDHE。TLS 1.3 で最も推奨される鍵交換曲線
- 高速で、実装上のミスによる脆弱性が起きにくい設計
- secp256r1（P-256）も広く使われるが、X25519の方が新しく安全性の懸念が少ない

## ハッシュ関数

任意の長さのデータから固定長のダイジェスト（ハッシュ値）を生成する一方向関数。TLSでは完全性検証とAEAD内部で使用される。

| アルゴリズム | 出力長 | 状態 |
|------------|--------|------|
| MD5 | 128ビット | **廃止**（衝突攻撃が実証済み） |
| SHA-1 | 160ビット | **廃止**（2017年にGoogleが衝突を実証） |
| SHA-256 | 256ビット | **現役**（TLS 1.3 標準） |
| SHA-384 | 384ビット | **現役**（より高い安全性マージン） |

## よくある落とし穴

### 1. 暗号化と認証を別々に行う（Encrypt-then-MAC vs MAC-then-Encrypt）

TLS 1.2 以前の CBC モードでは暗号化と MAC が分離されており、パディングオラクル攻撃（POODLE、Lucky Thirteen）の原因になった。TLS 1.3 では AEAD（暗号化と認証を一体化）のみが許可され、この問題は解消されている。

### 2. 「鍵長が長ければ安全」という単純化

AES-128 と AES-256 の安全性の差は理論的には大きいが、現実の脅威はアルゴリズムの総当たりではなく実装の脆弱性やプロトコルの欠陥にある。AES-128 でも十分に安全であり、AES-256 の計算コスト増が常に正当化されるわけではない。

### 3. 自前の暗号実装

暗号アルゴリズムの自前実装はタイミング攻撃やサイドチャネル攻撃に対して脆弱になりやすい。言語やプラットフォームが提供する暗号ライブラリ（OpenSSL、BoringSSL、libsodium 等）を使うべき。
