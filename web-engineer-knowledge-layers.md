## Layer 0: コンピュータサイエンス基礎

**解決する問題：** 計算機の上で「何ができて何ができないか」の限界と、効率の良い問題解決手法を知る。

### 本質的な理解ポイント

- **データ構造とアルゴリズム** — 配列・ハッシュテーブル・木構造・グラフ。「検索が O(1) なのか O(n) なのか」がシステム全体のスケーラビリティを決める。Web開発で最も頻出するのはハッシュマップ（オブジェクト/辞書）とキュー。
- **計算量（Big-O）** — 「N が100万になったとき壊れるか？」を事前に判断する道具。ループのネストを見て爆発を予測する感覚。
- **文字コード（UTF-8）** — Webは世界中の文字を扱う。エンコーディングの誤解は文字化けとセキュリティホールの両方を生む。
- **並行性の基本概念** — 「2つの処理が同時に動くとき何が起きるか」。競合状態・デッドロックの原理を知らないと、非同期処理やDBトランザクションの設計で必ず事故が起きる。

### 依存関係
このレイヤーは他に依存しない「土台の土台」。ここが弱いと上の全レイヤーで「なぜそうなるのか」が理解できず、暗記ベースの開発になる。

## Layer 1: OS・インフラ

**解決する問題：** アプリケーションが動く「物理的・論理的な基盤」を理解し、リソースを適切に管理する。

### 本質的な理解ポイント

- **プロセスとスレッド** — 「Webサーバーがリクエストをどう捌いているか」の根本。Node.jsのシングルスレッド+イベントループも、Apacheのマルチプロセスモデルも、ここを理解しないと選択できない。
- **ファイルシステムとI/O** — ディスクI/Oはメモリアクセスの10万倍遅い。この感覚があると「なぜキャッシュが必要か」「なぜDBインデックスが重要か」が直感的に分かる。
- **メモリ管理** — ヒープ/スタック、GC。メモリリークはWeb長期運用の静かな敵。
- **コンテナ（Docker）** — 「環境の差異」という問題を解決する。「自分の環境では動く」を根絶する技術。本質は**プロセスの隔離**であり、仮想マシンとの違いはカーネルの共有。
- **Linux基本操作** — 本番環境のほぼ全てがLinux。ログを読む、プロセスを調べる、権限を設定する、という「トラブル時の第一手段」。

### 依存関係
Layer 0 の並行性の知識がプロセス/スレッド理解に直結する。

---

## Layer 2: ネットワーク・プロトコル

**解決する問題：** 物理的に離れた2台のマシンが、信頼性のある通信を行う。

### 本質的な理解ポイント

- **TCP/IP** — 「パケットが届く保証」を実現する仕組み。3ウェイハンドシェイクのコストを知ると、HTTP/2のコネクション多重化やKeep-Aliveの意味が分かる。
- **DNS** — ドメイン名→IPアドレスの変換。DNSの伝播遅延を知らないとドメイン移行で事故を起こす。TTLの概念が運用の鍵。
- **HTTP/HTTPS** — Webの共通言語。ステートレス性（毎回「初対面」）がCookieやセッションの存在理由。メソッド（GET/POST/PUT/DELETE）はただの「約束」だが、この約束がキャッシュ可否やべき等性に直結する。
- **TLS/SSL** — 通信の暗号化。「なぜHTTPSでないと危険か」は中間者攻撃を理解すると腑に落ちる。証明書の仕組みは「信頼の連鎖（Chain of Trust）」。
- **WebSocket** — HTTPの「リクエスト-レスポンス」モデルでは解決できない「サーバーからのプッシュ」を実現する。チャットやリアルタイム更新に必要。

### 依存関係
Layer 1 のプロセス/ポート/ソケットの概念に依存。TCP/IPの理解がHTTPの理解の前提になる。

## Layer 3: データ永続化

**解決する問題：** プロセスが終了してもデータが消えないようにし、大量のデータから高速に必要な情報を取り出す。

### 本質的な理解ポイント

- **RDB（PostgreSQL, MySQL）** — 「データの整合性」を最優先にした設計。正規化は「同じ事実を2箇所に書かない」というシンプルな原則。ACID特性（特にトランザクション）は「途中で失敗しても壊れない」保証。
- **インデックス** — B-Treeの原理。「なぜWHERE句が速くなるのか」を構造で理解する。インデックスの追加はメリットだけではなく書き込み性能とのトレードオフ。
- **NoSQL（Redis, MongoDB, DynamoDB）** — RDBの「柔軟性の欠如」と「水平スケーリングの難しさ」を解決するために生まれた。CAP定理を知ると「なぜ万能なDBが存在しないか」が分かる。
- **キャッシュ戦略** — 「同じ計算を二度しない」という原則の実装。ただしキャッシュの無効化はCS最難問の一つ。TTL、Write-through、Write-behindの使い分け。
- **マイグレーション** — スキーマの変更を安全に行う仕組み。「本番DBを壊さずに構造を変える」という実務上極めて重要な問題を解決する。

### 依存関係
Layer 1 のファイルI/Oの知識が「なぜインデックスが速いのか」の理解に必要。Layer 0 のデータ構造（B-Tree、ハッシュ）が内部構造の理解に直結。

## Layer 4: アプリケーション

**解決する問題：** ユーザーの要求をビジネスロジックとして表現し、適切なUIで提供する。

### 4-A: バックエンド

- **ルーティングとミドルウェア** — URLとビジネスロジックの対応付け。ミドルウェアは「共通処理（認証・ログ・エラーハンドリング）をDRYにする」仕組み。
- **認証と認可** — 認証は「あなたは誰か」、認可は「あなたは何ができるか」。この2つは別概念であり混同するとセキュリティホールになる。JWT vs セッションはステートレス vs ステートフルのトレードオフ。
- **API設計（REST / GraphQL）** — フロントエンドとバックエンドの「契約」。RESTはHTTPの意味論を活用する設計哲学。GraphQLは「フロントが必要なデータだけ取る」という問題を解決する（Over-fetching/Under-fetching問題）。
- **エラーハンドリング** — 「失敗は必ず起きる」という前提での設計。エラーを握りつぶさず、適切な粒度でキャッチし、ユーザーには分かりやすく・ログには詳細に記録する。
- **バリデーション** — 「外部からの入力は全て信用しない」。フロントのバリデーションはUXのため、バックのバリデーションはセキュリティのため。二重に必要。

### 4-B: フロントエンド

- **HTML/CSS/JS の本質** — HTMLは「文書の構造（セマンティクス）」、CSSは「見た目の宣言」、JSは「振る舞いの追加」。この3つの責務分離が全フロントエンド設計の基盤。
- **DOM と仮想DOM** — DOMの直接操作が遅い→差分だけ更新する仮想DOMが生まれた。ReactやVueの存在理由そのもの。
- **状態管理** — UIの複雑さの本質は「状態の管理」。どのコンポーネントがどの状態を持ち、変更をどう伝播するか。propsドリリング→Context→外部ストア（Redux等）は複雑さの段階に応じた解決策。
- **コンポーネント設計** — 再利用性と保守性のため。「このUIの単位をどう分割するか」は設計センスの問題であり、正解は「変更の理由が1つ」になる粒度（単一責任原則）。
- **アクセシビリティ（a11y）** — 「全てのユーザーが使える」という問題を解決する。セマンティックHTMLとARIA属性。法的要件でもある国が増えている。

### 依存関係
バックエンドは Layer 2（HTTP）と Layer 3（DB）に強く依存。フロントエンドは Layer 2（HTTP）とブラウザのレンダリングエンジンに依存。フロントとバックはAPI（Layer 2上の約束）を介して結合する。

## Layer 5: パフォーマンス・信頼性

**解決する問題：** 「動く」だけでなく「速く・落ちずに動き続ける」を実現する。

### 本質的な理解ポイント

- **CDN** — 「物理的な距離」がレイテンシの根本原因。ユーザーの近くにコンテンツを置くことで解決する。
- **ロードバランシング** — 1台のサーバーには限界がある。複数台に分散する際の「どう振り分けるか」と「セッション維持（Sticky Session）」の問題。
- **非同期処理・メッセージキュー** — 「今すぐやらなくていい処理」を切り離す。メール送信やレポート生成をリクエスト中にやるとレスポンスが遅くなる。RabbitMQ、SQS等。
- **モニタリング・オブザーバビリティ** — 「問題が起きたことを知る」ための仕組み。メトリクス（数値）、ログ（イベント）、トレース（経路追跡）の3本柱。「見えないものは直せない」。
- **Core Web Vitals** — フロントエンドの速度指標。LCP（最大コンテンツの表示速度）、INP（操作への応答速度）、CLS（レイアウトの安定性）。Googleの検索順位にも影響するためビジネス上も重要。

### 依存関係
Layer 1（OS・インフラ）の理解がボトルネック分析に不可欠。Layer 2（ネットワーク）のレイテンシ概念がCDNの理解の前提。

## Layer 6: セキュリティ

**解決する問題：** 悪意のある攻撃からシステムとユーザーを守る。

### 本質的な理解ポイント

- **XSS（クロスサイトスクリプティング）** — 「ユーザー入力がHTMLとして解釈される」ことで任意のJSが実行される。エスケープ処理とCSP（Content Security Policy）で防ぐ。
- **CSRF（クロスサイトリクエストフォージェリ）** — 「ブラウザがCookieを自動送信する」性質を悪用する攻撃。CSRFトークンやSameSite Cookieで防ぐ。
- **SQLインジェクション** — 「ユーザー入力がSQLとして解釈される」。プリペアドステートメント/パラメタライズドクエリで根本解決する。ORMを使っていても生SQLを書く箇所は要注意。
- **CORS** — ブラウザの「同一オリジンポリシー」を制御する仕組み。「なぜAPIが呼べないか」のトラブルの大半はCORS設定の問題。
- **最小権限の原則** — 「必要最低限の権限だけ与える」。DB接続ユーザー、APIトークン、IAMロール、全てに適用すべき基本思想。

### 依存関係
Layer 2（HTTP、Cookie）の理解がXSS/CSRFの前提。Layer 3（SQL）の理解がSQLインジェクションの前提。セキュリティは全レイヤーに横断的に関わる。

## Layer 7: 設計・アーキテクチャ

**解決する問題：** コードベースが成長しても、変更が容易で理解可能な状態を保つ。

### 本質的な理解ポイント

- **関心の分離** — 全設計原則の根本。「変更の理由が異なるものは分離する」。MVC、レイヤードアーキテクチャ、マイクロサービス、全てこの原則の応用。
- **SOLID原則** — 特に重要なのは「依存性逆転の原則（DIP）」。具体に依存せず抽象に依存することでモジュールの交換可能性を確保する。テスタビリティに直結する。
- **モノリス vs マイクロサービス** — 最初からマイクロサービスにするのはほぼ間違い。モノリスの「デプロイが一体」「変更の影響が波及する」という問題が**実際に痛みになってから**分割を検討する。
- **CI/CD** — 「手動デプロイが怖い」問題を解決する。自動テスト→自動ビルド→自動デプロイのパイプライン。小さく頻繁にリリースすることでリスクを下げる。
- **テスト戦略** — ユニットテスト（速い・安い・狭い）、インテグレーションテスト（中間）、E2Eテスト（遅い・高い・広い）のピラミッド。「何をどのレベルでテストするか」の判断が設計力。
- **イベント駆動 / CQRS** — 読み取りと書き込みで最適なモデルが異なるという認識から生まれた。大規模システムでの「読み書きの非対称性」を解決する。

### 依存関係
このレイヤーは全ての下位レイヤーを俯瞰した上での判断。技術的な知識だけでなく、ビジネス要件・チーム規模・成長見込みといった文脈依存の判断が求められる。

## レイヤー間の核心的な関係

```
「なぜこの技術が存在するか」は、
常に下のレイヤーの制約から説明できる。

例:
TCP が遅い → HTTP/2 で多重化
DOM 操作が遅い → 仮想DOM
ディスクI/O が遅い → キャッシュ, インデックス
1台に限界 → ロードバランサ, 水平スケーリング
手動デプロイが危険 → CI/CD
状態が複雑 → 状態管理ライブラリ
```

## 学習の順序について

下のレイヤーから順に学ぶ必要はない。実務では Layer 4（アプリケーション）から入り、壁にぶつかるたびに下のレイヤーに潜って理解を深める「トップダウン+必要に応じてボトムアップ」が最も効率的。ただし、Layer 0（CS基礎）と Layer 2（HTTP）は早期に押さえておくと全体の理解速度が大きく変わる。
